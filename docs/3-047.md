# 数据分析师或数据工程师或分析工程师或 BI 工程师？

> [`towardsdatascience.com/data-analyst-or-data-engineer-or-analytics-engineer-or-bi-engineer/`](https://towardsdatascience.com/data-analyst-or-data-engineer-or-analytics-engineer-or-bi-engineer/)

如果<mdspan datatext="el1745456944555" class="mdspan-comment">你一直跟随</mdspan>我，你可能知道我最初是一名**质量保证工程师**，后来转型进入数据分析领域。我没有为此上学，没有导师，也没有进入正式的培训计划。我今天所知道的一切——从 SQL 到建模到用数据讲故事——都是自学得来的。相信我，这是一段充满尝试、错误、学习和重新学习的旅程。

## 改变我职业生涯的困境

几年前，我开始考虑更换组织。像许多在快速发展的技术角色中的人一样，我面临着一个出人意料困难的问题：

> 我实际上在做什么角色？我应该申请哪些角色？

在纸面上，我是一个**数据分析师**。但在现实中，我的角色跨越了几个职能：编写 SQL 管道、构建仪表板、定义 KPI 和深入产品分析。我不确定我应该申请分析师角色、BI 角色，还是完全不同的角色。

更糟糕的是，当时，职位名称含糊不清，职位描述充满了行业术语。你会找到一个标题为*“数据分析师”*的职位，列出了如下要求：

+   构建机器学习管道

+   编写复杂的 ETL 脚本

+   维护数据湖

+   创建仪表板

+   展示高管级洞察

+   顺便说一下，要擅长利益相关者管理

这令人难以置信且令人困惑。我知道我不是唯一一个这样想的人。

快进到今天：幸运的是，事情正在演变。角色之间仍然存在重叠，但组织已经开始更清晰地定义它们。在这篇文章中，我想通过一个现实世界的例子来分析**数据角色之间的真正差异**。

### 现实世界场景：认识*Quikee*

让我们想象一家名为**Quikee**的虚构快速商业初创公司，它在多个印度城市推出。他们的价值主张？在**10 分钟**内配送杂货和必需品。

客户通过应用程序或网站下单。幕后，城市中遍布着微型仓库（也称为“暗仓”），以及一支快速配送的合作伙伴车队。

现在，让我们来看看这家公司的数据需求——从下单的那一刻开始，到高管在周一早晨会议上使用的仪表板。

### 第 1 步：捕获和存储原始数据

当客户下单的那一刻，就会生成**交易数据**：

+   时间戳

+   订单编号

+   订单商品

+   价格

+   折扣代码

+   客户位置

+   支付方式

+   指派的配送合作伙伴

假设 Quikee 使用**Amazon Kinesis**将数据实时流式传输到**S3 数据湖**。这个流是高容量、时间敏感的，对业务跟踪至关重要。

但这里有个问题：原始数据很杂乱。你不能直接用它来做决策。

那么，接下来会发生什么？

### 第 2 步：构建数据管道

出现了**数据工程师**。

他们负责：

+   摄入实时数据

+   验证模式一致性

+   处理失败和重试

+   编写将数据从 S3 移动到数据仓库（例如，Snowflake 或 Redshift）的管道

这正是**ETL**（提取、转换、加载）或**ELT**管道发挥作用的地方。数据工程师清理、格式化和结构化数据，使其可查询。

例如，一个订单表可能被拆分为：

+   **订单** → 每个订单一行

+   **订单项** → 每个订单中的每一项一行

+   **支付** → 每次支付尝试一行

在这个阶段，原始日志被转换为分析师可以工作的结构化表格。

### 第 3 步：维度建模与 OLAP

当领导开始提出战略性问题，如：

+   “哪个城市带来的收入最多？”

+   “哪个店铺表现不佳？”

+   “我们按区域平均配送时间是多少？”

…变得明显，直接查询事务数据是无法扩展的。

这就是**维度建模**的用武之地。

而不是扁平的原始表，数据被结构化为事实表和维度表。

### 🔸 **事实表**

+   大型、量化的数据表，其中包含外键以及度量标准和指标（*好吧，大多数时候是这样的。还有一些无事实的事实表，它们没有任何度量标准*）。

+   示例：`fact_orders`、`fact_payments`、`fact_deliveries`

+   包含如收入、订单数量、配送时间等指标

### 🔹 **维度表**

+   较小、描述性的表格，有助于理解事实表中的数据

+   示例：`dim_store`、`dim_product`、`dim_customer`、`dim_delivery_agent`

+   帮助过滤、分组和连接事实，以获得更深入的见解

这种结构使得**OLAP**（在线分析处理）成为可能——在多个维度上进行快速分析查询。例如，你现在可以运行如下查询：

> “给我展示过去 7 天内按店铺和每天的小时数平均配送时间。”

在大多数组织中，这一步是由数据工程师完成的，但我在亚马逊担任商业智能工程师时确实构建了一些维度和事实表。

### 第 4 步：定义关键绩效指标和度量标准

这正是**分析工程师（或 BI 工程师）**大显身手的地方。

他们位于技术数据层和业务用户之间。他们的职责通常包括：

+   定义关键绩效指标（例如，流失率、重复购买百分比、履行时间）

+   编写复杂度量的逻辑（例如， cohort 保留率、活跃用户数）

+   在 dbt 或 Looker 等工具中创建**语义模型**或**度量层**

+   确保公司内部定义的一致性

> 例如，在亚马逊，我们的团队并不是每次都查询原始数据来计算收入。相反，我们在每日、每周和每月粒度上创建了**预聚合的事实表**。这样，仪表板加载更快，指标在团队间保持一致。

分析工程师在工程和业务之间充当翻译者——定义**我们测量什么**以及**如何测量**。

### 第 5 步：分析、报告与讲故事

现在是**数据分析员**的角色。

拥有干净、建模好的数据，他们专注于回答真正的业务问题，例如：

+   “为什么上个月班加罗尔的留存率下降了？”

+   “哪些优惠券代码推动了最多新用户的增长？”

+   “前 30 天内哪些产品被重新订购最多？”

他们使用 Tableau、Power BI 或 Looker 等工具构建仪表板。他们运行即席 SQL 查询。他们深入研究 A/B 测试结果、用户行为趋势和活动有效性。

但最重要的是，他们用数据**讲述故事**——使复杂的数据对利益相关者变得可理解并可操作。

### TL;DR：你在哪里？

我认为它是这样的：

+   喜欢构建健壮的管道和解决可扩展性问题？→ 你是一名**数据工程师**

+   喜欢定义业务指标和组织复杂的数据集？→ 你是一名**分析工程师**

+   喜欢用数据挖掘洞察力和讲故事？→ 你是一名**数据分析员**

当然，现实世界的角色往往融合了这些。特别是在较小的公司，你可能需要戴多顶帽子。这是可以的。

关键不在于头衔——而在于**你为哪里增加了最大价值**以及**什么让你充满活力**。

## 最后的想法

我花了很长时间才真正理解我所做的事情——不仅仅是我的工作头衔所描述的。如果你曾经有过这种困惑，你并不孤单。

今天，我可以清楚地说我处于**数据建模**、**业务逻辑**和**讲故事**的交汇点——这是分析和工程之间的一个甜蜜点。我了解到，连接点的能力比完美地适应一个盒子更重要。

如果你走过类似的路径——或者在你的角色中戴多顶帽子——我很乐意听你的故事。
