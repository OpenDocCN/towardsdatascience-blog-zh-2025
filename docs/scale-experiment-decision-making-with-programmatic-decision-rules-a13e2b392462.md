实验的生命周期就像人类生命周期一样。首先，一个人或一个想法诞生，然后它发展，然后它被测试，然后测试结束，然后神（或产品经理）决定它的价值。

但在生命或实验过程中会发生很多事情。有时，一个人或一个想法在一个方面很好，但在另一个方面却很糟糕。神应该如何决定？他们必须做出一些权衡。这是不可避免的。

关键在于在实验之前和看到结果之前做出这些权衡。我们不希望根据我们对哪些想法值得进入天堂（嗯……发射——我想我已经把这个隐喻拉伸得足够远了）的先入为主的偏见来决定规则。我们希望在实验开始之前就写下我们的教义（好吧，再来一次）。

这篇博客的目的是提出我们应该明确地写下我们将如何做出决策——不是用英语，因为英语允许模糊的语言，例如，“我们还会考虑对参与度的影响，与收入平衡”以及类似的含糊不清、未量化的陈述——而是**用代码**。

我提议一个“分析合同”，它强制我们如何做出决策。

合同是你喜欢的编程语言中的一个函数。合同将实验的“基本结果”作为参数。确定哪些基本结果对决策很重要是定义合同的一部分。通常，在实验中，基本结果是处理效果、处理效果的标准误差以及配置参数，如[查看次数](https://zachlog.com/peeking-not-considered-harmful-0ed9c02aaf28)。有了这些结果，合同返回一个臂或实验的变体作为将要发射的变体。例如，在一个标准的 A/B 测试中，它将返回“A”或“B”。

它可能看起来像这样：

```py
int 
analysis_contract(double te1, double te1_se, ....)
{
  if ((te1/se1 < 1.96) &amp;&amp; (...conditions...))
    return 0 /* for variant 0 */
  if (...conditions...)
    return 1 /* for variant 1 */

  /* and so on */
}
```

然后，实验平台会将合同与特定的实验关联起来。当实验结束时，平台处理合同并根据合同中指定的规则发送获胜的变体。

我在这里加上一个警告，这是一个**想法**。**这**不是一个关于我在实践中看到的技术的故事，所以在实际部署中可能会有各种细节上的实际问题，这些问题会在实际部署中得到解决。我认为分析合同会减轻临时决策的问题，并迫使我们深入思考并预先注册我们将如何处理实验中最常见的场景：我们认为会有很大影响的效应实际上并不显著。

通过使用分析合同，我们可以……

## 提前做出决策。

我们不希望因为实验产生的特定数据集而改变我们的决策方式。

没有理由（好的理由）我们应该等到实验之后才说在情景 X 下我们会否发布。我们应该在实验之前就能说出这个决定。如果我们不愿意这样做，这可能意味着我们依赖于数据之外和实验结果之外的某些东西。这些信息可能是有用的，但那些不依赖于实验结果的信息在实验之前就已经有了。为什么我们没有承诺当时就使用它呢？

统计推断是基于行为模型。在这个模型中，我们知道如果我们知道某些参数，我们会如何做出决策。我们收集数据来估计这些参数，然后根据我们的估计来决定做什么。没有指定我们的决策函数会破坏这个模型，如果我们改变根据我们看到的数据来调用实验的方式，那么我们理所当然认为的许多统计属性就不再成立。

我们可能会说：“我们承诺不会这样做出决策。”但实验之后，结果并不很清晰。很多事物都不显著。因此，我们以百万种方式切割数据，找到一些“显著”的结果，并从它们中讲述一个故事。这很难遵守我们的承诺。

解决问题的方法不是做出我们无法实现的承诺。解决问题的方法是要做出一个系统不会让我们（悄悄地）打破的承诺。

## 在我们做决策时，要保持一致、明确和精确。

英语是一种模糊的语言，用英语写我们的指南留下了很多解释的空间。代码迫使我们明确地、定量地决定我们将做什么，例如，为了在长期内改善我们的订阅产品，我们愿意在短期内放弃多少收入。

代码极大地提高了沟通效率，因为我不必去解释你的意思。我可以输入不同的结果，看看如果结果不同，你会做出什么决策。这对于对过去实验的回顾性分析也非常有用。因为我们有一个实际的功能映射到决策上，我们可以运行各种模拟、自举等，并根据这些数据重新决定实验。

## 但如果我与分析合同的决策不一致怎么办？

分析合同的主要反对意见之一是，在实验之后，我们可能会决定我们做出了错误的决定函数。通常，问题是我们没有意识到实验会对度量 Y 产生什么影响，而我们的合同忽略了这一点。

既然如此，我们有两个选择的道路：

1.  如果我们有 1000 个度量，并且实验对每个度量的真实效果为 0，一些度量可能会产生很大的效果。一个解决方案是这次选择分析合同，并记住下次在合同中考虑这个度量。随着时间的推移，我们的合同将逐渐演变，更好地代表我们的真实目标。我们不应该过分重视第 20 个最重要的度量发生的事情。那可能只是噪音。

1.  如果效果确实非常大，我们无法安心忽略它，另一个解决方案是覆盖合同，确保在某个显眼的地方记录下这一事件。然后，更新合同，因为我们显然非常关注这个度量。随着时间的推移，我们覆盖合同次数的记录应作为我们实验系统的一个关键绩效指标。随着我们越来越接近最佳决策函数，我们应该停止覆盖。这可以是一个监控即兴、非统计决策发生频率的好方法。如果我们经常覆盖合同，那么我们知道合同并没有什么意义，我们并没有遵循良好的统计实践。这是内置的责任，它为覆盖合同创造了成本。

## 合同作为谓词

合同不需要是完全灵活的代码（即使从概念上讲很吸引人，允许直接在实验平台上指定可能存在安全问题）。但我们可以有一个系统，使实验者能够指定谓词，即 IF TStat(Revenue) ≤ 1.96 AND Tstat(Engagement) > 1.96 THEN X 等。我们可以暴露标准比较操作与 Tstat 的效果大小一起，并以此方式指定决策。
