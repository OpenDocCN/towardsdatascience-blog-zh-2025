# “系统思维帮助我把大局放在了首位”

> 原文：[`towardsdatascience.com/systems-thinking-helps-me-put-the-big-picture-front-and-center/`](https://towardsdatascience.com/systems-thinking-helps-me-put-the-big-picture-front-and-center/)

*在“作者聚焦”系列中，TDS 编辑们与我们的社区成员就他们的数据科学和人工智能职业道路、他们的写作以及他们的灵感来源进行交谈。今天，我们很高兴分享我们与[*Shuai Guo*](https://towardsdatascience.com/author/shuaiguo/)的对话。*

*Shuai 是一位工业人工智能研究员，与物理、数据和机器学习合作，解决工程、安全和智能系统中的实际问题。他在计算力学和机器学习的交叉领域拥有博士学位。他的工作涵盖了各种主题，包括异常检测、数字孪生、物理信息学习以及 LLM/代理应用。*

**你的** [**LangGraph 作品**](https://towardsdatascience.com/langgraph-101-lets-build-a-deep-research-agent/) **引导读者了解构建深度研究代理的过程。当你真正尝试端到端运行时，最让你惊讶的是什么，你下次会做些什么不同的事情？**

我想说，最让我惊讶的是深度研究代理在端到端运行时犯错的容易程度。整个“生成查询→搜索→反思→重复”的循环在纸面上看起来很棒，但很快就会崩溃。我清楚地记得两个主要问题。首先，时不时地，代理开始将找到的内容与它从预训练中记住的内容混淆起来。这并不理想，因为我只想让 LLM 综合信息并识别知识差距，而完全依赖网络搜索来定位答案。

另一个让我不断头疼的问题是信息污染，即搜索返回了类似的内容，但模型却将其视为你确切要求的内容。例如，我曾经通过研究一个特定的错误报告（比如，代码库中的问题#4521）来测试深度研究代理，搜索会返回与问题#4522 相关的内容，并开始混合它们的症状，好像它们都是同一个问题。

除了这两个主要问题之外，我在处理冲突信息以及确定深入研究的充分性方面也遇到了挑战。仅仅通过增加更多搜索结果或运行更多迭代，这些问题都无法解决。

对我来说，一个关键的领悟是，如果想要超越“仅仅是一个演示”并构建一个真正在生产中运行的系统，那么护栏与代理架构一样关键，甚至更为关键。我认为“测试驱动开发”的心态在这里非常合适：在构建之前先定义“好”是什么样的。下次，我会从定义明确的规则开始，然后在这些约束下构建代理架构。

**您曾写道** [**分析型人工智能（SQL/BI + 经典机器学习）并不会因为智能体变得热门而消失**](https://towardsdatascience.com/from-fomo-to-opportunity-analytical-ai-in-the-era-of-llm-agents/)**。如果您今天设计一个现代数据堆栈，您会给智能体分配哪些工作，而哪些工作会保留在分析通道中？**

分析型人工智能是可重复的且数值精确的。另一方面，基于 LLM 的智能体擅长消化非结构化上下文，翻译结果，并与人们沟通。在分配分析型人工智能和智能体人工智能之间的任务时，我会说如果任务更侧重于量化，我会默认使用分析型人工智能；但如果它更侧重于定性，例如综合、讲故事或判断，我会考虑 LLM/智能体作为更好的替代方案。

我们可以考虑一个具体的构建客户流失预测系统的问题。在高层面上，它通常涉及两个步骤：识别有风险的客户，并对他们采取行动。对于标记有风险客户的第一个步骤，我会依靠分析型人工智能来构建信息性特征，在历史行为数据上训练梯度提升模型，并使用训练好的模型来计算流失倾向分数。此外，我还会运行 SHAP 分析来获取解释预测的特征重要性分数。每一步都是精确且可重复的，并且有大量最佳实践可用于获得准确可靠的结果。

但接下来是更有趣的部分：你实际上会如何使用这些预测？这就是基于 LLM 的智能体可以接管的地方。他们可以通过拉取客户的历史记录来起草个性化的保留电子邮件，也许会建议他们尚未尝试的相关产品功能，并根据他们过去支持票证的走向调整语气。这里没有数学。只是以情境智能的方式进行交流。

**你早期投资了哪一项技能，现在随着 AI 工具变得更强大，这让你获得了优势？**

系统思维。

对于我来说，系统思维基本上是询问如何将系统分解为组件。不同的组件是如何相互交流的？交接点在哪里？反馈循环在哪里？如果我触摸*这个*，还有什么会改变？

我是在大学里接触到这个的。我主修航空航天工程，专注于航空发动机设计。关于喷气发动机的事情是，一切都会影响一切，研究它真的帮助我养成了三个习惯：分解系统，定义清晰的接口，并始终留意耦合效应。

诚然，AI 工具的能力在不断提升，例如我们获得了更好的编码助手、更有效的 RAG 管道，或者能够处理更长上下文的 LLMs，但大多数进步都发生在狭窄的领域。与其总是追逐最新的工具并试图将其以某种方式融入现有工作中，系统思维帮助我把大局放在首位。对于一个 LLM 应用，我通常会先绘制组件草图，确定组件之间的交互和输入/输出，确保添加了检查和防护措施，然后随着工具的改进进行组件的替换。

事实上，构建 LLM 应用让我想起了设计喷气发动机：新技术来来去去，但一个稳固的系统设计会累积价值。

**如果你放大视角，目前数据科学或人工智能的哪些部分变化太快，哪些部分又不够快？**

我认为多智能体 AI 系统无疑是正在迅速发展的最热门领域之一。我们不时会看到一些炫目的演示（无论是编码助手还是研究助手）。能够使开发者高效构建自己的多智能体应用的全新开源框架也不断涌现。所有这些都令人兴奋。但问题是：我们是否在比我们理解它们在实际中如何表现的速度更快地推出这些复杂的系统？

我认为这就是差距所在：围绕那些多智能体系统的整个“保证”层进化得太慢。为了应对这一挑战，我们可以（也许应该）将这些多智能体系统视为任何其他工业系统。在制造业中，采用数据驱动的方法来辅助系统设计、控制、状态监测和故障分析是一种常见的做法。这种方法同样可以惠及多智能体系统。例如，我们是否可以使用贝叶斯优化来设计多智能体架构？是否可以使用基于机器学习的异常检测来监控智能体的性能并捕捉安全威胁？

好消息是，势头正在形成。我们看到了 LLMs 的可观察性平台、评估框架等，它们正在为应用这些工业级、数据驱动的方法奠定基础。我认为这个领域有很多机会，这也是让我兴奋的原因：有机会将工业系统的严谨性带到智能体 AI 中，并使这些工具可靠且值得信赖。
