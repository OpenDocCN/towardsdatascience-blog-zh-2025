# 使用功能测试掌握提示工程：可靠 LLM 输出的系统指南

> 原文：[`towardsdatascience.com/mastering-prompt-engineering-with-functional-testing-a-systematic-guide-to-reliable-llm-outputs/`](https://towardsdatascience.com/mastering-prompt-engineering-with-functional-testing-a-systematic-guide-to-reliable-llm-outputs/)

为大型语言模型创建高效的提示通常开始于一项简单的任务……但并不总是保持这种状态。最初，遵循基本最佳实践似乎足够：扮演专家的角色，撰写清晰的指令，要求特定的响应格式，并包含几个相关示例。但随着需求的增加，矛盾出现，即使是微小的修改也可能引入意外的失败。在一个提示版本中完美工作的事情，在另一个版本中突然就失效了。

如果你曾经感到被困在无休止的试错循环中，调整一个规则却看到另一个失败，你并不孤单！现实是，传统的提示优化显然缺少一个结构化、更科学的途径，这将有助于确保可靠性。

这就是提示工程的功能测试的作用！这种方法，受实验科学方法论的启发，利用自动化输入-输出测试、多次迭代和算法评分，将提示工程转化为可衡量的、数据驱动的流程。

没有更多的猜测。没有更多的繁琐的手动验证。只有精确且可重复的结果，这允许你高效且自信地微调提示。

在这篇文章中，我们将探讨掌握提示工程系统化方法，确保你的 LLM 输出对于最复杂的 AI 任务既高效又可靠。

## 在提示优化中平衡精确性和一致性

在提示中添加大量规则可能会在规则之间引入部分矛盾，导致意外的行为。这尤其适用于从一般规则开始，随后跟随多个例外或具体矛盾用例的模式。添加特定的规则和例外可能会与主要指令冲突，甚至可能相互冲突。

看起来像是微小的修改可能会意外地影响提示的其他方面。这不仅适用于添加新规则时，也适用于添加现有规则的更多细节时，比如改变指令集的顺序，甚至只是简单地改写。这些微小的修改可能会无意中改变模型解释和优先处理指令集的方式。

你添加到提示中的细节越多，意外副作用的风险就越大。通过试图为任务的每个方面提供过多的细节，你也增加了获得意外或变形结果的风险。因此，找到清晰度和高度规范之间的正确平衡，以最大化响应的相关性和一致性是至关重要的。在某个点上，固定一个要求可能会破坏另外两个，从而在优化过程中产生一步向前两步向后的挫败感。

手动测试每个变化很快就会变得令人不堪重负。这尤其适用于需要优化必须遵循复杂 AI 任务中众多竞争性规范的提示时。这个过程不能仅仅是通过修改提示来满足一个要求后再满足下一个要求，希望之前的指令不受影响。它也不能是一个选择示例并手动检查的系统。一个更好的过程，采用更科学的方法，应该专注于确保提示优化的可重复性和可靠性。

## 从实验室到 AI：为什么测试 LLMs 的响应需要多次迭代

科学教导我们使用重复实验来确保可重复性，并建立对实验结果的信心。我在化学和生物学领域的学术研究工作已超过十年。在这些领域，实验结果可能受到众多因素的影响，从而导致显著的变异性。为确保实验结果的可靠性和可重复性，科学家们主要采用一种称为重复三次的方法。这种方法涉及在相同条件下重复进行相同的实验三次，使得实验变异性在结果中变得不那么重要。对结果进行的统计分析（标准均值和偏差），主要在生物学领域，使实验者能够确定结果的一致性，并增强对发现结果的信心。

就像在生物学和化学中一样，这种方法可以与 LLMs（大型语言模型）一起使用，以实现可靠的响应。使用 LLMs 时，响应的生成是非确定性的，这意味着由于模型的概率性质，相同的输入可能导致不同的输出。这种可变性在评估 LLMs 输出的可靠性和一致性时具有挑战性。

就像生物学/化学实验需要重复三次以确保可重复性一样，测试 LLMs 也应该需要多次迭代来衡量可重复性。因此，仅通过用例进行单一测试是不够的，因为它不能代表 LLMs 响应固有的可变性。每个用例至少进行五次迭代可以更好地进行评估。通过分析这些迭代中响应的一致性，可以更好地评估模型的可靠性并识别任何潜在的问题或变化。这确保了模型输出的正确控制。

将此应用于 10 到 15 个不同的提示需求，人们可以很容易地理解，如果没有结构化的测试方法，我们最终会花费时间在试错测试中，而没有有效的方法来评估质量。

## **系统方法：提示优化的功能测试**

为了应对这些挑战，可以使用结构化评估方法来简化并加速测试过程，并提高 LLM 输出的可靠性。这种方法有几个关键组成部分：

+   **数据测试用例：** 该方法的核心是数据测试用例，它由专门为提示测试创建的预定义输入-输出对组成。这些测试用例作为受控场景，代表 LLM 必须处理的多种需求和边缘情况。通过使用多样化的测试用例集，可以在不同条件下有效地评估提示的性能。

+   **自动测试验证：** 该方法通过比较测试用例中定义的预期输出和 LLM 响应来自动化对一组数据测试用例的要求验证。这种自动比较确保了一致性，并减少了评估过程中的人为错误或偏见。它允许快速识别差异，从而实现精细和高效的提示调整。

+   **多次迭代：** 为了评估 LLM 响应的固有可变性，该方法对每个测试用例进行多次迭代。这种迭代方法模仿了在生物/化学实验中使用的三重方法，为分析提供了更稳健的数据集。通过观察迭代中响应的一致性，我们可以更好地评估提示的稳定性和可靠性。

+   **算法评分：** 每个测试用例的结果都是通过算法进行评分的，这减少了长时间且繁重的“人工”评估需求。这个评分系统旨在客观和量化，为评估提示的性能提供清晰的指标。通过关注可衡量的结果，我们可以做出基于数据的决策，有效地优化提示。

### 第一步：定义测试数据测试用例

选择或创建兼容的测试数据测试用例是我们系统方法中最具挑战性的步骤，因为它需要仔细思考。测试用例不仅包括任何输入-输出对；它必须精心制作，以确保尽可能准确地评估 LLM 针对特定需求的表现。这个过程需要：

1. 对任务和模型行为的深入理解，以确保选定的示例能够有效地测试预期的输出，同时最大限度地减少歧义或偏见。

2. 预见测试期间如何通过算法进行评估。

因此，测试用例的质量不仅取决于示例的良好代表性，还在于确保其能够高效地通过算法进行测试。

测试用例包括：

• **输入示例:** 这是将被提供给 LLM 处理的数据。它应该代表 LLM 预期处理的典型或边缘情况。输入应设计为涵盖 LLM 在生产中可能必须处理的广泛可能的变体。

• **预期输出:** 这是 LLM 应该使用提供的输入示例生成的预期结果。它用于在验证期间与实际 LLM 响应输出进行比较。

### 第 2 步：运行自动化测试

一旦定义了测试数据固定装置，下一步就是执行自动化测试，以系统地评估 LLM 响应在所选用例上的性能。如前所述，此过程确保提示在各种场景中得到彻底测试，从而提供对其效率的可靠评估。

### 执行过程

1. **多次迭代:** 对于每个测试用例，相同的输入被多次提供给 LLM。在 nb_iter 中使用简单的 for 循环，nb_iter = 5，就这样！

2. **响应比较:** 在每次迭代后，LLM 响应与固定装置的预期输出进行比较。此比较检查 LLM 是否根据指定要求正确处理了输入。

3. **评分机制:** 每次比较结果都会得到一个分数：

◦ 通过（1）：响应与预期输出匹配，表明 LLM 已正确处理输入。

◦ 失败（0）：响应与预期输出不匹配，表明需要修复的差异。

4. **最终分数计算:** 所有迭代的分数被汇总以计算总体最终分数。这个分数代表成功响应占总迭代次数的比例。当然，高分数表示高提示性能和可靠性。

### **示例:** **从文章中移除作者签名**

让我们考虑一个简单的场景，其中 AI 任务是从文章中移除作者签名。为了有效地测试这个功能，我们需要一组代表各种签名风格的固定装置。

这个示例的数据集可以是：

| 示例输入 | 预期输出 |
| --- | --- |
| 一篇长文 Jean Leblanc | 长文 |
| 一篇长文 P. W. Hartig | 长文 |
| 一篇长文 MCZ | 长文 |

**验证过程:**

+   **签名移除检查:** 验证函数检查签名是否已从重写的文本中移除。这可以通过在输出文本中搜索签名针来程序化地轻松完成。

+   **测试失败标准:** 如果签名仍然在输出中，则测试失败。这表明 LLM 没有正确移除签名，并且需要进一步调整提示。如果不是，则测试通过。

测试评估提供了一个最终分数，允许对提示效率进行数据驱动评估。如果得分完美，则无需进一步优化。然而，在大多数情况下，你不会得到满分，因为 LLM 对某个案例的响应一致性可能很低（例如，5 次迭代中有 3 次得分积极）或者存在模型难以处理的边缘情况（5 次迭代中 0 次得分积极）。

反馈清楚地表明，仍有进一步改进的空间，并指导你重新审视你的提示，以消除模糊的措辞、冲突的规则或边缘情况。通过持续监控你的分数以及提示修改，你可以逐步减少副作用，提高效率和一致性，并接近最优和可靠的输出。

然而，使用所选模型并不总是能够获得满分。改变模型可能只是解决了问题。如果不行，你就知道你系统的局限性，并可以在你的工作流程中考虑这一事实。幸运的话，这种情况可能只是通过简单的模型更新在不久的将来得到解决。

### 这种方法的益处

+   **结果的可靠性**：进行五到十次迭代可以提供关于提示性能的可靠统计数据。单次测试运行可能成功一次但不会两次，而多次迭代的一致成功表明提示是健壮且经过良好优化的。

+   **过程效率**：与可能需要数周或数月才能复制的传统科学实验不同，LLM 的自动测试可以快速进行。通过设置高迭代次数并等待几分钟，我们可以获得高质量、可重复的提示效率评估。

+   **数据驱动优化**：这些测试获得的分数提供了对提示满足要求能力的数据驱动评估，允许有针对性的改进。

+   **并行评估**：结构化测试使得对提示版本进行评估变得容易。通过比较测试结果，可以确定实现预期结果的最有效参数集（指令措辞、指令顺序）。

+   **快速迭代改进**：快速测试和迭代提示的能力是精心构建提示的一个真正优势，确保在提示变得更加复杂和长时，之前验证的要求仍然保持不变。

通过采用这种自动测试方法，我们可以系统地评估和提升提示性能，确保符合要求的持续和可靠的输出。这种方法节省了时间，并为持续的提示优化提供了一个健壮的分析工具。

## 系统性提示测试：超越提示优化

实施系统性的提示测试方法不仅提供了初始提示优化的优势，而且在人工智能任务的其它方面也具有价值：

1. **模型比较**：

◦ 提供商评估：这种方法允许高效地比较不同 LLM 提供商，如 ChatGPT、Claude、Gemini、Mistral 等，在同一任务上的表现。评估哪个模型最适合他们的特定需求变得容易。

◦ 模型版本：当提示得到良好优化时，即使是复杂的 AI 任务，也不一定需要最先进的模型版本。轻量级、更快的版本可以提供相同的结果，并具有更快的响应速度。这种方法允许对模型的各个版本进行并行比较，例如 Gemini 1.5 闪存版与 1.5 专业版与 2.0 闪存版或 ChatGPT 3.5 与 4o 迷你版与 4o，并允许基于数据驱动选择模型版本。

2. **版本升级：**

◦ 兼容性验证：当发布新模型版本时，系统性的提示测试有助于验证升级是否保持了或提高了提示性能。这对于确保更新不会意外地破坏功能至关重要。

◦ 无缝过渡：通过识别关键需求和测试它们，这种方法可以促进向新模型版本的更好过渡，以便在必要时快速调整，以保持高质量的输出。

3. **成本优化：**

◦ 性能与成本比：系统性的提示测试有助于根据性能与成本比选择最佳性价比模型。我们可以有效地在性能和运营成本之间识别最有效的选项，以获得最佳的 LLM 成本回报。

## 克服挑战

这种方法的最大挑战是准备测试数据集，但随着时间的推移，投入的努力将显著回报。准备良好的数据集可以节省大量的调试时间，并通过为评估 LLM 响应提供坚实的基础来提高模型效率和可靠性。在 LLM 开发和部署中的改进效率和有效性将迅速收回初始投资。

### 快速的利弊分析

**主要优势**：

+   **持续改进**：随着时间的推移，能够添加更多要求的同时确保现有功能保持完整是一个显著的优势。这允许 AI 任务根据新要求进行演变，确保系统保持最新和高效。

+   **更好的维护**：这种方法使得使用 LLM 更新轻松验证提示性能变得容易。这对于保持高质量和可靠性标准至关重要，因为更新有时会引入行为上的意外变化。

+   **更多灵活性**：通过一套质量控制测试，切换 LLM 提供商变得更加简单。这种灵活性使我们能够适应市场变化或技术进步，确保我们始终能够使用最适合工作的最佳工具。

+   **成本优化：** 数据驱动的评估能够帮助我们在性能与成本比上做出更好的决策。通过理解不同模型的性能提升，我们可以选择最符合需求且成本效益最高的解决方案。

+   **节省时间：** 系统性评估提供快速反馈，减少了手动测试的需求。这种效率允许快速迭代提示的改进和优化，加速开发过程。

### 挑战

+   **初始时间投资：** 创建测试工具和评估函数可能需要大量的时间投入。

+   **定义可衡量的验证标准：** 并非所有人工智能任务都有明确的通过/失败条件。为验证定义可衡量的标准有时可能具有挑战性，尤其是对于涉及主观或细微输出的任务。这需要仔细考虑，可能涉及对评估指标的艰难选择。

+   **与多次测试相关的成本：** 与 5 到 10 次迭代相关的多个测试用例可能会为单个测试自动化生成大量大型语言模型请求。但如果单个大型语言模型调用的成本可以忽略不计，如大多数文本输入/输出调用的情况，测试的整体成本仍然保持最低。

## 结论：何时应该实施这种方法？

实施这种系统性的测试方法当然并非总是必要的，尤其是对于简单任务。然而，对于精度和可靠性至关重要的复杂人工智能工作流程，这种方法通过提供一种系统性的方式来评估和优化提示性能，防止了无休止的试错循环，变得非常有价值。

通过将功能测试原则融入提示工程，我们将一个传统上主观且脆弱的过程转变为可衡量、可扩展和稳健的过程。这不仅增强了大型语言模型输出的可靠性，还有助于实现持续改进和高效资源分配。

实施系统化提示测试的决定应基于项目的复杂性。对于需要高精度和一致性的场景，投入时间建立这种方法可以显著提高成果并加快开发过程。然而，对于更简单的任务，更经典、轻量级的方法可能就足够了。关键是要在严谨性和实用性之间取得平衡，确保你的测试策略与你的目标和限制相一致。
