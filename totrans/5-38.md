# ChatGPT 时代人类编写代码教程的终结……还是不会？

亲爱的 AI，请生成一张高级 AI 聊天机器人编写编程教程文章供人们阅读的图片。

在 ChatGPT 统治的世界里，人们可能会预期人类编写的代码教程会变得无效。

总的来说，当人们可以简单地询问 ChatGPT 以获得他们想要的精确输出时，谁会去阅读关于详细软件流程的人类编写的教程呢？再加上高级 LLM 订阅和几个高级提示工程技巧，任何人都可以生成关于任何主题的详细解释。为什么还要浪费时间和精力去研究、头脑风暴和撰写这样的文章？

让我们从一个问题开始，这个问题的答案将支撑我的整个论点：*ChatGPT 究竟是什么？*

好吧，ChatGPT，或者任何大型语言模型（LLM），都是通过机器学习算法来预测基于训练数据输入后应跟上的单词。将 LLM 与早期语言模型区分开来的是其训练数据的大小。例如，ChatGPT 的训练包括了 Common Crawl 数据集——[这是一个包含数十亿网页的巨型数据集](https://www.edureka.co/blog/how-chatgpt-works-training-model-of-chatgpt/)。

这一点至关重要，需要记住。无论这些生成的响应看起来多么令人难以置信和无缝，它们本质上只是基于先前数据的概率预测。它们是根据模型之前所见到的最可能输出。**它们不是新颖的**。

与之相反，人类的创造力通常源于新的洞察——通常被称为“灵光一闪”的时刻。是的，拥有先前的知识有帮助——但就其核心而言，这些新想法的美丽之处在于它们确实是：*新的*。当一个科学家、作家或艺术家有这种罕见的时刻时，他们就是中了大奖。

让我们将话题重新引回到手头的主题：人类编写的编程教程。人们在何时撰写文章或出版关于编程主题的书籍？通常，出于以下两个原因之一：

1.  为了对一个现有的想法提供新的*视角*。许多教程教授相同的一般主题，但在风格、方法或方法上有所不同。

1.  为了传达全新的想法或主题。

一个由 LLM 生成的响应无法做到这两点。

让我们考虑一个例子。在 2023 年初，我为 Towards Data Science 写了“作为 Python 初学者必须学习的 4 个基本技巧”一文，这篇文章普遍受到好评，阅读量也很大。下面，我插入了两段摘录：

1.  我实际撰写的关于 lambda 函数的文章部分。

1.  ChatGPT-4o 在我提供了必要细节并请求相关内容后撰写的关于 lambda 函数的部分。

## 示例 1：由人类撰写的 Lambda 教程

假设你正在使用 Jupyter notebook 处理一些数据，只是进行一些快速探索和分析。你仍然处于数据清洗和处理的早期阶段，远未达到任何生产就绪的模型、可视化或应用。但你有一个截止日期要遵守，所以你快速而高效地探索，充分利用你的卓越 Python 技能。

在你的冒险过程中，你发现数据中的一个列需要转换。你只需要将列中的数字平方。这并不疯狂，但遗憾的是，这也是那些简单到足以快速完成，但复杂到没有内置函数的奇怪需求之一。

因此，你决定使用 pandas 的 `apply` 函数通过你自己的自定义函数转换数据列。为此，你需要编写一个平方数字的函数，而你正是以你唯一知道的方式做到这一点：

```py
def square(num):
    return num * num
```

这完成了工作，但有点令人烦恼和杂乱，尤其是在 Jupyter notebook 中。它与大多数 pandas 操作的单行结构不太融合，因此当你的同事审查你的笔记本时，看起来不会很好看。

但不要绝望，我的朋友，因为 lambda 函数就在这里来拯救你。**Lambda 函数**——或者更普遍地说，匿名函数——为 Python 中定义函数提供了一种替代方法。最好的部分是，你可以一行代码就写出来！这最容易通过例子来展示：

```py
square = lambda num: num * num
```

上述代码与我们的早期函数定义相同。以下是一些你应该知道的事情：

+   `lambda` 关键字与 `def` 关键字类似，让 Python 知道我们想要定义一个函数。

+   lambda 函数的参数位于冒号的左侧，返回语句位于冒号的右侧（实际上并不使用 `return` 关键字）。

+   我们不必给函数命名，但如果我们想这样做，我们可以通过变量赋值来实现。

最后一点很重要。使用 lambda 函数可以让我们定义一个函数，并可以在不给它命名的情况下调用它或将其作为参数传递给另一个函数。让我们通过回到我们之前的例子并使其具体化来阐述这一点。

让我们想象一下，我们有一个名为 `my_df` 的 DataFrame，其中包含三个人的薪水信息：

```py
Name  Salary
0   John   45000
1   Mary   60000
2  Julie  100000
```

在这个极其理想化的世界里，雇主刚刚宣布每个人的薪水都将被**平方**。我们可以通过使用 lambda 函数在一行内更新我们的 DataFrame：

```py
>>> my_df['Salary'] = my_df['Salary'].apply(lambda num: num * num)
>>> my_df
    Name       Salary
0   John   2025000000
1   Mary   3600000000
2  Julie  10000000000
```

哇哦——无尽的财富！或许有些夸张，但希望你现在能记住 lambda 函数是什么以及如何使用它们。如果你想更详细地了解它们的细微差别和微妙之处，我建议查看我的[两篇详细讨论 lambda 函数的文章系列](https://towardsdatascience.com/whats-in-a-lambda-c8cdc67ff107)。

## 示例 2：大型语言模型编写的 Lambda 教程

***提示**：我正在为 Medium 上的 Towards Data Science 出版物撰写一篇名为“作为 Python 初学者必须学习的 4 个基本技巧”的文章。我将涵盖的 4 个主题是 1) lambda 函数，2) 列推导式，3) continue 和 break 语句，以及 4) 用户输入。你能为我生成关于 lambda 函数的部分吗？这部分应该大约 300 字，包含一个示例，并且要有亲切的语气，但不要过于随意。

### Lambda 函数：解锁 Python 的匿名英雄

Lambda 函数，也称为匿名函数，是 Python 最灵活的工具之一。虽然一开始它们可能听起来有些吓人，但它们的目的是简单的：创建小型、单次使用的函数，无需使用常规的 `def` 关键字或函数名。把它们想象成你的 Python 工具箱中的快速、一次性工具。

让我们分解一下。lambda 函数通过关键字 `lambda` 定义，后跟参数，一个冒号，以及要评估的表达式。以下是语法：

`lambda 参数: 表达式`

为什么使用它们？Lambda 函数在需要简单函数进行短期使用的情况下表现出色，例如在处理内置函数如 `map()`、`filter()` 或 `sorted()` 时。

这里有一个示例来说明：

```py
# Sort a list of tuples by the second element using lambda
data = [(1, 'apple'), (3, 'banana'), (2, 'cherry')]
sorted_data = sorted(data, key=lambda x: x[1])
print(sorted_data)
```

**输出**：`[(1, 'apple'), (3, 'banana'), (2, 'cherry')]`

在这个片段中，lambda 函数 `lambda x: x[1]` 提取每个元组的第二个元素，使 `sorted()` 能够按字母顺序排列数据。

虽然 lambda 函数非常方便，但它们也有一个缺点：可读性。过度使用它们或在 lambda 函数中塞入复杂的逻辑会使你的代码更难以理解。当你的逻辑变得更加复杂时，最好坚持使用具有描述性名称的常规函数。

从本质上讲，lambda 函数是一种快速编写简短、一次性代码的方法。作为一个初学者，掌握它们不仅会使你的代码更整洁，而且有助于你更好地理解 Python 如何拥抱简洁和强大。

让我们分解一下这两段之间的一些重要差异。在下面的每个项目符号中，我将描述差异，然后解释为什么它很重要。

+   人类版本以一个激励性问题开头，而人工智能版本则直接进入定义。**人们喜欢例子。**它们使难以理解的概念更容易理解，并使写作更具吸引力。值得指出的是，你可以**要求**人工智能从一个例子开始，但这忽略了基本点：与高质量的人类作家不同，人工智能默认不知道如何有效地编辑和调整写作以实现针对不同受众的最佳结果。没有以一个激励性的开头开始这个特定的摘录，只是这个更广泛现实的一个例子。

+   人工智能生成的版本未能解决这样一个事实：如果希望重用，lambda 函数**可以**被分配给一个变量，并且它们被限制在单次使用，句子“本质上，lambda 函数是编写简短、一次性代码的快捷方式。”**这两个缺陷——前者是重要信息的遗漏，后者是一个不必要的绝对陈述——都有可能误导那些通常是代码教程目标受众的新手程序员。**读者应该意识到有一种方法可以防止 lambda 函数立即消失，他们也应该理解虽然 lambda 函数**有时**可以用作一次性函数，但在某些编程环境中（例如在[函数式编程范式](https://en.wikipedia.org/wiki/Functional_programming)中）它们也可以是一个有用的工具。或者，即使这些细节对初学者来说可能不必要描述，教程绝对不应该明确指出 lambda 函数本质上总是一次性函数。

+   关于人工智能版本中的写作，**感觉**有些不对劲，但一开始很难确定具体是什么。**如果你仔细阅读并分析这两个摘录——就像在写作/修辞课程中做的那样——你会意识到人类版本流畅且易于跟随。**这是因为人类版本围绕一个单一的线索展开——处理数据集并使用 lambda 函数更有效地实现特定目标——这一线索贯穿整个摘录。其他所有内容，包括关于 lambda 的核心内容，都是围绕这个线索构建的。相比之下，人工智能版本是一系列松散相连的事实和例子。重要的是，尽管普通读者不会以这种方式分解文本；他们会在潜意识中注意到如果有什么**感觉**不对劲，这将对他们的学习和保留产生负面影响。

还有一些其他差异，例如 GPT 版本中缺少一个包含实际数据集的例子。这些差异并不那么严重——它们可以通过一些提示调整来纠正——但仍然值得注意，作为人工智能经常在回答中挣扎于细微差别和微妙之处的例子。

这让我回到了我的主要观点。人们一次又一次地声明，人类的创造力是无法被机器取代的。在这里，我同样提出这个观点，但是在本文的具体背景下。为什么我们特别需要人类的创造力来编写编程教程？

***因为人类会像你一样亲自引导你通过这个过程——一个活生生的、有思考能力的、试图学习一个令人困惑的新概念的人——而机器至多只能给你它所知道的事实，没有情感或激情。***

虽然表面上看不出这些有什么重要性，但这些事情确实很重要。“情感”和“激情”这两个词似乎与代码相去甚远，但上面提到的我的 lambda 示例中的每一个例子都落在这两个词的范畴之下。我们只是没有总是注意到。机器可能会用加重的词汇和一系列形容词来假装情感和激情，但最终，它们的写作还是不会一样。

有时候，这完全可以接受。我首先承认这一点。如果你只需要一些基本的 Python 代码来完成一个任务——那么没问题，使用 ChatGPT。但这本质上与去 Stack Overflow 上复制粘贴代码是一样的——我们多年来一直在做这件事，但我们并没有假装这样的资源使编程教程变得过时。为什么？因为这些教程填补了这类资源所没有提供的空白。长篇大论的详细说明，不仅仅是 GPT 给出的常见错误，还包括你在编码时可能遇到的精确错误。有意义的问题，不仅仅是 Claude 生成的示例数据，还有人类正在解决的真实、根本的问题。以及充满激情的写作，不仅仅是 Gemini 提供的枯燥、基于事实的说明，还有你对当前任务的同样情感投入。由人类编写的教程会给你这些。

人工智能不会。

这不是限制，而是一个机会。一个机会，如果我们可以花点时间认识到我们最初所重视的东西，那么像 Towards Data Science 这样的出版物中的作者和读者就能抓住这个机会。

我认为，这确实是一项值得追求的事业。
