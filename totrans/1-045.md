# 产品健康评分：通过统一监控和n8n自动化，我将关键事故减少了35%

> 原文：[https://towardsdatascience.com/the-product-health-score-how-i-reduced-critical-incidents-by-35-with-unified-monitoring-and-n8n-automation/](https://towardsdatascience.com/the-product-health-score-how-i-reduced-critical-incidents-by-35-with-unified-monitoring-and-n8n-automation/)

## <mdspan datatext="el1764204303854" class="mdspan-comment">简介</mdspan>

对于SaaS（软件即服务）公司来说，监控和管理他们的产品数据至关重要。对于那些未能理解这一点的人来说，等到他们注意到事故时，损害已经造成了。对于陷入困境的公司，**这可能是致命的**。

为了防止这种情况，我建立了一个与他们的数据库相连的n8n工作流程，该工作流程将每天分析数据，如果发现任何事故，将立即启动日志和通知系统进行调查。我还建立了一个仪表板，以便团队可以实时查看结果。

## 背景

一个专注于数据可视化和自动化报告的B2B SaaS平台，大约服务于4500个客户，分散在三个细分市场：

+   小型企业

+   中端市场

+   企业

周产品使用量超过30000个活跃账户，对实时数据（管道、API、仪表板、后台任务）有强烈依赖。

产品团队与以下方面紧密合作：

+   成长（获取、激活、入职）

+   收入（定价、ARPU、流失率）

+   SRE/基础设施（可靠性、可用性）

+   数据工程（管道，数据新鲜度）

+   支持和客户成功

![](../Images/1baf33a358ab8c086deed654dda6e6d7.png)

图片由Yassin Zehar提供

去年，公司观察到事故数量呈上升趋势。在10月到12月之间，事故总数从250增加到450，增长了**80%**。随着这一增长，有超过45起高和关键事故影响了数千名用户。受影响最大的指标是：

+   api_error_rate

+   checkout_success_rate

+   net_mrr_delta

+   data_freshness_lag_minutes

+   churn_rate

![](../Images/589eb6298df3dd19f67566779477bf3d.png)

当事故发生时，客户会根据公司如何处理和反应来评判公司。而**产品团队**因其如何管理并确保不会再次发生而受到尊敬。

> 一次事故可能发生，但同样的事故再次发生就是错误。

+   连续几周出现明显的**活跃账户下降**

+   多个企业客户报告并投诉过时仪表板（延迟高达45+分钟）

> 为什么这个问题是关键的？

作为数据平台，公司无法承受拥有：

+   慢或过时的数据

+   API 错误

+   管道失败

+   错过的或延迟的同步

+   不准确的仪表板

+   持续亏损（降级、取消）

内部，事件分散在几个系统中：

+   用于产品跟踪的 Notions

+   用于警报的 Slack

+   用于存储的 PostgreSQL

+   即使在 Google Sheets 的客户支持中

没有一个单一的真实来源。产品团队必须手动交叉引用和双重检查所有数据，寻找趋势并拼凑在一起。这是一项调查和解决谜题的工作，使他们每周损失许多小时。

> **解决方案**：使用 N8N 自动化事件系统警报并构建数据仪表板。因此，事件被检测、跟踪、解决并理解。

## 为什么选择 n8n？

目前，有几个自动化平台和解决方案。但并非所有都符合需求和要求。根据需求选择正确的一个是至关重要的。

具体的要求是无需 API 即可访问数据库（n8n 支持 API），拥有非技术人员可以理解的视觉工作流和节点，自定义节点、自托管选项且具有规模成本效益。因此，在 Zapier、Make 或 n8n 等现有平台中，选择的是最后一个。

## 使用 n8n 设计自动化检测系统

![](../Images/477379df25fa6f4d6f2edac8bf6a1830.png)

图片由 Yassin Zehar 提供

该系统由 **4 个流** 组成：

+   **流 1**：检索最近的收入指标，识别 MRR 持续亏损的不寻常峰值，并在需要时创建事件。

![](../Images/6888faacb53d97532b9cb6425e4b9fa8.png)

[PRE3]

+   **流 2**：监控功能使用指标，以检测采用或参与度的突然下降。

    事件以严重性、上下文和警报的形式记录给产品团队。

![](../Images/66cce41ef084de00e72456ffad1c9cbd.png)

图片由 Yassin Zehar 提供

+   **流 3**：对于每个开放的事件，从数据库中收集额外的上下文（例如，按国家或计划进行的持续亏损），使用 AI 生成清晰的根本原因假设和建议的下一步行动，向 Slack 和电子邮件发送总结报告，并更新事件

![](../Images/426620a58dddf2eec16aa35c72b84f5d.png)

图片由 Yassin Zehar 提供

+   **流 4**：每天早上，工作流汇总前一天的所有事件，创建用于文档的 Notion 页面，并向领导团队发送报告

![](../Images/bf9e4fae042d668b95209acc18d58ec8.png)

图片由 Yassin Zehar 提供

我们为8个不同的指标部署了类似的检测节点，根据增加或减少是否成问题来调整z-score方向。

**AI代理通过SQL查询接收额外的上下文**（按国家、按计划、按细分市场划分的流失率）以生成更准确的根本原因假设。所有这些数据都会收集并在每日电子邮件中发送。

工作流程生成每日汇总报告，按指标和严重程度汇总所有事件，通过电子邮件和Slack分发给利益相关者。

## 人工智能驱动的根本原因分析

![图片](../Images/782e39a09e2f178643b4b758e22c22fa.png)

图片由Yassin Zehar提供

使用AI可以节省大量时间。尤其是在需要在不同数据库中进行调查，而不知道从何开始时。在循环中添加AI代理可以**节省你相当多的时间**，因为它处理数据的速度很快。为了获得这个，需要一个**详细的提示**，因为代理将取代人类。所以，为了获得最准确的结果，即使是AI也需要理解上下文并得到一些指导。否则，它可能会进行调查并得出不相关的结论。不要忘记**确保你对问题的原因有全面的理解**。

[PRE4]

重要的是要注意，一旦获得结果，必须进行最终检查，以确保分析是正确进行的。**AI是一个工具**，但它也可能出错，所以不要只依赖它；它是一个有用的工具。对于这个系统，AI将建议每个事件的**前三个可能的原因**。

## 结论与收获。

总之，构建产品健康仪表板有多个好处：

+   更早地检测负面趋势（MRR、DAU、参与度）。

+   通过识别根本原因模式来减少关键事件。

+   理解真实业务影响（受影响的用户、流失风险）。

+   根据风险和影响优先考虑产品路线图。

+   围绕单一事实来源将产品、数据、SRE和收入对齐。

> 这正是许多公司所缺乏的：统一的数据方法。

## 产品团队的教训。

![图片](../Images/ad3a59e115ef658bdcde672a61042bdf.png)

图片由Yassin Zehar提供。

+   **从简单开始**：构建自动化系统和仪表板需要明确定义。你并不是在为顾客构建产品，而是在为你的合作者构建产品。了解每个团队的需求至关重要，因为它们是你的核心用户。考虑到这一点，首先构建一个能满足你所有需求的最小可行产品（MVP）。然后你可以通过添加功能或指标来改进它。

+   **统一指标比完美的检测更重要**：我们必须牢记，正是由于它们，我们才能节省时间，并加深理解。良好的检测是必不可少的，但如果指标不准确，节省的时间将被寻找散布在不同环境中的指标的团队浪费掉。

+   **自动化每周节省10小时**的手动调查时间：通过自动化一些手动和重复性任务，你将节省调查时间，就像在事件警报工作流程中，我们直接知道首先调查哪里，以及原因的假设甚至可以采取的一些行动。

+   **记录一切**：一份恰当且详细的文档是必不可少的，它将使所有相关方对正在发生的事情有一个清晰的理解和观点。文档也是数据的一部分。
