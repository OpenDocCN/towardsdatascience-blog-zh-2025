## **Looker 表达式缺乏 DAX 公式的强大功能和功能。**

> *(Looker **1:1** Power BI)*

然而，我不喜欢 Looker 表达式。

与 [DAX（数据分析表达式）](https://learn.microsoft.com/en-us/dax/) 相比，它们在函数数量方面的发展程度并不相同。

DAX 提供了一个丰富的函数库，用于复杂计算和时间智能，这对于高级分析至关重要。

我在 Power BI 中最喜欢的 DAX 函数无疑是 [=CALCULATE 函数](https://learn.microsoft.com/en-us/dax/calculate-function-dax)。`CALCULATE` 改变筛选条件以给出你想要的结果——它对于基于时间或特定类别的计算等非常有用：

```py
Previous_Month_Gross_Volume = 
CALCULATE(
    [gross_volume], 
    DATEADD(dim_date[Date], -1, MONTH),
    dim_product[Category] = "Electronics",
    dim_region[Region] = "North America"
)
```

虽然 Looker 提供了一系列用于数据操作的表达式，它们称之为 [Looker 表达式](https://cloud.google.com/looker/docs/functions-and-operators)（有时称为 Lexp），但它们并不匹配 Power BI 中 DAX 的深度和多功能性。

> 与 DAX 相比，Looker 的表达式被采用得较少，在线论坛、文档和教育资源中的存在感也较小。这也因为 DAX 还用于 Analysis Services 和 Excel 中的 Power Pivot。

我还注意到，生成式 AI 模型有时会难以处理 Looker 表达式的精确结构/公式，因为它们是在 DAX 之后引入的——即使为模型设置了正确的上下文和额外的参数。

> 还有一点需要记住的是，Looker 表达式与 [LookML 语言](https://cloud.google.com/looker/docs/write-lookml-intro) 并不相同——这可能是某些人产生混淆的地方。
> 
> Looker 表达式用于执行计算，并使用你在 LookML 中定义的字段。

例如，当你将一个字段添加到表达式中时，**Looker 使用该字段的 LookML 标识符**，其形式类似于 `${view_name.field_name}`。

## **Looker 仅基于 Web。**

> *(Looker **1:3** Power BI)*

我喜欢 Power BI 作为一个桌面应用程序，这让我可以使用***离线***和***本地***数据集工作，并利用我本地机器的处理能力！

![Power BI 是一个桌面应用程序，而 Looker 不是（来源：本生产）](img/e4a3000e5a5436e4d21aad182c92f2ca.png)

Power BI 是一个桌面应用程序，而 Looker 不是（来源：本生产）

Looker 不是一个桌面应用程序——它仅基于 Web。如果你更喜欢 Mac，基于 Web 是一个优势——特别是考虑到 Power BI 的桌面应用程序仅适用于 Windows。

> 我必须说，我喜欢 Looker 的一点是，您可以使用浏览器上的前后按钮轻松地导航到您的更改，无论是向前还是向后。这是因为 Looker 探索中的每个更改都会创建一个新的唯一 URL 链接。当您修改查询或调整设置时，URL 会更新以反映您探索的当前状态。这与 Looker 的缓存功能相辅相成，允许您再次访问查询和视觉元素，而无需再次运行它们，因此它们可以即时加载（默认缓存保留策略为 [一小时](https://cloud.google.com/looker/docs/caching-and-datagroups#:~:text=Warning%3A%20For%20queries%20run%20on,retention%20policy%20is%20one%20hour.)）。

虽然 ***Power BI 服务*** – 用户在此处上传和编辑报告、协作和创建应用程序，也是基于 Web 的，但 Looker 的完全基于浏览器的方法完全消除了操作系统限制。

> *(Looker **3:6** Power BI)*

## **使用 LookML 进行数据建模确保了组织内度量值和维度的统一。**

> *(Looker **4:6** Power BI)*

Looker 非常出色，我喜欢它使用专有的 LookML 语言进行集中式数据建模的方法。

> LookML 代码允许您在组织内部署中保持定义的一致性。在实践中，这意味着因为我们所有人都是作为数据分析师/科学家用一种语言编写一切，我们最终都会使用由我们的团队定义和批准的相同度量值和维度。简单来说，Looker 允许您在数据仓库中预建模数据的基础上构建内容。

![在您的视图中（=表）中定义的维度和度量值使用 LookML，这确保了在不同模型中只有一个定义（来源：自产）](img/9cdcce28aae7868711df8778359ea8b5.png)

在您的视图中（=表）中定义的维度和度量值使用 LookML，这确保了在不同模型中只有一个定义（来源：自产）

您的平均员工无法定义内部的数据模型，但将能够自由地使用这些数据。这旨在创建组织内每个人都想要的单一事实来源。

> 技术上，当您更改业务中指标的定义时，您应该能够在一个地方编辑您的 LookML，并且使用该指标的所有探索、查看和仪表板都将自动更新。

我还喜欢 Looker 的地方是，当您创建视图时，可以使用 LookML 中的 `sql:` 参数进行 SQL 编辑（视图反映了从您的数据仓库加载到 Looker 的表）。

+   **报告 → 仪表板 → 应用**

+   **探索 → Look → 仪表板。**

1.  这些 Look 包括图表，可以之后拼接到一起形成 ***仪表板***，以全面概述您的关键指标。

> 我非常喜欢 Looker 中的 Explorers，因为它们赋予组织中的任何人都深入数据的能力，而无需了解 SQL。当你作为一个 Looker 开发者准备 Explorers 以供使用时，它使得财务专家、市场营销经理、收入总监——实际上任何人——都可以与 Explorers 中的准备视图进行交互。他们不需要技术专业知识或 SQL 知识——他们可以提出自己的问题，并通过旋转、切片和过滤来获得即时答案。

当使用 Looker 时，任何人的即席探索选项是组织拥有的非常有用的工具。

## Power BI 可以免费使用 - Looker 则不行。

> *(Looker **5:7** Power BI)*

尽管有一个名为 ***Looker Studio*** 的免费工具，但它与功能齐全的 ***Looker 企业 BI 平台*** 有显著差异——这使得学习 Looker 非常具有挑战性。

然而，我发现对于获得 Looker 的实际操作经验来说，[互动的 Google Cloud 实验室](https://www.cloudskillsboost.google/paths/28)（这些实验室允许你在模拟、时间有限的会话中尝试 Looker - 此外，完成这些实验室还可以获得 Google 徽章！）非常有帮助。

> Power BI Desktop 可以完全免费下载作为独立的桌面应用程序，而且其付费的企业版本与 Looker 相比更具成本效益，因为随着分析团队的扩大，Looker 的成本可能会螺旋式上升。

## Gartner 表示，Power BI 是当前的选择。

> *(Looker **5:8** Power BI)*

根据 Gartner 2024 年的[分析和商业智能平台魔力象限](https://www.gartner.com/doc/reprints?id=1-2HVUGEM6&ct=240620&st=sb)，微软的 Power BI 在其与微软生态系统的集成、功能性和性价比方面脱颖而出。

## 作为高级数据分析师，我的偏好。

我已经是一个重度的 Power BI 用户 4-5 年了，我认为它是一个出色的工具。它易于使用，功能丰富，非常适合想要立即开始构建的人。***在功能方面，我更喜欢 Power BI。***

但现在我已经过渡到 Looker 并花时间了解它是如何工作的，***我开始看到 Looker 有可能成为我未来将使用的 #1 BI 工具。***

如果我在功能上给 Power BI 打了更高的分，我为什么会这么说？***这一切都归结于公司的环境（正如 Gartner 研究也指出的一样）。***

如果更多公司开始采用 BQ 及其云市场份额增长，它会产生一种锁定效应。即使你换工作，你也可能会再次遇到 Looker。所以你的#1 BI 工具将由你决定（当然，也可以反过来适用于 Azure 和 PowerBI）。

在此之前，我曾在[Zensai](https://zensai.com/)（原名 LMS365）工作，在那里（https://cloud.google.com/customers/trustpilot）我们严重依赖 Microsoft 生态系统——甚至我们的产品都集成到了 Microsoft 中。但现在在[Trustpilot](https://business.trustpilot.com/jobs)工作，由于我们使用 Google Cloud Platform（GCP），Looker 更有意义——您可以在这里了解更多我们如何使用 GCP。

> 如果您的公司已经使用 GCP 和 BigQuery，Looker 可以完美集成。它简化了所有事情——从探索数据、创建图表到调试问题。它并不完美（我可以没有定制选项的缺乏、随机错误、基于 Web 的缓慢以及几乎需要为所有事情编写 LookML 代码的需求），但在 Google Cloud 环境中，我认为 Looker 是一个很好的选择，因为它变成了一种非常简单的工具，用于创建图表、在构建仪表板之前探索您的数据，以及快速调试——这些是我最看重的任何可视化工具的品质。

话虽如此，Power BI 和 Looker 之间的选择实际上取决于您的公司使用什么作为其 1) ***数据仓库***，2) ***人力资源***，3) ***前员工经验***，以及您的 4) ***财务资源***。
