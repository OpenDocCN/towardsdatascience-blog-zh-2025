**总结**是 LLM（大型语言模型）中最实用和方便的任务之一。然而，与其他 LLM 任务（如提问或分类）相比，评估 LLM 在总结方面的挑战要大得多。

因此，我自己也忽视了总结的评估，尽管我开发的两个应用程序严重依赖总结（*Podsmart*总结播客，而*[aiRead](https://www.airead.me/)*根据您的突出显示创建个性化的 PDF 总结）。

但最近，我受到了启发——多亏了 AI 行业思想领袖的深刻见解——评估在系统地评估和改进 LLM 系统中的关键作用。([[链接](https://applied-llms.org/)](https://hamel.dev/blog/posts/evals/) 和链接）。这激励我开始调查总结的评估。

因此，在这篇文章中，我将讨论一个**易于实现、基于研究和定量的框架来评估总结**，该框架改进了 Confident AI 创建的[DeepEval](https://docs.confident-ai.com/docs/getting-started)框架中的总结指标。

我将通过一个示例笔记本（代码在[Github](https://github.com/thamsuppp/summary-eval-article)）来说明我的评估过程，尝试评估一篇约 2500 字的关于“Securing the AGI Laurel: Export Controls, the Compute Gap, and China’s Counterstrategy”的约 500 字总结（可在[这里](https://www.csis.org/analysis/securing-agi-laurel-export-controls-compute-gap-and-chinas-counterstrategy)找到，发表于 2024 年 12 月）。

### **为什么评估总结是困难的**

在我开始之前，让我详细说明一下为什么我认为总结是一个难以评估的任务。

首先，总结的输出本质上是开放式的（与分类或实体提取等任务相反）。因此，什么使一个总结变得优秀取决于定性指标，如流畅性、连贯性和一致性，这些指标难以进行定量测量。此外，这些指标往往是主观的——例如，相关性取决于上下文和受众。

其次，创建用于评估你的系统摘要的黄金标签数据集很困难。对于 RAG 来说，创建一个用于评估检索器的合成问题-答案对数据集是直接的（参见这个[很好的教程](https://playbooks.capdev.govtext.gov.sg/appendix_dataset/)）。

对于摘要来说，没有明显的生成参考摘要的自动方法，所以我们不得不求助于人类来创建它们。虽然研究人员已经整理了摘要数据集，但这些数据集不会针对你的用例进行定制。

第三，我发现学术文献中大多数的摘要度量标准并不适合以实用为导向的 AI 开发者实现。一些论文训练了神经摘要度量标准（例如[Seahorse](https://github.com/google-research-datasets/seahorse)，[Summac](https://github.com/tingofurro/summac)等），这些度量标准有数 GB 大，并且在大规模运行时具有挑战性（*也许我只是懒惰，应该学习如何在本地和 GPU 集群上运行 HuggingFace 模型，但无论如何，这对大多数人来说仍然是一个入门障碍*）。其他传统度量标准，如 BLEU 和 ROUGE，依赖于精确的单词/短语重叠，并且是在 LLM 时代之前为提取式摘要而创建的，可能不适合评估 LLM 生成的抽象摘要，因为 LLM 可能会改写原文。

然而，根据我的经验，人类可以很容易地区分好的摘要和不好的摘要。一种常见的失败模式是含糊其辞和拐弯抹角（例如，“*这个摘要描述了……的原因”）。*

### **什么是好的摘要**

那么，什么是好的摘要呢？Eugene Yan 的[文章](https://eugeneyan.com/writing/abstractive/)对各种摘要度量标准提供了很好的细节。对我来说，我会把它们提炼成 4 个关键品质：

1.  **相关性** – 摘要保留了原文中的重要观点和细节

1.  **简洁性** – 摘要信息密集，不重复相同的观点，也不冗长

1.  **连贯性** – 摘要结构良好，易于理解，不仅仅是压缩的事实杂乱无章

1.  **忠实性** – 摘要不胡编乱造原文不支持的信息

一个关键的见解是，你实际上可以将前两个问题表述为**精确度和召回率**问题——摘要中保留了原文中多少事实（**召回率**），以及摘要中有多少事实得到了正文的支持（**精确度**）。

这种公式使我们回到了机器学习中的分类问题更为熟悉的领域，并提出了一个评估摘要的量化方法。

这里的一些差异包括：首先，更高的召回率更好，*保持摘要长度不变*。你不想得到 100%的召回率，而摘要的长度与原文相同。其次，理想情况下，你希望精确度尽可能接近 100%——胡编乱造信息是非常糟糕的。我稍后会回到这些点上。

### **DeepEval 简介**

在所有不同的 LLM 评估框架中，你会有很多选择——从 Braintrust 到 Langfuse 等等。然而，今天我将使用 DeepEval，这是一个非常用户友好的框架，可以快速入门，无论是总体上还是特别针对摘要。

DeepEval 提供了许多关键 RAG 指标的开箱即用实现，并且有一个基于思维链的 LLM 作为裁判的工具 GEval，让你可以定义任何你想要的定制标准（我稍后会使用这个工具）。

此外，它还提供了有用的基础设施来组织和加速评估：他们已经很好地使用异步并行化了所有操作，因此你可以快速运行整个数据集的评估。他们有方便的**合成**数据生成功能（将在后续文章中介绍），并允许你定义**自定义指标**来调整他们的指标（这正是我们今天要做的），或者定义基于非 LLM 的评估指标以实现更经济高效且更稳健的评估（例如，实体密度，稍后介绍）。

### **DeepEval 的摘要度指标**

DeepEval 的摘要度指标（更多关于它的信息[在这里](https://docs.confident-ai.com/docs/metrics-summarization)）是一个无参考指标（即不需要黄金标准的摘要），只需要源文本（你将其作为`input`字段）和要评估的生成摘要（`actual_output`字段）。正如你所看到的，下面的设置和评估代码非常简单！

```py
# Create a DeepEval test case for the purposes of the evaluation
test_case = LLMTestCase(
  input = text,
  actual_output = summary
)

# Instantiate the summarization metric
summarization_metric = SummarizationMetric(verbose_mode = True, n = 20, truths_extraction_limit = 20)

# Run the evaluation on the test case
eval_result = evaluate([test_case], [summarization_metric])
```

摘要度指标实际上在底层评估了两个独立的组件：**对齐**和**覆盖**。这些与之前我介绍的**精确度**和**召回率**公式密切相关！

对于对齐，评估 LLM 从摘要中生成一系列**声明**，对于每个声明，LLM 将确定有多少这些声明得到了从源文本中提取的**真相**的支持，从而产生**对齐得分**。

在覆盖度的情况下，LLM 从源文本中生成一系列评估问题，然后尝试回答这些问题，只使用摘要作为上下文。如果找不到答案，LLM 将被提示回答“idk”。然后，LLM 将确定有多少这些答案是正确的，以获得**覆盖得分**。

最终的摘要度得分是对齐得分和覆盖得分的最低值。

### **改进摘要度指标**

然而，尽管 DeepEval 所做的是一个非常好的起点，但当前形式的摘要度指标存在三个关键问题，阻碍了其可靠性和有用性。

因此，我构建了一个**自定义摘要度指标**，它适应了 DeepEval 的版本。以下，我将解释每个问题和为克服这些问题而实施的相应解决方案：

**1: 使用是/否问题作为覆盖度指标过于简单**

目前，评估问题被限制为是/否问题，其中问题的答案是肯定的——看看这些问题：

![作者提供的图片](img/11a7b191c5f92032a7229b4ac9f8d834.png)

作者提供的图片

这有两个问题：

首先，将问题框定为二元是/否，这限制了它们的信息量，特别是在确定细微的定性观点方面。

其次，如果 LLM 在给出摘要时产生了“是”的幻觉答案（因为只有三种可能的答案：“是”、“否”、“不知道”，它产生“是”的幻觉并不罕见），评估者会错误地认为这个答案是正确的。对于开放式问题，产生正确答案的幻觉要困难得多。此外，如果你看看这些问题，它们是以一种几乎暗示答案为“是”的方式提出的（例如，“中国是否将信息不透明作为策略？”），这增加了产生幻觉“是”的可能性。

我的解决方案是让 LLM**从源文本生成开放式问题**——在代码中，这些被称为“复杂问题”。

此外，我还让 LLM 分配一个**问题的权重**（这样我们可能可以在覆盖率分数中提高更重要问题的权重）。

由于问题现在是开放式的，我使用了一个**LLM 进行评估**——我让 LLM 给出一个**0-5 分的相似度评分**，以判断从摘要生成的答案与使用源文本（参考答案）生成的答案的相似程度，以及一个解释。

```py
def generate_complex_verdicts(answers):
    return f"""You are given a list of JSON objects. Each contains 'original_answer' and 'summary_answer'.
    Original answer is the correct answer to a question. 
    Your job is to assess if the summary answer is correct, based on the model answer which is the original answer.
    Give a score from 0 to 5, with 0 being completely wrong, and 5 being completely correct.
    If the 'summary_answer' is 'idk', return a score of 0.

    Return a JSON object with the key 'verdicts', which is a list of JSON objects, with the keys: 'score', and 'reason': a concise 1 sentence explanation for the score.
..."""

def generate_complex_questions(text, n):
        return f"""Based on the given text, generate a list of {n} questions that can be answered with the information in this document.
        The questions should be related to the main points of this document. 
        Then, provide a concise 1 sentence answer to the question, using only information that can be found in the document.
        Answer concisely, your answer does not need to be in full sentences.
        Make sure the questions are different from each other. 
        They should cover a combination of questions on cause, impact, policy, advantages/disadvantages, etc.

        Lastly, rate the importance of this question to the document on a scale of 1 to 5, with 1 being not important and 5 being most important. 
        Important question means the question is related to an essential or main point of the document,
        and that not knowing the answer to this question would mean that the reader has not understood the document's main point at all.
        A less important question is one asking about a smaller detail, that is not essential to understanding the document's main point.

 ..."""
```

**2: 从源文本提取真理以进行对齐是有缺陷的**

目前，对于对齐指标，使用 LLM（一个可以控制的参数`truths_extraction_limit`）从源文本中提取一系列**事实**。这导致源文本中的一些事实/细节被省略在事实之外，然后摘要的陈述与这些事实进行比较。

说实话，我不确定团队当时是如何想的，他们这样实现——也许我错过了一个细微之处或误解了他们的意图。

然而，这导致了两个问题，使得对齐分数“不可用”，正如一个 GitHub 用户[指出](https://github.com/confident-ai/deepeval/issues/937)。

首先，LLM 生成的真理列表是非确定性的，因此[人们已经报告](https://github.com/confident-ai/deepeval/issues/565)对齐分数变化很大。这种不一致性可能源于 LLM 每次选择不同的真理子集。更重要的是，真理提取过程使得这并不是对摘要忠实度的一个公平评判，因为摘要中的一个细节可能在源文本中找到，但不在提取的真理中。据观察，所有被检测为不忠实的主张确实在正文中有，但不在提取的真理中。此外，人们已经报告说，当你将摘要作为输入传递时，对齐分数小于 1，这是奇怪的。

为了解决这个问题，我进行了简单的调整——即将**整个源文本**传递给评估摘要声明的 LLM，而不是传递事实列表。由于所有声明都在一个 LLM 调用中一起评估，这不会显著增加 token 成本。

**3: 最终得分取最小值（对齐得分和覆盖得分）是有缺陷的**

目前输出的得分是对齐得分和覆盖得分中的最小值（实际上没有方法可以访问这些得分，除非将其放入日志中）。

这是有问题的，因为覆盖得分可能会低于对齐得分（如果不是这样，那么就真的有问题了！）这意味着对齐得分的改变不会影响最终得分。然而，这并不意味着我们可以忽视对齐得分的下降（例如从 1 下降到 0.8），这可能会被认为是摘要（即产生幻觉）的一个更严重的问题的信号。

我的解决方案是将**最终得分改为 F1 得分**，就像在机器学习分类中一样，以捕捉精确度和召回率的重要性。一个扩展是改变精确度和召回率的权重。（例如，如果你认为幻觉是无论如何都要避免的事情，可以增加精确度的权重——参见[这里](https://stats.stackexchange.com/questions/559736/is-there-a-metric-that-combines-recall-and-precision-other-than-the-f1-score)）

经过这 3 个改动，摘要指标现在更好地反映了生成摘要的相关性和忠实度。

### **简洁性指标**

然而，这仍然提供了一个不完整的图景。摘要还应该**简洁**且信息密集，将关键信息压缩成更短的形式。

**实体密度**是一个有用且成本低的指标。密度链论文显示，人类创建的摘要以及人类偏好的 AI 生成的摘要，其实体密度约为 0.15 实体/标记，在清晰度（倾向于密度较低）和信息量（倾向于密度较高）之间取得了平衡。

因此，我们可以创建一个**密度得分**，惩罚实体密度远离 0.15 的摘要（要么过于密集，要么不够密集）。初始的 AI 生成的摘要通常密度较低（0.10 或更低），密度链[论文](https://arxiv.org/pdf/2309.04269)展示了一个迭代过程来增加摘要的密度。Ivan Leo 和 Jason Liu 撰写了一篇很好的[文章](https://python.useinstructor.com/blog/2023/11/05/chain-of-density/)，介绍了使用实体密度作为关键指标微调密度链摘要的方法。

```py
import nltk
import spacy
nlp = spacy.load("en_core_web_sm")

def get_entity_density(text):
  summary_tokens = nltk.word_tokenize(text)
  num_tokens = len(summary_tokens)
  # Extract entities
  doc = nlp(text)
  num_entities = len(doc.ents)
  entity_density = num_entities / num_tokens
  return entity_density
```

接下来，我使用一个**句子模糊度**指标来明确惩罚模糊的句子（例如，“*这个摘要描述了……的原因”*），这些句子实际上并没有陈述关键信息。

为了做到这一点，我将摘要分解成句子（类似于对齐指标），并让一个 LLM（大型语言模型）判断每个句子是否模糊，最终得分是被评为模糊的句子的比例。

```py
prompt = ChatPromptTemplate.from_template(
    """You are given a list of sentences from a summary of a text.
    For each sentence, your job is to evaluate if the sentence is vague, and hence does not help in summarizing the key points of the text.

    Vague sentences are those that do not directly mention a main point, e.g. 'this summary describes the reasons for China's AI policy'. 
    Such a sentence does not mention the specific reasons, and is vague and uninformative.
    Sentences that use phrases such as 'the article suggests', 'the author describes', 'the text discusses' are also considered vague and verbose.
  ...
    OUTPUT:"""
)

class SentenceVagueness(BaseModel):
    sentence_id: int
    is_vague: bool
    reason: str

class SentencesVagueness(BaseModel):
    sentences: List[SentenceVagueness]

chain = prompt | llm.with_structured_output(SentencesVagueness)
```

最后，重复相同信息的摘要是不高效的，因为它浪费了本可以用来传达新有意义见解的宝贵空间。

因此，我们使用**GEval**构建了一个**重复性**得分。如我上面简要提到的，GEval 使用具有思维链的 LLM 作为评判者来评估任何自定义标准。由于检测重复概念是一个更复杂的问题，我们需要一个更智能的检测器，也就是一个 LLM。（*警告：这个指标的某些结果似乎相当不稳定——当我在相同的输入上反复运行时，LLM 会改变其答案。也许可以尝试一些提示工程）*

```py
from deepeval.metrics import GEval
from deepeval.test_case import LLMTestCaseParams

repetitiveness_metric = GEval(
    name="Repetitiveness",
    criteria="""I do not want my summary to contain unnecessary repetitive information.
    Return 1 if the summary does not contain unnecessarily repetitive information, and 0 if the summary contains unnecessary repetitive information.
    facts or main points that are repeated more than once. Points on the same topic, but talking about different aspects, are OK. In your reasoning, point out any unnecessarily repetitive points.""",
    evaluation_params=[LLMTestCaseParams.ACTUAL_OUTPUT],
    verbose_mode = True
)
```

### **一致性指标**

最后，我们希望确保 LLM 输出的连贯性——有一个逻辑流程，将相关点放在一起，并实现平滑过渡。Meta 最近的大型概念模型[论文](https://arxiv.org/pdf/2412.08821)使用了 Parola 等人（2023）的局部一致性指标——每个第 n 句和第 n+2 句之间的平均余弦相似度。这是一个简单且易于实现的指标。我们发现 LLM 摘要的得分为~0.45。作为一个感觉检查，如果我们随机排列摘要中的句子，连贯性分数会低于 0.4。

```py
# Calculate cosine similarity between each nth and n+2th sentence
def compute_coherence_score(sentences):
  embedding_model = OpenAIEmbeddings(model="text-embedding-3-small")
  sentences_embeddings = embedding_model.embed_documents(sentences)
  sentence_similarities = []
  for i in range(len(sentences_embeddings) - 2):
    # Convert embeddings to numpy arrays and reshape to 2D
    emb1 = np.array(sentences_embeddings[i])
    emb2 = np.array(sentences_embeddings[i+2])
    # Calculate cosine distance
    distance = cosine(emb1, emb2)
    similarity = 1 - distance
    sentence_similarities.append(similarity)
  coherence_score = np.mean(sentence_similarities)
  return coherence_score
```

### **整合一切**

我们可以将上述每个指标打包成自定义指标。好处是，我们可以在你的摘要数据集上并行评估所有这些指标，并在一个地方获得所有结果！（见[代码笔记本](https://github.com/thamsuppp/summary-eval-article/blob/main/summary_eval_article_notebook.ipynb)）

然而，有一个注意事项，即对于一些这些指标，如一致性或召回，我们并没有一个关于摘要“最佳”值的感知，我们只能比较不同 AI 生成摘要的分数，以确定哪个更好或更差。

### **未来工作**

我在这篇文章中介绍的内容为评估你的摘要提供了一个坚实的起点！

虽然它并不完美，但仍有未来探索和改进的空间。

一个方面是更好地测试摘要是否捕捉到了源文本中的**重要点**。你不想得到一个召回率很高的摘要，但其中包含的是不重要的细节。

目前，当我们生成评估问题时，我们会要求 LLM 对其**重要性**进行评级。然而，将这些重要性评级作为事实真相也并非易事——如果你这么想的话，当 LLMs 进行总结时，它们本质上也在评估不同事实的重要性。因此，我们需要一个**LLM 之外的**重要性度量。当然，理想的情况是拥有人类参考摘要，但这些成本高昂且不可扩展。另一个参考摘要的来源是带有执行摘要的报告（例如，财务提案、幻灯片摘要、论文摘要）。我们还可以使用像嵌入体的 PageRank 这样的技术来算法性地识别中心概念。

一个值得尝试的有趣想法是生成**合成源文章**——从一个给定主题的主要观点（代表真实“重要”点）开始，然后要求 LLM 扩展成完整的文章（多次以高温度运行以生成许多多样化的合成文章！）然后运行完整的文章通过摘要过程，并评估摘要是否保留了原始的主要观点。

最后但同样重要的是，确保我介绍的每个摘要度量指标都与人类对摘要偏好的评估相关联非常重要。虽然研究人员已经在大型摘要数据集上对一些指标进行了这样的操作，但这些发现可能并不适用于你的文本和/或受众。（也许你的公司更喜欢特定风格的摘要，例如包含许多统计数据）。

关于这个话题的精彩讨论，请参阅 Hamel Husain 关于评估的文章中的“Level 2”部分。[评估文章](https://hamel.dev/blog/posts/evals/#automated-evaluation-w-llms)。例如，如果你发现 LLM 的句子模糊度评分与你认为的模糊句子相关性不好，那么一些提示工程（提供模糊句子的例子，进行更多阐述）可能会希望提高相关性。

虽然这一步可能很耗时，但它是必要的，以确保你可以信任 LLM 评估。从长远来看，这会节省你的时间——当你的 LLM 评估对齐时，你实际上获得了一个无限可扩展的、定制于你的需求和偏好的评估者。
