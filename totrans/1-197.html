How to Build Tools for AI Agents

原文：https://towardsdatascience.com/how-to-build-tools-for-ai-agents/

AI agents are programs performing a series of LLM calls. These agents typically receive a task from a user and utilize tools to effectively solve the task. Tools are essentially functions that an agent can invoke. However, building an agent is much more than simply defining a set of tools and providing them in the agent’s context. You need to make careful considerations of how your tools are defined, and which context you are feeding to your agent

The goal of this article is to highlight what to focus on when building tools for AI agents. Proper tools are essential for high-performing agents, and I’ll thus make a deeper dive into agentic tool usage. I’ll cover topics such as good tool definitions and descriptions, and the optimal way to make tools available to your agent.

Why tools are important for AI agents

Having tools available is a large part of what makes agents effective. Without access to tools, the LLM wouldn’t have the same options, such as performing internet searches or finding information in a database table. How you define and utilize your agentic tools is thus essential for the performance of your agent.

There are various considerations to make when defining tools for your agents. From working with agents myself, I find that a lot of these considerations also apply to humans using the tools as well, for example:

  • Tools should be well-defined with proper naming and descriptions
  • A tool should have one specific purpose
  • You should have proper type definitions, both for what the tool inputs and outputs

In general, I think you can come a long way by following the principle quoted below. All tools you create should be defined in such a way that a human finds it easy to understand the tool and how it works.

Tools should be defined in such a way that it’s easy to utilize for a human

I find that this principle applies a lot when working in ML, for example, also when analyzing the prompts you feed to your LLMs. You should always ask yourself the question of whether a human can understand the task provided in the prompt. If not, the prompt needs to be clarified.

Proper tool definitions

Proper tool definitions are a simple improvement you can make to your AI agents. With proper tool definitions, I refer to:

  • You’re tools should have a clear name, representing the function of the tool
  • You’re tools should have a well-descriptive docstring, including a short description of what the tool does, a description of all input parameters along with their type, and a definition of what the tool returns

I’ll first show a bad example of a tool, highlight the issues, and then provide an example of a good tool definition.

# bad tool definition
def search(query):
    results = search_database() 
    return results

This is a bad tool definition because of the following issues:

  • The tool naming is not descriptive. Search is ambiguous; it could, for example, also refer to performing a semantic search
  • It’s missing a docstring defining input and output types, and what the tool does
  • The tool is missing types for both input and output parameters. The LLM utilizing this tool is probably able to infer that the query is a string. However, the model will have to spend time understanding the output format once it uses the tool

Instead, you should thus define the tool as follows:

@dataclass
class KeywordSearchResult:
    id: str
    filename: str
    document_content: str
    

# good tool definition
def keyword_search(query: str) -> list[KeywordSearchResult]:
    """
    Performs keyword search in the database. 

    Input parameters:
        query: str - the keywords to search for

    Output:
        A list of all keyword search results, with each result containing:
            - id (str) - the id of the document as defined in the database
            - filename (str) - the filename of the document
            - document_content (str) - the text contents of the document
    """
    results = search_databse()
    return results

In this example, I made a separate data class, representing the output format of the tool. This makes it a lot easier for your agent to understand how the tool works, and will make it easier to handle the output. Furthermore, I defined a proper docstring, which contains a simple description of what the tool does, and the input and output parameters. I made the tool name more specific by calling it keyword_search instead of just search.

Making these simple improvements to your tools will yield large increases in your agents’ performance. However, there are also other techniques you can apply to how you define your tools, which makes your agent even more effective.

Tool functionality

Continuing on the topic of your tool, you can also improve agentic performance by making tools more specific and providing the model with clean outputs from the tools. I’ll continue with the keyword example to specify what I mean.

Making specific tools

Tools should be as specific as possible. Creating vague tools makes it harder for the model to know when to utilize the tool. You’ll therefore more often encounter the model incorrectly utilizing the tool. This could, for example, be:

  • Using the tool at incorrect times. For example, using keyword_search when it should be using semantic_search
  • Using the model with incorrect parameters
  • Incorrectly handling the output of the tool

Your tools should thus always have a clear, singular purpose.

Provide clean outputs

Another way to greatly improve the quality of your agents’ tools is to provide the agent with a clean output. This means parsing the results within the tool into a string that is structured in an easy way for the agent to understand. I’ll again describe this using an example below.

I’ll make a _parse_keyword_search_output tool that takes in the keyword search outputs and parses them into a structured string. In this example, I also include that the KeywordSearchResult contains some unnecessary results. These results are naturally excluded when we parse the output.

@dataclass
class KeywordSearchResult:
    id: str
    filename: str
    document_content: str
    uneccesary_field_a: str
    uneccesary_field_b: str    

def _parse_keyword_search_output(keyword_search_results: list[KeywordSearchResult]) -> str:
    """Parse the output from the keyword search tool, into a structured string, only containing necessary information"""
    output_string = ""
    tot_num_documents = len(keyword_search_results)


    for i, result in enumerate(keyword_search_results):
        document_id, filename, document_content = result["id"], result["filename"], result["document_content"]
        output_string += f"""\n\n
                            Document {i+1}/{tot_num_documents}:
                                ID: {document_id}
                                filename: {filename}
                                content: {document_content}
                        """
    return output_string

Parsing the output like this makes it much easier for the agent to handle the outputs of the keyword search tool. Remember to ensure that the docstring of the keyword_search tool describes the output format that the parse keyword search tool will provide.

Avoid returning all results

When using tools such as keyword search, the tools can sometimes return hundreds, if not thousands, of results. This will immediately bloat the context of your agent. To prevent this, you should add options to sort the outputs of the tool and to only return a maximum number of items, as I highlight in the example below:

def keyword_search(query: str, url: str, sort_ascending=True, max_return_items=10) -> list[KeywordSearchResult]:
    """<docstring>"""
    
    results = search_databse()
    if sort_ascending:
        results = sort(results, ascending=True)
    else:
        results = sort(results, ascending=False)

    results = parse_results(results)

    if max_return_items < len(results):
        return results[:max_return_items]
    return results

Usually, it’s enough for the model to see the top 10 or 20 results, to both fetch the most useful information (in keyword searches the top results are usually the most relevant ones), and to understand the format of the output

Informative error handling

Informative error handling is a critical failsafe to have in place when your agents inevitably encounter issues. This issue can be caused by:

  • The agent incorrectly uses a tool, in which case you should inform the agent why it used the tool incorrectly, and potentially also how to fix it
  • An API fails because of a third-party provider. In which case, you might inform the model to either wait or inform the user of the issue
  • Missing packages, in which case you should inform your agent to install the required packages
def keyword_search(query: str, url: str) -> list[KeywordSearchResult]:
    """<docstring>"""
    try:
        results = search_database()
        return parse_results(results)
    except RatelimitError as e:
        raise RuntimeError(
            f"Rate limit error: {e}. "
            f"Wait before retrying. If this error persists, contact support."
        )
    except APINotAvailableError as e:
        raise RuntimeError(
            f"API not available: {e}. "
            f"Check that the provided URL is correct and the endpoint is operational."
        )
    except Exception as e:
        raise RuntimeError(
            f"Unexpected error: {e}. Please try again."
        )

This informative error handling makes it much easier for your agent to handle the scenario where a tool fails. You should, in general, ensure the following in your error handling:

  • You return a descriptive error message, including both the error trace and a string describing the error
  • You should ideally inform the agent how it should act after receiving this error. For example, if the model encounters the rate limit error, it should run time.sleep() if it’s the first time it’s encountering this error, else it should inform the user.

Make tools available to your agents

Now that you have both functional and properly defined tools, it’s time to make them available to your agents. There are, however, still some considerations to make when providing your agent with tools.

  • How many tools fit into the context?
  • When to make tools available?
  • How to make tools available in the context?

To the first bullet point, I think it’s important to consider how many tools you should fix in your context. Providing the agent with too many tools makes it hard for the agent to know when to use which tools, and thus, the model will struggle with effectively utilizing the tools. You should thus review your tools and consider if they’re all necessary for your agent to perform well.

Continuing on this topic, you may also consider when to make tools available. For example, the keyword_search tool might not be relevant when the agent is performing simple summarisation tasks for the user. You can modify your agent’s context to make some tools only available when they are relevant to use.

Lastly, you should also consider how you make tools available. In your prompt, you should create a separate section for your tools, either using a markdown heading or a separate XML tag. You should then put all your tools and the descriptions for how and when to utilize your tools in this section.

Conclusion

In this article, I’ve covered how to work with tools for your agents. I’ve discussed how tools are the most important aspect for AI agents, which allows them to be as effective as they are. Furthermore, I also discussed how to create proper tool definitions and how to create specialized tools. I believe working with your tools, improving how they’re defined and how your agent utilizes them, is the most important aspect you can spend time on when working on your AI agent.
