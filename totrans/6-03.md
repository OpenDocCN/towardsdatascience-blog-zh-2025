## 简介

今天，我们将在这篇文章中稍微偏离通常以 Snowflake 和数据仓库为中心的概念。我们将指导您创建自己的 API，该 API 位于存储在另一种数据库（DynamoDB）中的数据之上，然后构建一个小型 Web 前端来测试它。

你可能在日常工作中编写过许多 API 代码，但创建自己的 API 是什么感觉呢？在本文中，我们将向您展示如何创建一个 API，这将使您对云服务的工作方式有更多的了解，同时也希望给您在将来构建自己的 API 带来灵感。

以下是我们今天将要构建的内容的示意图：一个托管在 AWS Lambda 中的 API，全部使用 Python 编写，并连接到功能强大的 DynamoDB。然后，API 通过 AWS API Gateway 进行暴露，这使得我们能够为 API 添加保护措施，如速率限制。我们还使用了 Route 53（他们的域名管理工具）和 CloudWatch 进行日志记录。

## 为什么需要 API？

如前所述，您肯定使用过 API。但它们是如何构建的？它们是如何帮助您访问数据的？让我们从一些潜在的优势开始。

+   **抽象和解耦**：API 抽象了底层架构，因此用户不需要了解有关数据存储方式或如何查询系统的任何信息。您还可以集中业务规则、验证和转换，从而防止逻辑重复。

+   **增强安全性**：通过 API，您可以限制用户获取数据和与之交互的方式——他们可以更新记录吗？他们可以删除吗？无论您想公开什么，都可以通过 API 实现，从而防止不希望出现的错误。此外，您可以实施速率限制和节流，这可以防止对服务的滥用。

+   **可维护性**：通过为您的 API 添加版本控制（/v1、/v2），您可以在不破坏旧版本的情况下安全地演进您的应用程序。而且一个巨大的优势是，您可以更换数据存储的位置（也许您以后想迁移到 MySQL 或 Snowflake），API 仍然可以与客户端保持 100%的兼容性。您只需要更改 API 层中数据检索的位置即可。

+   **跨平台访问**：最后——我们正在构建的 API 与 Windows、Mac、Linux 和移动设备上的任何编程语言兼容。

## 在 DynamoDB 中存储数据

对于这个应用程序，我选择了 [DynamoDB](https://aws.amazon.com/dynamodb/)。DynamoDB 实际上非常出色，对其有所了解是必须的。它被称为 NoSQL 的键值存储。有很多这样的数据库的不同版本，但这是最受欢迎的之一。它看起来不像一系列的表，也不支持您典型的 SQL 语法或通常的将表连接在一起的想法。相反，DynamoDB 倾向于使用 [反规范化](https://www.geeksforgeeks.org/denormalization-in-databases/) 的模式，如果可能的话，所有数据都存储在单个表中。正因为如此，某些用例肯定不适合它。[在做出选择之前](https://www.reddit.com/r/aws/comments/177sfoz/when_would_you_use_dynamodb_over_rds/) 了解一下何时可以使用以及不应使用此类数据库。

要开始，我们需要在 DynamoDB 控制台中创建一个新的表，称为 Items。您现在需要处理的唯一设置是将分区键设置为 `id`，这将代表我们数据的唯一标识符。

之后，我们可以使用 [AWS CLI](https://docs.aws.amazon.com/cli/latest/userguide/getting-started-install.html) 将项目添加到我们的新表中。您会注意到，我们不需要像使用 SQL 那样提前定义任何模式；我们只需在添加时添加键值对并指定它们的数据类型。就这么简单。

```py
aws dynamodb put-item 
      --table-name Items 
      --item '{
      "id": {"S": "2"},
      "id": {"S": "6ec6736f-ae36-4091-9cc0-1d8a98babee1"},
      "name": {"S": "Smartphone"},
      "description": {"S": "Latest model with 5G capability"}
      }'
```

您可以通过创建四个或五个额外的项目来重复该过程以开始。

注意：唯一标识符是以 [UUID4](https://en.wikipedia.org/wiki/Universally_unique_identifier) 值创建的。您可以使用 Python 或互联网上的任何数量 [网站](https://www.uuidgenerator.net/) 来创建一个。这对于您需要的任何数据标识符来说都是一个非常好的实践，因为创建的标识符之间几乎零碰撞的可能性，而且您不需要跟踪顺序或之前创建的值。只需简单地生成一个新的即可。

## 使用 FastAPI 构建路由

对于我们的 API，我们将使用库 [FastAPI](https://fastapi.tiangolo.com/)。FastAPI 是一个简单但具有生产质量的 Python API。首先，让我们在 FastAPI 中定义什么是路由。路由定义了一个客户端可以访问的特定端点（或 URL 路径），以及用于与该端点交互的 HTTP 方法（例如，GET、POST、PUT、DELETE）。让我们看看一个用于健康检查的非常简单的例子：

```py
app = FastAPI ()

@app.get("/health")
async def health():
    logger.info("Health check endpoint accessed")
    return {"status": "healthy"}
```

这就是其最基本的形式！`@app.get` 装饰器告诉当 `<URL>/health` 被 ping 时，传入的请求应该做什么。GET 是 HTTP 方法。我们可以继续使用这种模式添加我们想要添加的所有其他端点，例如列出所有项目、创建新项目、编辑项目以及删除项目；也称为 CRUD 操作。让我们看看获取所有项目的路由。

```py
@app.get("/items", response_model=List[Dict[str, Any]])
async def get_items(table: Any = Depends(get_dynamodb)):
    logger.info("Handling GET request for all items")
    # Add request path logging
    logger.info("FastAPI root_path: %s", app.root_path)
    logger.info("FastAPI routes: %s", [route.path for route in app.routes])
    try:
        response = table.scan()
        items = response.get("Items", [])
        logger.info(f"Successfully retrieved {len(items)} items")
        return items
    except ClientError as e:
        --- ERROR HANDLING HERE ---
```

这个过程会为你要实现的每个端点重复进行。你只需确保你选择了正确的 HTTP 方法，并调用适当的 DynamoDB API 来对数据进行操作。

## 创建 Lambda 函数

接下来，我们想要将我们的 API 包裹在部署为 Lambda 函数所需的代码中。幸运的是，将一组普通的 Python 代码迁移到 Lambda 中并不涉及太多内容。我们可以做的主要事情之一是确保我们实现了日志记录。我们可以使用标准的 Python 日志记录模块，使用这个包所做的任何操作都会自动写入 CloudWatch。

```py
# Configure logging for CloudWatch
logger = logging.getLogger()
logger.setLevel(logging.INFO)
# Lambda automatically captures logs from stdout/stderr and sends to CloudWatch
formatter = logging.Formatter(
    "%(asctime)s | %(levelname)s | %(name)s:%(funcName)s:%(lineno)d - %(message)s"
)
handler = logging.StreamHandler()
handler.setFormatter(formatter)
logger.addHandler(handler)
```

对于这个应用程序，我们需要做的最后一步是利用一个名为 Mangum 的包。Mangum 包充当适配器，将 ASGI（异步服务器网关接口）框架（如 FastAPI）与 AWS Lambda 和 API 网关桥接。它本质上允许你将 ASGI 应用程序作为无服务器函数部署到 AWS Lambda。

通常，像 FastAPI 这样的 ASGI 框架需要持久化的服务器来运行（例如，Uvicorn）。然而，AWS Lambda 是事件驱动的，是短暂的，没有在后台运行持久化的服务器。Mangum 通过允许 FastAPI 作为 AWS Lambda 处理器来运行，从而填补了这个差距。这一切都是通过几行代码完成的。

```py
from mangum import Mangum

# Create Lambda handler with API Gateway v2 configuration
lambda_handler = Mangum(app, lifespan="off", api_gateway_base_path="/default")
```

## 部署函数

部署 Lambda 函数需要你将函数所需的所有内容打包成一个 zip 文件，包括 AWS 在 Lambda 环境中可能没有立即可用的库。由于 Lambda 有文件大小限制，**不要**打包已经安装的库，例如 boto3，Python AWS SDK。

为了简化这个过程，你可以创建一个 Bash 脚本来为你完成所有打包工作。只需运行这个脚本来部署你的脚本和任何更新。

```py
#!/bin/bash

# Remove existing files
rm -rf package lambda.zip lvenv

# Create and activate virtual environment
python3.9 -m venv lvenv
source lvenv/bin/activate

# Create package directory
mkdir -p package

# Install dependencies
pip install -r requirements.txt --target ./package

# Copy lambda function to package directory
cp lambda_function.py ./package/

# Create deployment package
cd package &amp;&amp; zip -r ../lambda.zip . &amp;&amp; cd ..

# Cleanup
deactivate
rm -rf lvenv package

# Update Lambda function
aws lambda update-function-code --function-name DynamoAPI --zip-file fileb://lambda.zip
```

## 使用 API 网关公开端点

此处的最后一步是将我们的 Lambda 函数通过 AWS API Gateway 公开。我们首先从 AWS 控制台的 Lambda 函数页面开始，将新的 API Gateway 附加到 Lambda 函数。选择添加触发器，然后选择 API Gateway。从这里，创建一个新的 API Gateway，选择 HTTP API 并将安全设置设置为开放。

此过程的下一个也是最重要的步骤是将路由映射到您的 Lambda 函数。这是一个简单的从 API Gateway 控制台执行的过程。导航到您新创建的网关，并从侧边栏选择路由。为 Lambda 函数中的每个路由创建一个新的路由。您还可以包括 FastAPI 自动生成的路由，例如`/docs`。

在您添加了所有路由后，通过侧边栏导航到集成部分，然后为每个路由，将 Lambda 函数附加到端点。这将告诉 API 网关在收到来自路由的请求时应该触发什么。

**注意**：您还可以在控制台下的节流中为您的 API 设置自定义速率限制。

## 使用 Streamlit 测试您的 API

如果您是本博客的常客，您知道我们非常喜欢 Streamlit。它非常容易为数据驱动应用程序构建快速的用户界面。因此，当然，我们使用它来构建我们 API 的小型测试框架。

当您与 RESTful API 一起工作时，您将使用的一个核心概念是`requests`。这个包允许您执行我们在 API 中设置的各个网络 API 调用。让我们看看第一个，即获取项目路由。

```py
Import requests

# Configure API URL
API_URL = "https://api.dataknowsall.com"

# Fetch items from API
try:
    response = requests.get(f"{API_URL}/items")
    response.raise_for_status()  # Raise an exception for bad status codes
    items = response.json()
```

您会在 try 块中看到我们使用`requests.get`直接调用 API。这将返回包含您查询结果的`JSON`。在我们的例子中，我们可以直接从这个结果创建一个 Pandas DataFrame，并使用`st.dataframe()`函数在 Streamlit 中显示它。

当涉及到我们 API 的其他动词（`POST, PUT, DELETE`）时，我们可以遵循类似的约定。让我们看看`EDIT`的代码。

```py
# Create the updated item dictionary
updated_item = {
    'id': selected_item['id'],
    'name': name,
    'description': description
}

# Update item using API
response = requests.put(f"{API_URL}/items/{selected_item['id']}", json=updated_item)
```

要更新一条记录，我们使用 `requests.put` 并根据我们的 API 规范传递 URL，最后包含更新项目所需的 `JSON` 数据。在这种情况下，我们表中只有三个字段，我们可以简单地创建一个包含我们想要更新的键值对的字典。

注意：添加项目 API 的模式完全相同，只是您使用`requests.post`。
