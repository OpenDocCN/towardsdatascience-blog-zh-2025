PyTorch 提供了非凡的灵活性，允许你在几秒钟内编写复杂的 GPU 加速操作。然而，这种便利是有代价的。PyTorch 按顺序执行你的代码，导致性能不佳。这转化为模型训练速度变慢，影响实验的迭代周期、团队的稳健性、财务影响等等。

在这篇文章中，我将探讨加速你的 PyTorch 操作的三种策略。每种方法都以`softmax`作为我们的“Hello World”演示，但你可以用任何你喜欢的函数替换它，讨论的方法仍然适用。

我们将从`torch.compile`开始，然后转向编写自定义的 Triton 内核，最后深入设计 CUDA 内核。

所以，这篇文章可能会变得复杂，但请耐心等待。

## `torch.compile` – 提升性能的快捷方式

> 💥 *"等等，你只是打开了一个函数调用，代码就变快了？就这样？听起来太好了，不太可能是真的。"*

+   是的。

`torch.compile`是 PyTorch 中一个相对较新的 API，它使用运行时图捕获和内核融合。通过一个装饰器，你通常可以在不显著更改代码的情况下看到速度的提升。

简单来说，例如，我们可以通过将操作合并到一个 GPU 函数中来加快计算，这消除了单独 GPU 调用的开销。或者，更好的是，通过用等效的一个操作替换操作链来优化操作链！

在常规的 PyTorch 执行模式（eager）中，这些优化是不可能的，因为它会按照代码中调用的顺序执行操作。

下面是一个简单的示例，展示了如何使用`torch.compile`实现和编译 softmax 函数。将其替换到你的模型的前向传递中，你的代码（希望）会运行得更快。

**优点**：

+   只需一行代码即可启用编译器。

+   不需要任何黑魔法仪式（也许动态形状除外）。

+   在编译过程中，第一次运行可能会慢一些；之后，它会加快速度。

+   并非总是对所有模型都能产生显著的加速效果，如果你的代码过于创新，有时甚至可能会出错。

+   仍然存在处理动态形状的问题。

> 😡 当输入形状改变且我们不希望为每个特定大小重新编译代码时，需要动态形状编译模式。

调试这些问题需要一篇全新的文章。

## Triton 代码 – 使用 Python Breeze 编写 GPU 内核

### 为什么使用 Triton？

**Triton**是一种编译成高效 GPU 内核的语言，同时允许你编写 Python 风格的代码。它在 PyTorch 的 dynamo/inductor 堆栈下使用，但你也可以编写自己的自定义操作！对于许多矩阵/张量操作，例如 softmax，你可以获得巨大的速度提升。因为**为什么**要等待官方 PyTorch 内核，当你可以编写自己的内核时？

## 纯 CUDA（即全力以赴）

有时候甚至 Triton 也达不到要求，或者你只是喜欢生活在边缘。在这种情况下，你可以用 C++编写一个自定义 CUDA 内核，编译它，并通过自定义扩展将其绑定到 PyTorch。像[[这个融合 CUDA softmax 参考]](https://github.com/fattorib/CudaSoftmax?ref=alexdremov.me)这样的项目展示了人们如何为最大速度构建专用内核。

**优点**:

+   最大的控制权。“如果你想做得正确，就自己来做。”

+   如果优化得当，有可能实现最快的内核。

**缺点**:

+   需要深入理解 CUDA。

+   内存管理、块大小、共享内存——这些都是难点！

+   维护开销可能**非常高**。

## 结论

当涉及到加速 PyTorch 操作时，你可以选择从越来越复杂的方法中进行选择：

1.  **`torch.compile`**: 需要的代码更改最少。

1.  **Triton 内核**：对内核行为的控制更多，但编码仍然相当简单。

1.  **纯 CUDA**：最大化的优化潜力，但**复杂性更高**。

如果你正在寻找最简单的改进，从`torch.compile`开始。如果这还不够，探索 Triton。对于高级用户，编写自定义 CUDA 内核可以带来进一步的提升，尽管这需要深厚的 GPU 编程技能。
