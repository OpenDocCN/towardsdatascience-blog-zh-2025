## GPU 架构基础

在本节中，我们将介绍 (*Nvidia*) GPU 的基础知识，以便我们开始，并在本文末尾编写我们的第一个 Triton 内核。

从最小的软件单元开始，我们可以描述执行单元的层次结构如下：

+   **线程**：最小的**工作单元**，它们运行用户定义的内核代码。

+   **Warps**：最小的**调度单元**，它们总是由 32 个并行线程组成，每个线程都有自己的指令地址计数器和寄存器状态。warp 中的线程**同时开始**，但**可以自由分支**和**独立执行**。

+   **线程块**：warp 的组，其中所有线程都可以通过共享内存**协作**并同步屏障。要求线程块可以**独立**且按任意顺序执行，以并行或顺序方式。这种独立性允许线程块可以在任何数量的核心上**按任意顺序调度**，这样 GPU 程序就可以随着核心数量的增加而高效扩展。如果需要，我们可以在内核的特定点同步块内的线程，例如同步内存访问。

+   **流式多处理器 (SM)**：负责**并行执行多个 warps**的单元，它拥有共享内存和一个 L1 缓存（存储 SM 访问过的最新全局内存行）。一个 SM 拥有一个专门的**warp 调度器**，从准备运行的线程块中提取 warps。

在硬件方面，最小的作业单元是**CUDA核心**，物理的**算术逻辑单元（ALU**），它为**线程**（或其部分）执行算术运算。

用类比来总结本节，我们可以将**CUDA核心**看作**个体工人**，而**warp**是**32个工人组成的队伍**，他们同时接收到相同的指令。他们可能以相同或不同的方式执行这项任务（分支），并且可能在不同的时间点完成它（独立性）。**线程块**由**几个共享相同工作空间的队伍**组成（即有共享内存），工作空间中所有队伍的工人可以同时等待午餐。**流式多处理器**是一个**有许多队伍共同工作并共享工具和存储的工厂车间**。最后，**GPU**是一个**整个工厂**，有许多楼层。

## 优化基础

当优化深度学习模型时，我们正在处理三个主要组件：

1.  **计算**: GPU执行浮点运算（FLOPS）所花费的时间。

1.  **内存**: 在GPU内部传输张量所花费的时间。

1.  **开销**: 所有其他操作（Python解释器、PyTorch调度等）。

考虑这些组件有助于找到解决瓶颈的正确方法。例如，如果大部分时间都花在内存传输上，那么增加计算（例如使用更强大的GPU）是没有帮助的。理想情况下，大部分时间应该花在计算上，更确切地说是在矩阵乘法上，这是GPU优化的精确操作。

这意味着最小化数据传输的成本，无论是从CPU到GPU（“**数据传输成本**”），从一个节点到另一个节点（“**网络成本**”），还是从CUDA全局内存（**DRAM**，便宜但速度慢）到CUDA共享内存（**SRAM**，昂贵但设备上最快的内存）。后者被称为**带宽成本**，目前将是我们的主要关注点。降低带宽成本的常见策略包括：

1.  **重用**在共享内存中加载的数据以进行多个步骤。一个典型的例子是分块矩阵乘法，我们将在未来的文章中介绍。

1.  **融合**单个内核中的多个操作（因为每次内核启动都意味着将数据从DRAM移动到SRAM），例如我们可以将矩阵乘法与激活函数融合。通常，**算子融合**可以提供巨大的性能提升，因为它防止了大量的全局内存读写，并且任何两个操作都提供了一个融合的机会。

![图片](../Images/0b08ce14c44126525c935679abca1a79.png)

在没有算子融合的情况下，矩阵乘法后跟ReLU激活。 （由作者绘制）

在这个例子中，我们执行矩阵乘法`x@W`并将结果存储在中间变量`a`中。然后我们对`a`应用`relu`并将结果存储在变量`y`中。这需要GPU从全局内存中的`x`和`W`读取，将结果写入`a`，再次从`a`读取，最后写入`y`。相反，操作融合允许我们在单个内核中执行矩阵乘法和应用ReLU，从而将全局内存的读写次数减半。

## Triton

现在，我们将编写我们的第一个Triton内核，一个简单的向量加法。首先，让我们了解一下这个操作是如何在GPU上分解和执行的。

考虑想要对两个向量`X`和`Y`的条目求和，每个向量有7个元素（`n_elements=7`）。

我们将指示GPU以每次3个元素的分块处理这个问题（`BLOCK_SIZE=3`）。因此，为了覆盖输入向量的所有7个元素，GPU将启动3个并行的“程序”，即我们的内核的独立实例，每个实例都有一个唯一的程序ID，`pid`：

+   程序0分配了元素`0, 1, 2`。

+   程序1分配了元素`3, 4, 5`。

+   程序2分配了元素`6`。

然后，这些程序将结果写回到全局内存中存储的向量`Z`。

一个重要的细节是，内核不会接收整个向量`X`，而是接收指向第一个元素内存地址的**指针**，`X[0]`。为了访问`X`的实际值，我们需要手动从全局内存中加载它们。

我们可以通过程序ID访问每个块的数据：`block_start = pid * BLOCK_SIZE`。从那里，我们可以通过计算`offsets = block_start + range(0, BLOCK_SIZE)`来获取该块的剩余元素地址，并将它们加载到内存中。

然而，请记住，程序2只分配了元素6，但其偏移量是`[6, 7, 8]`。为了避免任何索引错误，Triton允许我们定义一个**掩码**来识别有效的目标元素，这里`mask = offsets < n_elements`。

我们现在可以安全地加载`X`和`Y`并将它们相加，然后将结果以类似的方式写回到全局内存中的输出变量`Z`。

让我们分解一些Triton特定的语法：

+   首先，一个Triton内核总是由`<a href="http://twitter.com/triton" target="_blank" rel="noreferrer noopener">@triton</a>.jit`装饰。

+   第二，一些参数需要声明为静态的，这意味着它们在计算时是已知的。这是对`BLOCK_SIZE`的要求，通过添加`tl.constexpr`类型注解来实现。此外，请注意，我们没有注解其他变量，因为它们不是合适的Python变量。

+   通过调用 `tl.load` 和 `tl.store` 并传入指针数组来实现变量的加载和存储。请注意，这里没有 `return` 语句，这个角色被委托给了 `tl.store`。

要使用我们的内核，我们现在需要编写一个 **PyTorch 级别的包装器**，它提供内存指针并定义一个 **内核网格**。通常，内核网格是一个包含沿着每个轴分配给内核的 **线程块数量的 1D、2D 或 3D 元组**。在我们之前的例子中，我们使用了一个包含 3 个线程块的 1D 网格：`grid = (3, )`。

为了处理不同的数组大小，我们默认使用 `grid = (ceil(n_elements / BLOCK_SIZE), )`。

关于包装器的最后两点说明：

你可能已经注意到 `grid` 被定义为 lambda 函数。这允许 Triton 在启动时计算要启动的线程块数量。因此，我们根据存储在 `meta` 中的块大小来计算网格大小，`meta` 是一个包含编译时常数的字典，这些常数被暴露给内核。
