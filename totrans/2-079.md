# “我认为分析师就像数据巫师，他们帮助他们的产品团队解决问题”

> 原文：[`towardsdatascience.com/i-think-of-analysts-as-data-wizards-who-help-their-product-teams-solve-problems/`](https://towardsdatascience.com/i-think-of-analysts-as-data-wizards-who-help-their-product-teams-solve-problems/)

*Mariya 的故事是一段持续学习的旅程。从软件工程、数学和物理的坚实基础开始，她在过去 12 年多的时间里，在多个行业中积累了产品分析的专业知识，从搜索引擎和数据分析平台到金融科技。她独特的路径，包括作为产品经理的实践经验，使她能够全面了解分析团队如何帮助企业做出正确的决策。*

### 你广泛地写关于[代理式 AI](https://towardsdatascience.com/code-agents-the-future-of-agentic-ai/)和像 smolagents 和 LangGraph 这样的框架。你对这个新兴领域最感兴趣的是什么？

我最初开始探索生成式 AI 主要是出于好奇心，并且坦白说，还有一点恐惧错过（FOMO）。周围的人似乎都在使用 LLMs 或者至少在谈论它们。所以我抽出时间来亲自动手，从最基础的部分开始，比如提示技术（prompt techniques）和 LLM API。越深入，我就越兴奋。

最吸引我的是代理式系统正在塑造我们生活和工作的方式。我相信这种影响随着时间的推移只会继续增长。这就是为什么我抓住每一个机会使用像 Copilot 或 Claude Desktop 这样的代理式工具，或者使用 smolagents、LangGraph 或 CrewAI 等技术构建自己的代理。

对于我来说，Agentic AI 最具有影响力的应用案例是**编码**。GitHub Copilot 等工具如何提高工作效率和质量，这确实令人印象深刻。尽管 METR 最近的研究[《2025 年早期 AI 经验开发者研究》](https://metr.org/Early_2025_AI_Experienced_OS_Devs_Study.pdf)质疑这些效率提升是否真的如此显著，但我确实在我的日常工作中感受到了差异。这对于重复性任务（如 SQL 中的表转换）或处理不熟悉的技术（如使用 TypeScript 构建 Web 应用）时尤其有帮助。总的来说，我估计速度提高了大约 20%。但这种提升并不仅仅是关于生产力的；它也是一种范式转变，同时也扩展了可能性的感觉。我相信随着 Agentic 工具的持续发展，我们将看到那些学会如何利用这些技术的个人和公司，与那些没有学会的个人和公司之间效率差距的不断扩大。

当谈到分析时，我对自动报告代理特别兴奋。想象一下，一个能够提取正确数据、创建可视化、在需要时进行根本原因分析、记录开放问题，甚至创建演示文稿初稿的 AI。那将是神奇的。我已经构建了一个[原型](https://towardsdatascience.com/from-data-to-stories-code-agents-for-kpi-narratives/)，它可以生成这样的 KPI 叙事。尽管原型与一个可靠的生产解决方案之间还有很大的差距，但我相信我们最终会实现这一点。

### 你在“[产品分析师的实用计算机模拟](https://towardsdatascience.com/practical-computer-simulations-for-product-analysts-90b5deb6a54e/)”系列中写了三篇文章。是什么启发了这个系列，你认为模拟如何重塑产品分析？

模拟在产品分析中是一个被广泛低估的工具。我写这个系列是为了向人们展示模拟的强大和易用性。在我的日常工作中，我经常遇到诸如“*如果我们添加这个 KYC 控制，我们需要多少操作代理？*”或“*在新的市场中推出这个功能可能产生什么影响？*”之类的假设性问题。你可以模拟任何系统，无论其多么复杂。因此，模拟为我提供了一种定量和相对准确回答这些问题的方法，即使当时还没有硬数据。所以我希望更多的分析师开始使用这种方法。

当处理不确定性和分布时，模拟也表现出色。我个人更喜欢使用自助方法，而不是记住长长的统计公式和显著性标准列表。在实际操作中，模拟过程通常感觉更直观，且更不容易出错。

最后，我发现技术如何改变我们做事的方式非常迷人。凭借今天的计算能力，任何笔记本电脑都能在几分钟或几秒钟内运行数千次模拟，我们能够轻松解决仅仅三十年前就具有挑战性的问题。这对分析师来说是一个变革性的变化。

### 您的一些帖子专注于将 LLM 应用从[原型](https://towardsdatascience.com/from-prototype-to-production-enhancing-llm-accuracy-791d79b0af9b/)过渡到[生产](https://towardsdatascience.com/llm-evaluations-from-prototype-to-production/)。您在哪个阶段看到团队犯下了哪些常见的错误？

通过实践，我发现 LLM 原型与生产解决方案之间存在显著的差距，许多团队低估了这一点。最常见的错误是将原型视为已经准备好投入生产。

原型阶段可能看起来非常顺利。您可以在一两个小时之内构建一个功能性的产品，对几个示例进行测试，并感觉您已经解决了问题。原型是证明可行性和让团队对机会感到兴奋的绝佳工具。但这里往往是团队容易出错的地方：这些早期版本在面对多样化的真实世界场景时，无法保证一致性、质量或安全性。

我学到的是，成功的生产部署始于严格的评估。在扩展任何事物之前，您需要明确“良好性能”的定义，包括准确性、语气、速度以及任何特定于您用例的其他标准。然后，您需要在迭代过程中持续跟踪这些指标，确保您实际上是在改进而不是仅仅改变事物。

就像软件测试一样：您不会在没有适当测试的情况下发布代码，LLM 应用也需要同样的系统化方法。这在像金融科技或医疗保健这样的监管环境中尤为重要，您需要向内部团队以及合规利益相关者证明可靠性。

在这些监管领域，您需要全面的监控、人工审查流程和能够经得起审查的审计轨迹。支持所有这些所需的基础设施通常需要比构建原始 MVP 更多的发展时间。这是让专注于核心功能的团队感到惊讶的持续现象。

### 您的文章有时会将工程原理与数据科学/分析最佳实践相结合，例如您的“[每个数据分析师都应该知道的 10 个工程教训。](https://towardsdatascience.com/from-code-to-insights-software-engineering-best-practices-for-data-analysts-0dd6a2aaadfc/)”您认为数据与工程之间的界限是否正在模糊？

今天数据分析师或数据科学家的角色通常需要来自多个学科技能的混合。

+   我们编写代码，因此我们与软件工程师有共同的基础。

+   我们帮助产品团队思考战略并做出决策，因此产品管理技能是有用的。

+   我们利用统计学和数据科学来构建严格和全面的分析。

+   为了使我们的叙述具有说服力并真正影响决策，我们需要掌握沟通和可视化的艺术。

个人而言，我很幸运在上学和大学时期就获得了多样化的编程技能。这个背景在数据分析方面极大地帮助了我：它提高了我的效率，帮助我与工程师更好地协作，并教会了我如何构建可扩展和可靠的解决方案。

我强烈建议分析师采用软件工程的最佳实践。例如，版本控制系统、测试和代码审查有助于分析团队开发更可靠的流程，并交付更高品质的结果。我认为数据与工程之间的界限并不会完全消失，但我确实相信，那些拥抱工程思维的分析师在现代数据团队中将会更加高效。

### 你探索了[因果推断](https://towardsdatascience.com/linear-regressions-for-causal-conclusions-34c6317c5a11/)和前沿的 LLM 调优技术。你认为这些是共享工具箱的一部分还是不同的思维模式？

这实际上是一个很好的问题。我坚信所有这些工具（从统计方法到现代机器学习技术）都属于同一个工具箱。正如罗伯特·海因莱因著名地说：“专业化是昆虫的行为。”

我认为分析师是数据巫师，他们利用最适合的工具帮助产品团队解决问题：无论是构建一个用于 NPS 评论的 LLM 分类器，使用因果推断做出战略决策，还是构建一个自动化工作流程的 Web 应用程序。

与专注于特定技能相比，我更倾向于关注我们正在解决的问题，并尽可能保持工具集的广泛性。这种思维方式不仅能够带来更好的结果，而且还能培养持续学习的文化，这在当今快速发展的数据行业中至关重要。

### 你涵盖了广泛的主题，从[文本嵌入](https://towardsdatascience.com/text-embeddings-comprehensive-guide-afd97fce8fb5/)和[可视化](https://towardsdatascience.com/visualisation-101-choosing-the-best-visualisation-type-3a10838b150d/)到[模拟](https://towardsdatascience.com/practical-computer-simulations-for-product-analysts-90b5deb6a54e/)和[多智能体系统](https://towardsdatascience.com/multi-ai-agent-systems-101-bac58e3bcc47/)。是什么写作习惯或指导原则帮助你保持你的工作如此连贯和易于接近？

我通常写一些我此刻感兴趣的话题，无论是刚刚学到新知识，还是与同事进行了有趣的讨论。我的灵感通常来自在线课程、书籍或我的日常任务。

当我写作时，我总是考虑我的受众以及这篇文章如何真正对他人和未来的我有所帮助。我试图清晰地解释所有概念，并为想要深入了解的人留下线索。随着时间的推移，我的博客已经成为了一个个人知识库。我经常回到旧帖子：有时只是复制一段代码，有时是与正在从事类似工作的同事分享资源。

正如我们所知，数据中的所有事物都是相互关联的。解决现实世界的问题通常需要混合使用工具和方法。例如，如果你在估算进入新市场的影响，你可能使用模拟进行情景分析，使用大型语言模型（LLMs）来探索客户期望，以及使用可视化来展示最终的推荐。

我试图在我的写作中反映这些联系。技术通过建立在前人突破的基础上而发展，理解基础有助于你深入。这就是为什么我的许多帖子相互引用，让读者跟随他们的好奇心，揭示不同的部分是如何结合在一起的。

### 您的文章结构令人印象深刻，常常引导读者从基础概念到高级实现。在开始写作之前，您是如何概述复杂作品的？

我认为我在学校就形成了这种信息呈现方式，因为这些习惯有着深厚的根基。正如《文化地图》一书所解释的，不同的文化在沟通结构上有所不同。有些是概念优先（从基础开始，迭代地得出结论），而有些是应用优先（从结果开始，根据需要深入）。我确实内化了概念优先的方法。

在实践中，我的许多文章都是受到在线课程的启发。在观看课程的同时，我会并行地概述其大致结构，以免遗漏任何重要的细节。我也会记下任何不清楚的地方，并标记为将来阅读或实验。

课程结束后，我开始思考如何将这种知识应用到实际例子中。我坚信，直到你亲自尝试，你才能真正理解某件事。尽管大多数课程都有实际例子，但它们通常过于完美。因此，只有当你将相同的想法应用于自己的用例时，你才会遇到边缘情况和摩擦点。例如，课程可能使用 OpenAI 模型，但我可能想尝试本地模型，或者框架中的默认系统提示对我的特定情况不起作用，需要调整。

一旦我有一个工作示例，我就开始写作。我更喜欢先单独起草，然后再编辑。首先，我专注于将所有想法和代码记录下来，而不必担心语法或语气。然后我进入编辑模式：完善结构，选择合适的视觉元素，编写引言，并突出关键要点。

最后，我从头到尾阅读全文，以捕捉我可能遗漏的内容。然后我让我的伴侣审阅它。他们经常带来新的视角，指出我没有考虑到的方面，这有助于使文章更加全面和易于理解。
