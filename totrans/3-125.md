# 智能体 AI：单智能体与多智能体系统

> 原文：[`towardsdatascience.com/agentic-ai-single-vs-multi-agent-systems/`](https://towardsdatascience.com/agentic-ai-single-vs-multi-agent-systems/)

我们在过去几年中看到了这种转变，从构建僵化的编程系统到自然语言驱动的流程，这一切都得益于更高级的大型语言模型。

在这些智能体 AI 系统中，有趣的一个领域是构建单智能体与多智能体工作流程之间的差异，或者说是与更灵活的系统相比，与更受控的系统工作的差异。

本文将帮助您了解什么是智能体 AI，如何使用 LangGraph 构建简单的流程，以及不同架构可以实现的不同结果。我将通过构建具有各种数据源的技术新闻智能体来演示这一点。

至于用例，我对根据我的偏好获取自动新闻更新非常着迷，这样我每天就不会被信息过载淹没。

![图片](img/634b01e0e6b14fab819ddff41cfda5db.png)

让人工智能为我们总结信息，而不是我们自己搜集信息 | 图片由作者提供

与总结和收集研究一起工作，是智能体 AI 可以真正大放异彩的领域之一。

所以请跟我一起，看看我如何尝试让 AI 为我做基础工作，我们将比较单智能体与多智能体设置之间的差异。

*我总是保持我的工作术语无障碍，所以如果你对智能体 AI 是新手，这篇文章应该能帮助你了解它是什么以及如何与之合作。如果你不是新手，你可以跳过一些部分。*

## 智能体 AI（& LLMs）

智能体 AI 是关于用自然语言编程。你不再使用僵化的、明确的代码，而是指导大型语言模型（LLMs）通过普通语言路由数据和执行操作，以自动化任务。

在工作流程中使用自然语言并不新鲜，我们多年来一直使用 NLP 来提取和处理数据。新的地方在于我们现在可以给予语言模型多少自由，允许它们处理歧义并动态做出决策。

![图片](img/ddb08f1d953c287cca2b81c176493f7a.png)

从程序自动化到 NLP 再到 LLMs 的传统自动化 | 图片由作者提供

但仅仅因为 LLMs 能够理解细微的语言，并不意味着它们天生就验证事实或保持数据完整性。我认为它们主要是位于结构化系统及其现有数据源之上的一个通信层。

![图片](img/cc22bb72fe5e1a5597090c6acd4367ae.png)

LLMs 是一个通信层，而不是系统本身 | 图片由作者提供

我通常这样向非技术人员解释：它们的工作方式有点像我们。如果我们无法访问干净、结构化的数据，我们就会开始编造东西。LLMs 也是如此。它们根据模式生成响应，而不是进行事实核查。

所以，就像我们一样，他们尽其所能。如果我们想要更好的输出，我们需要构建能够提供可靠数据的系统。因此，在代理系统中，我们整合了它们与不同数据源、工具和系统交互的方式。

现在，仅仅因为我们*可以*在更多地方使用这些更大的模型，并不意味着我们*应该*这样做。LLMs 在解释细微的自然语言方面表现出色，例如客户服务、研究或人机协作。

但对于结构化任务——比如提取数字并将它们发送到某个地方——你需要使用传统方法。LLMs 在数学上并不比计算器更擅长。因此，与其让 LLM 进行计算，不如给 LLM 访问计算器的权限。

所以，无论何时你*可以*通过编程方式构建工作流程的一部分，这仍然是一个更好的选择。

尽管如此，LLMs 在适应混乱的现实世界输入和解释模糊指令方面非常出色，因此将两者结合起来可以是一种构建系统的绝佳方式。

## 代理框架

我知道很多人会直接跳到 CrewAI 或 AutoGen，但我建议检查[LangGraph](https://www.langchain.com/langgraph)、[Agno](https://github.com/agno-agi/agno)、[Mastra](https://github.com/mastra-ai/mastra)和[Smolagents](https://github.com/huggingface/smolagents)。根据我的研究，这些框架迄今为止收到了一些最强烈的反馈。

![](img/e3c7d0e6e691da7681e0744fd400402c.png)

我在 GitHub 仓库[这里](https://github.com/ilsilfverskiold/Awesome-LLM-Resources-List/tree/main?tab=readme-ov-file#top-agentic-frameworks)收集了最受欢迎的框架资源 | 图片由作者提供

LangGraph 更技术化且可能比较复杂，但它是许多开发者的首选选择。Agno 更容易上手但技术性较低。Mastra 是 JavaScript 开发者的一个稳健选择，而 Smolagents 作为一个轻量级替代品显示出很大的潜力。

在这个案例中，我选择了 LangGraph——建立在 LangChain 之上——并不是因为它是我最喜欢的，而是因为它正成为更多开发者采用的常用框架。

因此，熟悉它是值得的。

虽然它有很多抽象，你可能需要重建其中一部分，以便更好地控制和理解它。

我不会在这里详细讨论 LangGraph，因此我决定为那些需要回顾的人快速构建一个[指南](https://howtouseai.substack.com/p/agentic-ai-frameworks-learning-langgraph)。

对于这个用例，你将能够运行工作流程而无需编写任何代码，但如果你在这里学习，你可能还想了解它是如何工作的。

## 选择 LLM

现在，你可能会跳进来并想知道为什么我会选择某些 LLM 作为代理的基础。

你不能随便选择任何模型，尤其是在框架内工作时。它们需要兼容。需要关注的关键事项包括工具调用支持和生成结构化输出的能力。

我建议查看 HuggingFace 的代理[排行榜](https://huggingface.co/spaces/galileo-ai/agent-leaderboard)，看看哪些模型在现实世界的代理系统中实际表现良好。

对于这个工作流程，你应该可以使用 Anthropic、OpenAI 或 Google 的模型。如果你在考虑其他模型，请确保它与 LangChain 兼容。

## 单代理与多代理系统

如果你围绕一个 LLM 构建一个系统，并给它提供你希望它使用的工具，你就是在使用单代理工作流程。这很快速，如果你是初学者，可能会觉得模型应该自己解决问题。

![](img/b0d8d5fb08f6d983dd281d2e95a1f133.png)

一个代理可以访问许多工具 | 图片由作者提供

但问题是这些工作流程只是系统设计的一种形式。就像任何软件项目一样，你需要规划流程，定义步骤，构建逻辑结构，并决定每个部分应该如何表现。

![](img/d603d6251723466a1a4c7f2d25a9083c.png)

考虑一下你的用例中逻辑应该如何工作 | 图片由作者提供

这就是多代理工作流程的用武之地。

虽然并非所有的工作流程都是层次结构或线性的，但有些是协作性的。协作工作流程因此也会落入我更难以与之合作的更灵活的方法中，至少就目前存在的功能而言。

然而，协作工作流程也将不同的功能分解成各自的模块。

当你只是随意尝试时，单代理和协作工作流程是很好的起点，但它们并不总是能提供实际任务所需的精确度。

对于我将要构建的工作流程，我已经知道 API 应该如何使用——所以我的工作就是引导系统正确使用它。

我们将比较单代理设置与层次结构多代理系统，其中主要代理将任务委派给一个小团队，以便你可以看到它们在实际中的表现。

## 构建单代理工作流程

使用单线程——即一个代理——我们给 LLM 提供访问几个工具的权限。根据用户的问题，由代理决定使用哪个工具以及何时使用。

![](img/8d6dceb3e726ba120d636c2ded5645d6.png)

一个 LLM/代理可以访问许多带有许多选项的工具 | 图片由作者提供

单代理的挑战在于控制。

无论系统提示多么详细，模型可能不会遵循我们的要求（这在更受控的环境中也可能发生）。如果我们给它太多工具或选项，它很可能不会使用所有工具，甚至可能不会使用正确的工具。

为了说明这一点，我们将构建一个技术新闻代理，它可以访问几个具有自定义数据的 API 端点，这些工具的参数有几个选项。由代理决定使用多少以及如何设置最终的摘要。

*记住，我使用 LangGraph 构建了这些工作流程。在这里，我不会深入介绍 LangGraph，所以如果您想学习基础知识以便能够调整代码，[请点击这里](https://howtouseai.substack.com/p/agentic-ai-frameworks-learning-langgraph)。*

您可以在[这里](https://github.com/ilsilfverskiold/Awesome-LLM-Resources-List/tree/main/guides/langgraph/single-agent-workflow)找到单代理工作流程。要运行它，您需要安装 LangGraph Studio 和最新版本的 Docker。

一旦设置好，打开您电脑上的项目文件夹，在`.env`文件中添加您的`GOOGLE_API_KEY`，并保存。您可以从[这里](https://aistudio.google.com/app/apikey)获取密钥。

*Gemini Flash 2.0 有一个慷慨的免费层，所以运行这个不会花费任何费用（但如果你过度使用它，可能会遇到错误）。*

如果您想切换到另一个 LLM 或工具，可以直接调整代码。但再次提醒，LLM 需要兼容。

设置完成后，启动 LangGraph Studio 并选择正确的文件夹。

这将启动我们的工作流程，以便我们可以对其进行测试。

对于这个提示，它决定只检查按“公司”类别过滤的每周趋势关键词，然后它获取这些关键词的来源，并为我们总结。

它在提供统一总结方面存在一些问题，它只是简单地使用了最后得到的信息，而没有使用所有的研究成果。

在现实中，我们希望它获取几个类别（不仅仅是公司）中的趋势和顶级关键词，检查来源，跟踪特定关键词，并在返回响应之前将所有这些内容合理地总结。

我们当然可以对其进行探测并不断提问，但正如您所想象的那样，如果我们需要更复杂的东西，它可能会在工作流程中开始走捷径。

关键在于，代理系统不会仅仅按照我们的预期思考，我们必须实际编排它以完成我们想要的事情。

因此，单个代理非常适合简单的事情，但正如您所想象的那样，它可能不会像我们预期的那样思考和表现。

这就是为什么选择一个更复杂的系统，其中每个代理负责一件事，可能非常有用。

## 测试多代理工作流程

构建多智能体工作流程比构建具有工具访问权限的单个智能体要困难得多。为此，你需要事先仔细考虑架构以及数据如何在智能体之间流动。

我在这里设置的多智能体[工作流程](https://github.com/ilsilfverskiold/Awesome-LLM-Resources-List/tree/main/guides/langgraph/tech_news_bot_multiagent)使用了两个不同的团队——一个研究团队和一个编辑团队，每个团队下都有几个智能体。

每个智能体都可以访问一组特定的工具。

![图片](img/5fbed6d685f19080d42fdeff49f589e6.png)

多智能体工作流程逻辑与分层团队 | 图像由作者提供

我们引入了一些新工具，比如一个充当共享空间的研究便签，一个团队写下他们的发现，另一个从那里读取。最后一个 LLM 将阅读所有已研究和编辑的内容，以制作摘要。

使用研究便签的替代方案是将数据存储在状态中的便签中，为每个团队或智能体隔离短期记忆。但这也意味着需要仔细思考每个智能体的记忆应该包含什么。

我还决定更详细地构建工具，以便提供更丰富的数据，这样智能体就不必为每个关键词单独获取来源。在这里，我使用正常的程序逻辑，因为我可以这么做。

*一个需要记住的关键点：如果你可以使用正常的编程逻辑，那就这么做。*

由于我们使用多个智能体，你可以通过为大多数智能体使用更便宜的模型，并将更昂贵的模型保留用于重要任务来降低成本。

在这里，我使用 Gemini Flash 2.0 为所有智能体，除了摘要器，它运行在 OpenAI 的 GPT-4o 上。如果你想获得更高质量的摘要，你可以使用具有更大上下文窗口的更先进的 LLM。

工作流程已经为你设置好了[这里](https://github.com/ilsilfverskiold/Awesome-LLM-Resources-List/tree/main/guides/langgraph/tech_news_bot_multiagent)。在加载之前，请确保在`.env`文件中添加你的 OpenAI 和 Google API 密钥。

在这个工作流程中，路由（边）是动态设置的，而不是像我们处理单个智能体那样手动设置。如果你查看代码，它看起来会更复杂。

一旦你在 LangGraph Studio 中启动工作流程——与之前相同的过程——你将看到所有这些节点都准备就绪的图。

## 结束语

我在这里使用了一个好的数据源。没有它，你需要添加更多的错误处理，这将使一切变得更加缓慢。

清洁和结构化的数据是关键。没有它，LLM 无法发挥最佳性能。

即使有可靠的数据，它也不是完美的。你仍然需要工作在代理上，确保它们做它们应该做的事情。

你可能已经注意到了系统是有效的——但它还没有完全达到那里。

仍然有几个方面需要改进：将用户的查询解析为更结构化的格式，添加护栏以确保代理始终使用他们的工具，更有效地总结以保持研究文档简洁，改进错误处理，并引入长期记忆以更好地理解用户实际需要。

状态（短期记忆）对于想要优化性能和成本特别重要。

目前，我们只是将每条消息推入状态，并让所有代理都能访问它，这并不是理想的。我们真的希望在不同团队之间分离状态。在这种情况下，这是我还没有做的事情，但你可以通过在状态模式中引入一个草稿本来隔离每个团队所知道的内容。
