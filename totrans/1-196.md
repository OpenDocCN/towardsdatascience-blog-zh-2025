# 超越 Requests：为什么 httpx 是你有时需要的现代 HTTP 客户端（有时）

> [`towardsdatascience.com/beyond-requests-why-httpx-is-the-modern-http-client-you-need-sometimes/`](https://towardsdatascience.com/beyond-requests-why-httpx-is-the-modern-http-client-you-need-sometimes/)

<mdspan datatext="el1760555113192" class="mdspan-comment">如果你在 Python 中花费过时间进行 HTTP 调用，那么你很可能使用过 Requests 库。多年来，Requests 一直是*事实上的*标准，以其相对简单而闻名，并已成为无数 Python 应用的基石。从简单的脚本到更复杂的 Web 服务，其同步特性在许多不同类型的应用中表现良好。

然而，Python 库生态系统不断演变，尤其是随着使用 asyncio 的异步编程的兴起。这种转变为新的库打开了大门，这些库旨在利用非阻塞 I/O 以增强性能，尤其是在 I/O 密集型应用中。

正是在这里，HTTPX 库发挥了作用，这是一个相对较新的库，自称为“Python 的下一代 HTTP 客户端”，提供同步和异步 API，以及支持现代 Web 功能，如 HTTP/2。

### 什么是 Requests？

对于 Python 新手或需要复习的人来说，Requests 是一个简单优雅的 HTTP 库，由 Kenneth Reitz 在近十五年前创建。其主要目标是使 HTTP 请求变得简单且人性化。你想发送一些数据？发起 GET 或 POST 请求？处理头部、cookies 或身份验证？Requests 使这些任务变得直观。

它的同步特性意味着当你发起一个请求时，你的程序会在收到响应之前等待。这对于许多应用来说是可以接受的，但对于需要大量并发 HTTP 调用的任务（如网络爬虫或与多个微服务交互），这种阻塞行为可能会成为一个显著的瓶颈。

### 什么是 HTTPX？

根据其官方文档，HTTPX 是，

> “...一个功能齐全的 Python 3 HTTP 客户端，提供同步和异步 API，并支持 HTTP/1.1 和 HTTP/2。”

它是由 Encode（Starlette、Uvicorn 和 Django Rest Framework 背后的团队）开发的。

HTTPX 的一些卖点包括，

+   **异步支持:** 原生的 async/await 语法，用于非阻塞操作。

+   **HTTP/2 支持:** 与 Requests（默认主要支持 HTTP/1.1）不同，HTTPX 可以支持 HTTP/2，可能提供如多路复用等性能优势。

+   **类似 Requests 的 API:** 它旨在为那些习惯于 Requests 的用户提供一个熟悉的 API，从而简化过渡过程。

+   **传输 API:** 一个更高级的功能，允许自定义传输行为，对于测试或特定的网络配置很有用。

对于 HTTPX 的声明非常吸引人。一个与 Requests 兼容的 API，具有 async/await 的功能，以及潜在的性能提升。但它是不是显而易见的继承人，能够取代当前冠军，或者它是不是一个针对特定异步用例的利基工具？找出答案的唯一方法就是将它们都进行测试。

### 比较 HTTPX 和 Requests 的性能

为了比较性能，我们将使用这两个库运行一系列 HTTP GET 请求并计时。我们首先检查同步操作，然后探讨异步功能。

对于我们的目标，我们将使用 **httpbin.org**，这是一个用于测试 HTTP 请求的出色服务。将其视为为构建或使用制作 HTTP 请求的软件（如网络客户端、API 客户端、抓取器等）的开发者的测试和调试工具。您无需设置自己的 Web 服务器来查看您的 HTTP 请求的外观或测试您的客户端如何处理不同的服务器响应，您可以将您的请求发送到 httpbin.org 的测试服务器。它具有各种端点，旨在返回特定类型的响应，允许您检查和验证客户端的行为。

**本地 FastAPI 服务器设置**

让我们创建一个简单的 FastAPI 应用程序作为我们的异步端点。将其保存为 test_server.py：

```py
# test_server.py
from fastapi import FastAPI
import asyncio

app = FastAPI()
@app.get("/fast")
async def read_fast():
    return {"message": "Hello from FastAPI!"}
@app.get("/slow")
async def read_slow():
    await asyncio.sleep(0.1) # Simulate some I/O-bound work
    return {"message": "Hello slowly from FastAPI!"}
```

在一个单独的终端窗口中通过输入以下命令启动此服务器。

```py
uvicorn test_server:app --reload --host 127.0.0.1 --port 8000
```

我们已经设置了所需的一切。让我们开始我们的代码示例。

**示例 1—简单同步 GET 请求**

让我们从基本场景开始：连续 20 次获取简单的 JSON 响应。

```py
import requests
import httpx
import time

import nest_asyncio
nest_asyncio.apply()

URL = "https://httpbin.org/get"
NUM_REQUESTS = 20

# --- Requests ---
start_time_requests = time.perf_counter()
for _ in range(NUM_REQUESTS):
    response = requests.get(URL)
    assert response.status_code == 200

end_time_requests = time.perf_counter()
time_requests = end_time_requests - start_time_requests
print(f"Execution time (Requests, Sync): {time_requests:.4f} seconds")

# --- HTTPX (Sync Client) ---
start_time_httpx_sync = time.perf_counter()
with httpx.Client() as client: # Using a client session is good practice
    for _ in range(NUM_REQUESTS):
        response = client.get(URL)
        assert response.status_code == 200

end_time_httpx_sync = time.perf_counter()
time_httpx_sync = end_time_httpx_sync - start_time_httpx_sync
print(f"Execution time (HTTPX, Sync): {time_httpx_sync:.4f} seconds")
```

输出结果。

```py
Execution time (Requests, Sync): 22.6370 seconds
Execution time (HTTPX, Sync): 11.4099 seconds
```

这已经比 Requests 快得多了。在我们的测试中，它在同步检索方面几乎快了两倍。

**示例 2—使用 HTTPX 的简单异步 GET 请求（单个请求）**

现在，让我们通过向之前设置的本地 FastAPI 服务器发送单个请求来测试 HTTPX 的异步功能。

```py
import httpx
import asyncio
import time

LOCAL_URL_FAST = "http://127.0.0.1:8000/fast"

async def fetch_with_httpx_async_single():
    async with httpx.AsyncClient() as client:
        response = await client.get(LOCAL_URL_FAST)
        assert response.status_code == 200

start_time_httpx_async = time.perf_counter()
asyncio.run(fetch_with_httpx_async_single())
end_time_httpx_async = time.perf_counter()
time_httpx_async_val = end_time_httpx_async - start_time_httpx_async
print(f"Execution time (HTTPX, Async Single): {time_httpx_async_val:.4f} seconds")
```

输出结果。

```py
Execution time (HTTPX, Async Single): 0.0319 seconds
```

这很符合预期，对于本地请求来说很快。这个测试主要验证异步机制是否工作。真正的异步测试来自于并发。

**示例 3—并发异步 GET 请求**。

这正是 HTTPX 的异步能力应该真正超越 Requests 的地方。我们将并发地向我们的**/slow**端点发送 100 个请求。

```py
import httpx
import asyncio
import time
import requests

LOCAL_URL_SLOW = "http://127.0.0.1:8001/slow" # 0.1s delay
NUM_CONCURRENT_REQUESTS = 100 

# --- HTTPX (Async Client, Concurrent) ---
async def fetch_one_httpx(client, url):
    response = await client.get(url)
    return response.status_code
async def main_httpx_concurrent():
    async with httpx.AsyncClient() as client:
        tasks = [fetch_one_httpx(client, LOCAL_URL_SLOW) for _ in range(NUM_CONCURRENT_REQUESTS)]
        results = await asyncio.gather(*tasks)
        for status_code in results:
            assert status_code == 200

start_time_httpx_concurrent = time.perf_counter()
asyncio.run(main_httpx_concurrent())
end_time_httpx_concurrent = time.perf_counter()
time_httpx_concurrent_val = end_time_httpx_concurrent - start_time_httpx_concurrent
print(f"Execution time (HTTPX, Async Concurrent to /slow): {time_httpx_concurrent_val:.4f} seconds")

# --- For Comparison: Requests (Sync, Sequential to /slow) ---
# This will be slow, demonstrating the problem async solves
start_time_requests_sequential_slow = time.perf_counter()
for _ in range(NUM_CONCURRENT_REQUESTS):
    response = requests.get(LOCAL_URL_SLOW)
    assert response.status_code == 200

end_time_requests_sequential_slow = time.perf_counter()
time_requests_sequential_slow_val = end_time_requests_sequential_slow - start_time_requests_sequential_slow
print(f"Execution time (Requests, Sync Sequential to /slow): {time_requests_sequential_slow_val:.4f} seconds")
```

典型输出

```py
Execution time (HTTPX, Async Concurrent to /slow): 0.1881 seconds
Execution time (Requests, Sync Sequential to /slow): 10.1785 seconds
```

现在的***这个***功能相当不错！HTTPX 利用 asyncio.gather 在 1 秒多内完成了 100 个请求（每个请求有 0.1 秒的模拟延迟）。由于任务是 I/O 密集型的，asyncio 可以在它们等待服务器响应时在这些任务之间切换。总时间大约是单个请求最长时间加上一小部分用于管理并发的开销。

相比之下，同步的 Requests 代码用了超过 10 秒（100 个请求 * 0.1 秒/请求 = 10 秒，加上开销）。这展示了异步操作对于 I/O 密集型任务的强大能力。HTTPX 不仅仅在绝对意义上“更快”；它使处理并发 I/O 的方式更加高效。

### 那么关于 HTTP/2 呢？

如果服务器也支持 HTTP/2 并且安装了 h2 库（pip install httpx[h2]），HTTPX 支持 HTTP/2。HTTP/2 提供了诸如多路复用（通过单个连接发送多个请求）和头部压缩等好处。

```py
import httpx
import asyncio
import time

# A public server that supports HTTP/2
HTTP2_URL = "https://github.com"
# HTTP2_URL = "https://www.cloudflare.com" # Another option
NUM_HTTP2_REQUESTS = 20

async def fetch_http2_info():
    async with httpx.AsyncClient(http2=True) as client: # Enable HTTP/2
        for _ in range(NUM_HTTP2_REQUESTS):
            response = await client.get(HTTP2_URL)
            # print(f"HTTP Version: {response.http_version}, Status: {response.status_code}")
            assert response.status_code == 200
            assert response.http_version in ["HTTP/2", "HTTP/2.0"] # Check if HTTP/2 was used

start_time = time.perf_counter()
asyncio.run(fetch_http2_info())
end_time = time.perf_counter()
print(f"Execution time (HTTPX, Async with HTTP/2): {end_time - start_time:.4f} seconds for {NUM_HTTP2_REQUESTS} requests.")
```

**输出结果**

```py
Execution time (HTTPX, Async with HTTP/2): 0.7927 seconds for 20 requests.
```

虽然这个测试确认了 HTTP/2 的使用，但在简单的脚本中量化其相对于 HTTP/1.1 的速度优势可能有点棘手。HTTP/2 的优势通常在具有许多小型资源或高延迟连接的复杂场景中更为明显。对于许多常见的 API 交互，除非服务器特别优化以充分利用 HTTP/2 功能，否则差异可能不会很大。然而，拥有这种能力是一个重要的前瞻性功能。

### 超越原始速度

性能并非一切。开发者的体验、功能和易用性同样关键，因此让我们在比较这两个库时看看这些方面。

**异步/等待支持**

HTTPX。原生一级支持。这是其最重要的差异化特征。

请求。纯粹同步。要获得类似 Requests 的异步行为，你通常会考虑像 aiohttp（它有一个不同的 API）这样的库，或者在使用线程池执行器中的 Requests（这增加了复杂性，并且不是真正的 asyncio）。

**HTTP/2 支持**

HTTPX。我们之前已经提到过，但为了回顾，这个功能是内置的。

请求。没有原生的 HTTP/2 支持。存在第三方适配器，但它们并不那么集成。

**API 设计及易用性**

HTTPX。有意设计得与 Requests 非常相似。如果你熟悉 Requests，HTTPX 会感觉熟悉。以下是一个快速代码比较。

```py
# Requests
r = requests.get('https://example.com', params={'key': 'value'})
# HTTPX (sync)
r = httpx.get('https://example.com', params={'key': 'value'})
# HTTPX (async)
async with httpx.AsyncClient() as client:
```

请求。同步 HTTP 调用中的简洁标准。

**客户端会话/连接池**

两个库都强烈建议使用客户端会话（requests.Session()和 httpx.Client() / httpx.AsyncClient()）以获得性能优势，如连接池和 cookie 持久化。两者的使用方式非常相似。

**依赖足迹**

请求。相对轻量级（charset_normalizer，idna，urllib3，certifi）。

HTTPX。有几个核心依赖（httpcore、sniffio、anyio、certifi、idna），以及 h11 用于 HTTP/1.1。如果你添加 h2 用于 HTTP/2，那又是一个。考虑到其更广泛的功能集，这是可以理解的。

**成熟度与社区**

REQUESTS。极其成熟，拥有庞大的社区，经过十年的实战考验。

HTTPX。虽然较新，但由一个信誉良好的团队（Encode）积极开发，并且正在迅速获得关注。它被认为稳定且适用于生产。

### 何时选择 HTTPX？何时坚持使用 Requests？

说了这么多，你如何在这两者之间做出选择？以下是我的一些看法。

如果需要 HTTP/2 支持……

1.  **你需要异步操作**。这是主要原因。如果你的应用程序涉及许多 I/O 密集型的 HTTP 调用，使用 HTTPX 和 asyncio 将提供显著的性能改进和更好的资源利用率。

1.  **你需要 HTTP/2 支持**。如果你与利用 HTTP/2 提高性能的服务器交互，HTTPX 提供了这一功能。

1.  **你正在启动一个新项目，并希望使其面向未来**。HTTPX 的现代设计和异步能力使其成为新应用程序的强大选择。

1.  **你想要一个用于同步和异步 HTTP 调用的单一库**。如果你有混合需求，这可以简化你的依赖关系管理和代码库。

1.  **你需要高级功能**。比如它的传输 API，可以精细控制请求调度，或者用于测试。

如果需要成熟度和社区支持……

1.  **你的应用程序是纯同步的，并且有简单的 HTTP 需求**。如果 Requests 已经很好地完成了工作，并且你没有遇到 I/O 瓶颈，那么可能没有充分的理由切换。

1.  **你正在处理一个高度依赖 Requests 的遗留代码库**。除非你特别需要 HTTPX 的功能，否则迁移可能不值得付出努力。

1.  **最小化依赖至关重要**。Requests 的占用空间略小。

1.  **asyncio 的学习曲线对你的团队来说是一个障碍**。虽然 HTTPX 提供了同步 API，但它的主要优势在于其异步能力。

### 摘要

我的调查表明，HTTPX 是一个非常出色的库。虽然它并不能神奇地使单个同步 HTTP 调用比 Requests 快得多（网络延迟仍然是王），但它的真正力量在异步应用程序中显现出来。当进行大量的并发 I/O 密集型调用时，HTTPX 提供了实质性的性能提升和更有效的代码结构方式，正如我们在并发测试中所展示的那样。

许多人声称 HTTPX 是“更好的”，但这取决于上下文。如果“更好”意味着拥有原生的异步/等待支持、HTTP/2 功能以及同时满足同步需求的现代 API，那么是的，HTTPX 在新开发中可能确实具有优势。Requests 仍然是一个出色的、可靠的库，用于同步任务，其简洁性仍然是其最大的优势。

对于并发异步操作，使用 httpx 的**有效吞吐量**可以比顺序同步的 Requests 高一个数量级，这是一个颠覆性的变化。

如果你是一名处理 HTTP 调用的 Python 开发者，尤其是在现代 Web 应用、微服务或数据密集型任务中，那么 HTTPX 不仅仅是一个值得观察的库——它是一个开始使用的库。对于同步代码，从 Requests 库过渡非常顺畅，其整体功能集和异步能力使其成为 Python HTTP 客户端未来的一个有吸引力的选择。
