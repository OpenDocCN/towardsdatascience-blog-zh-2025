# 超越Requests：为什么httpx是你有时需要的现代HTTP客户端（有时）

<mdspan datatext="el1760555113192" class="mdspan-comment">如果你在Python中花费过时间进行HTTP调用，那么你很可能使用过Requests库。多年来，Requests一直是*事实上的*标准，以其相对简单而闻名，并已成为无数Python应用的基石。从简单的脚本到更复杂的Web服务，其同步特性在许多不同类型的应用中表现良好。

然而，Python库生态系统不断演变，尤其是随着使用asyncio的异步编程的兴起。这种转变为新的库打开了大门，这些库旨在利用非阻塞I/O以增强性能，尤其是在I/O密集型应用中。

正是在这里，HTTPX库发挥了作用，这是一个相对较新的库，自称为“Python的下一代HTTP客户端”，提供同步和异步API，以及支持现代Web功能，如HTTP/2。

### 什么是Requests？

对于Python新手或需要复习的人来说，Requests是一个简单优雅的HTTP库，由Kenneth Reitz在近十五年前创建。其主要目标是使HTTP请求变得简单且人性化。你想发送一些数据？发起GET或POST请求？处理头部、cookies或身份验证？Requests使这些任务变得直观。

它的同步特性意味着当你发起一个请求时，你的程序会在收到响应之前等待。这对于许多应用来说是可以接受的，但对于需要大量并发HTTP调用的任务（如网络爬虫或与多个微服务交互），这种阻塞行为可能会成为一个显著的瓶颈。

### 什么是HTTPX？

根据其官方文档，HTTPX是，

> “...一个功能齐全的Python 3 HTTP客户端，提供同步和异步API，并支持HTTP/1.1和HTTP/2。”

它是由Encode（Starlette、Uvicorn和Django Rest Framework背后的团队）开发的。

HTTPX的一些卖点包括，

+   **异步支持:** 原生的async/await语法，用于非阻塞操作。

+   **HTTP/2支持:** 与Requests（默认主要支持HTTP/1.1）不同，HTTPX可以支持HTTP/2，可能提供如多路复用等性能优势。

+   **类似Requests的API:** 它旨在为那些习惯于Requests的用户提供一个熟悉的API，从而简化过渡过程。

+   **传输API:** 一个更高级的功能，允许自定义传输行为，对于测试或特定的网络配置很有用。

对于 HTTPX 的声明非常吸引人。一个与 Requests 兼容的 API，具有 async/await 的功能，以及潜在的性能提升。但它是不是显而易见的继承人，能够取代当前冠军，或者它是不是一个针对特定异步用例的利基工具？找出答案的唯一方法就是将它们都进行测试。

### 超越原始速度

性能并非一切。开发者的体验、功能和易用性同样关键，因此让我们在比较这两个库时看看这些方面。

**异步/等待支持**

HTTPX。原生一级支持。这是其最重要的差异化特征。

请求。纯粹同步。要获得类似Requests的异步行为，你通常会考虑像aiohttp（它有一个不同的API）这样的库，或者在使用线程池执行器中的Requests（这增加了复杂性，并且不是真正的asyncio）。

**HTTP/2支持**

HTTPX。我们之前已经提到过，但为了回顾，这个功能是内置的。

请求。没有原生的HTTP/2支持。存在第三方适配器，但它们并不那么集成。

**客户端会话/连接池**

两个库都强烈建议使用客户端会话（requests.Session()和httpx.Client() / httpx.AsyncClient()）以获得性能优势，如连接池和cookie持久化。两者的使用方式非常相似。

**成熟度与社区**

REQUESTS。极其成熟，拥有庞大的社区，经过十年的实战考验。

HTTPX。虽然较新，但由一个信誉良好的团队（Encode）积极开发，并且正在迅速获得关注。它被认为稳定且适用于生产。

### 何时选择 HTTPX？何时坚持使用 Requests？

说了这么多，你如何在这两者之间做出选择？以下是我的一些看法。

如果需要 HTTP/2 支持……

1.  **你需要异步操作**。这是主要原因。如果你的应用程序涉及许多 I/O 密集型的 HTTP 调用，使用 HTTPX 和 asyncio 将提供显著的性能改进和更好的资源利用率。

1.  **你需要 HTTP/2 支持**。如果你与利用 HTTP/2 提高性能的服务器交互，HTTPX 提供了这一功能。

1.  **你正在启动一个新项目，并希望使其面向未来**。HTTPX 的现代设计和异步能力使其成为新应用程序的强大选择。

1.  **你想要一个用于同步和异步 HTTP 调用的单一库**。如果你有混合需求，这可以简化你的依赖关系管理和代码库。

1.  **你需要高级功能**。比如它的传输 API，可以精细控制请求调度，或者用于测试。

如果需要成熟度和社区支持……

1.  **你的应用程序是纯同步的，并且有简单的 HTTP 需求**。如果 Requests 已经很好地完成了工作，并且你没有遇到 I/O 瓶颈，那么可能没有充分的理由切换。

1.  **你正在处理一个高度依赖 Requests 的遗留代码库**。除非你特别需要 HTTPX 的功能，否则迁移可能不值得付出努力。

1.  **最小化依赖至关重要**。Requests 的占用空间略小。

1.  **asyncio 的学习曲线对你的团队来说是一个障碍**。虽然 HTTPX 提供了同步 API，但它的主要优势在于其异步能力。

### 摘要

我的调查表明，HTTPX 是一个非常出色的库。虽然它并不能神奇地使单个同步 HTTP 调用比 Requests 快得多（网络延迟仍然是王），但它的真正力量在异步应用程序中显现出来。当进行大量的并发 I/O 密集型调用时，HTTPX 提供了实质性的性能提升和更有效的代码结构方式，正如我们在并发测试中所展示的那样。

许多人声称 HTTPX 是“更好的”，但这取决于上下文。如果“更好”意味着拥有原生的异步/等待支持、HTTP/2 功能以及同时满足同步需求的现代 API，那么是的，HTTPX 在新开发中可能确实具有优势。Requests 仍然是一个出色的、可靠的库，用于同步任务，其简洁性仍然是其最大的优势。

对于并发异步操作，使用 httpx 的**有效吞吐量**可以比顺序同步的 Requests 高一个数量级，这是一个颠覆性的变化。

如果你是一名处理HTTP调用的Python开发者，尤其是在现代Web应用、微服务或数据密集型任务中，那么HTTPX不仅仅是一个值得观察的库——它是一个开始使用的库。对于同步代码，从Requests库过渡非常顺畅，其整体功能集和异步能力使其成为Python HTTP客户端未来的一个有吸引力的选择。
