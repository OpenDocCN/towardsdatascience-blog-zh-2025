## 2. 云架构

从概念上讲，我们的云架构将与我们的本地应用没有太大区别：在私有网络中有一组容器，并通过网关连接到外部世界，即我们的用户。

然而，我们不会在本地计算机上使用 Docker Compose 运行容器，而是将它们部署到一个计算环境中，该环境可以自动跨虚拟机或物理机进行扩缩，以支持许多并发用户。

## 2.1 扩缩性

扩缩性是云架构中的一个核心概念。这意味着能够动态地处理不同数量的用户（即 HTTP 请求）。运行我们 UI 和数据库 API 的 web 服务器 Uvicorn，已经可以处理大约 40 个并发请求。甚至可以使用另一个名为 Gunicorn 的 web 服务器作为进程管理器，在该容器中采用多个 Uvicorn 工作进程，从而进一步提高并发性。

现在，如果我们想支持更多的并发请求，我们可以给每个容器分配更多的资源，如 CPU 或内存（**垂直扩缩**）。然而，一种更可靠的方法是基于传入的 HTTP 请求数量或内存/CPU 使用情况动态创建容器的副本（**水平扩缩**）。每个副本容器都将分配一个 IP 地址，因此我们还需要考虑网络：如何集中接收所有请求并将它们分配到容器副本。

这种“棱镜”模式很重要：请求集中到达某个服务器（一个**负载均衡器**），然后分散到多个其他服务器进行并行处理（例如，几个相同的 UI 容器）**。**

## 2.2 Kubernetes 概念

[Kubernetes](https://kubernetes.io/)是自动化部署、扩缩和管理容器化应用程序的行业标准系统。其核心概念对于理解现代云架构，包括我们的架构，至关重要，所以让我们快速回顾一下基础知识。

+   **节点**：运行容器化应用程序或管理集群的物理或虚拟机。

+   **集群**：由 Kubernetes 管理的节点集合。

+   **Pod**：Kubernetes 中最小的可部署单元。运行一个主要的应用容器，可选地运行共享存储和网络的其他辅助容器。

+   **部署**：一种抽象概念，通过部署、扩缩和更新一组 Pod 副本来管理它们的期望状态。

+   **服务**：一种抽象概念，通过将传入流量分配到各种动态 Pod IP 地址来管理一个稳定的入口点（服务的 DNS 名称），以暴露一组 Pod。服务有多种类型：

+   **ClusterIP**服务在集群内暴露 Pod

+   **负载均衡器** 服务将 Pod 暴露给集群外部。它触发云提供商在集群外部提供外部公共 IP 和负载均衡器，可以用来访问集群。然后，这些外部请求通过服务路由到各个 Pod。

+   **入口**：定义集群入口更复杂规则的抽象。它可以路由流量到多个服务；为服务提供外部可访问的 URL；负载均衡流量；并处理安全的 HTTPS。

+   **入口控制器**：实现入口规则。例如，基于 Nginx 的控制器在底层运行一个 Nginx 服务器（[就像在我们的本地应用中](https://github.com/jsbaan/ai-app-from-scratch/blob/main/compose.yaml#L60)），该服务器根据入口规则动态配置以路由流量。要使入口控制器本身对外可见，可以使用负载均衡器服务。这种架构经常被使用。

## 2.3 Azure 容器应用

拥有了这些概念，我并不想直接使用 Kubernetes 部署我们的应用，而是想通过使用 Azure 容器应用 (ACA) 来进行一些实验。这是一个建立在 Kubernetes 之上的无服务器平台，它抽象了一些复杂性。

通过一条命令，我们可以创建一个容器应用环境，在底层，这是一个由 Azure 管理的不可见的 Kubernetes 集群。在这个环境中，我们可以运行一个容器作为容器应用，Azure 内部将其管理为 Kubernetes 部署、服务和 Pod。请参阅 [文章 1](https://techcommunity.microsoft.com/blog/startupsatmicrosoftblog/aca-vs-aks-which-azure-service-is-better-for-running-containers/3815164) 和 [文章 2](https://techcommunity.microsoft.com/blog/fasttrackforazureblog/azure-container-apps-networking-a-condensed-view-of-concepts/3634304) 以获取详细的比较。

容器应用环境还会自动创建：

1.  一个不可见的 **[Envoy 入口控制器](https://learn.microsoft.com/en-us/azure/container-apps/networking?tabs=workload-profiles-env%2Cazure-cli#http-edge-proxy-behavior)**，它将请求路由到内部应用，并基于请求量处理 HTTPS 和应用自动扩展。

1.  一个外部 **公共 IP 地址** 和 **Azure 负载均衡器**，它们将外部流量路由到入口控制器，入口控制器再将流量路由到应用（听起来类似于 Kubernetes LoadBalancer 服务，对吧？）。

1.  为每个容器应用生成一个 Azure URL，该 URL 可通过互联网或内部网络公开访问，基于其入口配置。

这为我们提供了运行容器所需的一切。唯一缺少的是数据库。我们将使用 Azure 管理的 PostgreSQL 服务器，而不是部署自己的容器，因为这更简单、更可靠且可扩展。我们的本地 Nginx 反向代理容器也因为 ACA 自动部署了 Envoy 入口控制器而变得过时。

值得注意的是，我们实际上不需要更改本地应用程序中的任何一行 [代码](https://github.com/jsbaan/ai-app-from-scratch/tree/main)，我们只需将其视为一堆容器！

## 2.4 Azure 架构：整合一切

这里是我们聊天应用程序的完整云架构图，包含所有 Azure 资源。让我们从高层次的角度看看用户请求是如何通过系统的。

![Azure 架构图。由作者在 draw.io 中制作。](img/34cdf0d4a0a70853e6b31ecf22e1e143.png)

Azure 架构图。由作者在 draw.io 中制作。

1.  用户发送 HTTPS 请求到 [chat.jorisbaan.nl](http://chat.jorisbaan.nl)。

1.  一个公共 DNS 服务器，如 Google DNS，将此域名解析为 Azure 公共 IP 地址。

1.  此 IP 地址上的 Azure 负载均衡器将请求路由到（对我们来说不可见的）Envoy 入口控制器。

1.  UI 网络服务器向数据库 API 和语言模型 API 应用发送请求，这两个应用都将请求路由到其一个副本。

1.  数据库 API 副本查询 PostgreSQL 服务器的主机名。Azure 私有 DNS 区域将主机名解析为 PostgreSQL 服务器的 IP 地址。

## 3. 部署

那么，我们实际上是如何创建所有这些的呢？与其在 Azure 门户中点击，不如使用像 [Terraform](https://www.terraform.io/) 这样的基础设施即代码工具来创建和管理云资源。然而，为了简单起见，我将使用 Azure CLI 创建一个 bash 脚本，逐步部署我们的整个应用程序。您可以在 [这里](https://github.com/jsbaan/ai-app-from-scratch/blob/main/azure-deployment/deploy_app.sh) 找到完整的部署脚本，包括环境变量。我们现在将逐步进行。

## 3.1 设置

我们需要一个 Azure 账户（我使用的是一个 [免费教育账户](https://azure.microsoft.com/en-us/free/students)），一个 [`github.com/jsbaan/ai-app-from-scratch`](https://github.com/jsbaan/ai-app-from-scratch) 仓库的克隆，Docker 来构建和推送容器镜像，[下载的模型](https://github.com/jsbaan/ai-app-from-scratch/blob/main/lm-api/README.md)，以及 Azure CLI 来开始创建云资源。

我们首先创建一个资源组，以便我们的资源更容易查找、管理和删除。`--location` 参数指的是我们将用于部署应用程序基础设施的物理数据中心。理想情况下，它靠近我们的用户。然后我们创建一个包含 256 个 IP 地址的私有虚拟网络，以隔离、安全和连接我们的数据库服务器和容器应用。

```py
brew update &amp;&amp; brew install azure-cli # for macos

echo "Create resource group"
az group create 
  --name $RESOURCE_GROUP 
  --location "$LOCATION"

echo "Create VNET with 256 IP addresses"
az network vnet create 
  --resource-group $RESOURCE_GROUP 
  --name $VNET 
  --address-prefix 10.0.0.0/24 
  --location $LOCATION
```

## 3.2 PostgreSQL 服务器部署

根据硬件的不同，Azure 管理的 PostgreSQL 数据库服务器每月的费用约为 13 美元到 7000 美元。为了与容器应用通信，我们将数据库服务器放在同一个私有虚拟网络中，但放在其自己的子网中。子网是一段专用的 IP 地址范围，可以有自己的安全和路由规则。

我们创建具有私有访问权限的 Azure PostgreSQL 灵活服务器。这意味着只有同一虚拟网络内的资源可以访问它。Azure 会自动创建一个私有 DNS 区域，该区域管理数据库的主机名，该主机名解析为其 IP 地址。数据库 API 稍后将使用此主机名来连接到数据库服务器。

我们将随机生成数据库凭据并将它们存储在安全的地方：Azure 密钥保管库。

```py
echo "Create subnet for DB with 128 IP addresses"
az network vnet subnet create 
  --resource-group $RESOURCE_GROUP 
  --name $DB_SUBNET 
  --vnet-name $VNET 
  --address-prefix 10.0.0.128/25

echo "Create a key vault to securely store and retrieve secrets, 
like the db password"
az keyvault create 
  --name $KEYVAULT 
  --resource-group $RESOURCE_GROUP 
  --location $LOCATION

echo "Give myself access to the key vault so I can store and retrieve 
the db password"
az role assignment create 
  --role "Key Vault Secrets Officer" 
  --assignee $EMAIL 
  --scope "/subscriptions/$SUBSCRIPTION_ID/resourceGroups/$RESOURCE_GROUP/providers/Microsoft.KeyVault/vaults/$KEYVAULT"

echo "Store random db username and password in the key vault"
az keyvault secret set 
  --name postgres-username 
  --vault-name $KEYVAULT  
  --value $(openssl rand -base64 12 | tr -dc 'a-zA-Z' | head -c 12)
az keyvault secret set 
  --name postgres-password 
  --vault-name $KEYVAULT  
  --value $(openssl rand -base64 16)

echo "While we're at it, let's already store a secret session key for the UI"
az keyvault secret set 
  --name session-key 
  --vault-name $KEYVAULT  
  --value $(openssl rand -base64 16)

echo "Create PostgreSQL flexible server in our VNET in its own subnet. 
Auto-creates Private DS Zone."
POSTGRES_USERNAME=$(az keyvault secret show --name postgres-username --vault-name $KEYVAULT --query "value" --output tsv)
POSTGRES_PASSWORD=$(az keyvault secret show --name postgres-password --vault-name $KEYVAULT --query "value" --output tsv)
az postgres flexible-server create 
  --resource-group $RESOURCE_GROUP 
  --name $DB_SERVER 
  --vnet $VNET 
  --subnet $DB_SUBNET 
  --location $LOCATION 
  --admin-user $POSTGRES_USERNAME 
  --admin-password $POSTGRES_PASSWORD 
  --sku-name Standard_B1ms 
  --tier Burstable 
  --storage-size 32 
  --version 16 
  --yes
```

## 3.3 Azure 容器应用环境部署

在网络和数据库就绪后，让我们部署运行容器的基础设施——容器应用环境（回想一下，这实际上是一个底层的 Kubernetes 集群）。

我们创建了一个包含 128 个 IP 地址的另一个子网，并将其管理权委托给容器应用环境。该子网应该足够大，以便在子网范围内，每十个新的副本可以获得一个新的 IP 地址。[了解详情](https://learn.microsoft.com/en-us/azure/container-apps/networking?tabs=workload-profiles-env%2Cazure-cli#subnet)。然后我们可以创建环境。这只是一个简单的命令，无需太多配置。

```py
echo "Create subnet for ACA with 128 IP addresses."
az network vnet subnet create 
  --resource-group $RESOURCE_GROUP 
  --name $ACA_SUBNET 
  --vnet-name $VNET 
  --address-prefix 10.0.0.0/25

echo "Delegate the subnet to ACA"
az network vnet subnet update 
  --resource-group $RESOURCE_GROUP 
  --vnet-name $VNET 
  --name $ACA_SUBNET 
  --delegations Microsoft.App/environments

echo "Obtain the ID of our subnet"
ACA_SUBNET_ID=$(az network vnet subnet show 
  --resource-group $RESOURCE_GROUP 
  --name $ACA_SUBNET 
  --vnet-name $VNET 
  --query id --output tsv)

echo "Create Container Apps Environment in our custom subnet.
By default, it has a Workload profile with Consumption plan."
az containerapp env create 
  --resource-group $RESOURCE_GROUP 
  --name $ACA_ENVIRONMENT 
  --infrastructure-subnet-resource-id $ACA_SUBNET_ID 
  --location $LOCATION
```

## 3.4 Azure 容器应用部署

每个容器应用都需要一个 Docker 镜像来运行。让我们首先设置一个容器注册库，然后在本地上构建所有镜像并将它们推送到注册库。请注意，我们只是简单地使用其[Dockerfile](https://github.com/jsbaan/ai-app-from-scratch/blob/main/lm-api/Dockerfile)将模型文件复制到语言模型镜像中，因此我们不需要像在第一部分的本地部署那样挂载外部存储。

```py
echo "Create container registry (ACR)"
az acr create 
  --resource-group $RESOURCE_GROUP 
  --name $ACR 
  --sku Standard 
  --admin-enabled true

echo "Login to ACR and push local images"
az acr login --name $ACR
docker build --tag $ACR.azurecr.io/$DB_API $DB_API
docker push $ACR.azurecr.io/$DB_API
docker build --tag $ACR.azurecr.io/$LM_API $LM_API
docker push $ACR.azurecr.io/$LM_API
docker build --tag $ACR.azurecr.io/$UI $UI
docker push $ACR.azurecr.io/$UI
```

现在，让我们进行部署。要创建容器应用，我们需要指定它们的 环境、容器注册库、镜像以及它们将监听请求的端口。ingress 参数控制容器应用是否可以从外部世界访问。我们的两个 API 是内部的，因此完全隔离，没有公共 URL，也没有流量从 Envoy Ingress Controller 路由。UI 是外部的，有一个公共 URL，但它通过虚拟网络发送内部 HTTP 请求到我们的 API。我们将这些内部主机名和数据库凭据作为环境变量传递。

```py
echo "Deploy DB API on Container Apps with the db credentials from the key 
vault as env vars. More secure is to use a managed identity that allows the 
container itself to retrieve them from the key vault. But for simplicity we 
simply fetch it ourselves using the CLI."
POSTGRES_USERNAME=$(az keyvault secret show --name postgres-username --vault-name $KEYVAULT --query "value" --output tsv)
POSTGRES_PASSWORD=$(az keyvault secret show --name postgres-password --vault-name $KEYVAULT --query "value" --output tsv)
az containerapp create --name $DB_API 
  --resource-group $RESOURCE_GROUP 
  --environment $ACA_ENVIRONMENT 
  --registry-server $ACR.azurecr.io 
  --image $ACR.azurecr.io/$DB_API 
  --target-port 80 
  --ingress internal 
  --env-vars "POSTGRES_HOST=$DB_SERVER.postgres.database.azure.com" "POSTGRES_USERNAME=$POSTGRES_USERNAME" "POSTGRES_PASSWORD=$POSTGRES_PASSWORD" 
  --min-replicas 1 
  --max-replicas 5 
  --cpu 0.5 
  --memory 1

echo "Deploy UI on Container Apps, and retrieve the secret random session 
key the UI uses to encrypt session cookies"
SESSION_KEY=$(az keyvault secret show --name session-key --vault-name $KEYVAULT --query "value" --output tsv)
az containerapp create --name $UI 
  --resource-group $RESOURCE_GROUP 
  --environment $ACA_ENVIRONMENT 
  --registry-server $ACR.azurecr.io 
  --image $ACR.azurecr.io/$UI 
  --target-port 80 
  --ingress external 
  --env-vars "db_api_url=http://$DB_API" "lm_api_url=http://$LM_API" "session_key=$SESSION_KEY" 
  --min-replicas 1 
  --max-replicas 5 
  --cpu 0.5 
  --memory 1 

echo "Deploy LM API on Container Apps"
az containerapp create --name $LM_API 
  --resource-group $RESOURCE_GROUP 
  --environment $ACA_ENVIRONMENT 
  --registry-server $ACR.azurecr.io 
  --image $ACR.azurecr.io/$LM_API 
  --target-port 80 
  --ingress internal 
  --min-replicas 1 
  --max-replicas 5 
  --cpu 2 
  --memory 4 
  --scale-rule-name my-http-rule 
  --scale-rule-http-concurrency 2
```

## 3.5 扩展我们的容器应用

让我们看看我们的容器应用是如何扩展的。容器应用可以扩展到零，这意味着它们没有副本并且停止运行（并停止产生成本）。这是无服务器范式的一个特性，其中基础设施是根据需求提供的。[不可见的 Envoy 代理根据触发器进行扩展](https://techcommunity.microsoft.com/blog/startupsatmicrosoftblog/from-chaos-to-clarity-simplifying-your-networking-with-azure-container-apps/4034625)，例如并发 HTTP 请求。启动新的副本可能需要一些时间，这被称为冷启动。我们将副本的最小数量设置为 1 以避免冷启动和首次请求导致的超时错误。

默认的扩展规则会在现有副本收到 10 个并发 HTTP 请求时创建一个新的副本。这适用于 UI 和数据库 API。为了测试这个扩展规则是否合理，我们需要进行负载测试来模拟真实用户流量并查看每个容器应用副本可以单独处理多少并发请求。我的猜测是它们可以处理比 10 更多的并发请求，我们可以放宽这个规则。

## 3.5.1 语言模型推理扩展。

即使是我们的小型量化语言模型，推理所需的计算量也比简单的 FastAPI 应用要多得多。推理服务器按顺序处理传入的请求，默认的容器应用资源为 0.5 个虚拟 CPU 核心和 1GB 内存，导致响应时间非常慢：生成 128 个标记并带有 1024 个上下文窗口的参数需要长达 30 秒（这些参数在[LM API 的 Dockerfile](https://github.com/jsbaan/ai-app-from-scratch/blob/main/lm-api/Dockerfile)中定义）。

将 vCPU 提升至 2 个和内存至 4GB 可以显著提高推理速度，并在 30 秒内处理大约 10 个请求。我在 http 扩展规则上配置得非常严格，设置为 2 个并发请求，所以当两个用户同时聊天时，LM API 会扩展。

有 5 个最大副本，我认为这将允许大约 10-40 个并发用户，具体取决于聊天记录的长度。显然，这并不是一个非常大规模的数字，但如果我们有更高的预算，我们可以增加 vCPU、内存和副本数量。最终我们可能需要转向基于 GPU 的推理。关于这一点稍后详述。

## 3.6 自定义域名和 HTTPS

从 UI 应用自动生成的 URL 看起来像[`chat-ui.purplepebble-ac46ada4.germanywestcentral.azurecontainerapps.io/`](https://chat-ui.purplepebble-ac46ada4.germanywestcentral.azurecontainerapps.io/)。这并不是很容易记住，所以我希望将我们的应用作为子域名在我的网站上提供：[chat.jorisbaan.nl](http://chat.jorisbaan.nl)。

我在我的域名注册商门户（如 GoDaddy）上简单地添加了两个 DNS 记录。一个 CNAME 记录将我的`chat`子域名链接到 UI 的 URL，一个 TXT 记录用于向 Azure 证明子域的所有权并获取 TLS 证书。

```py
# Obtain UI URL and verification code
URL=$(az containerapp show -n $UI -g $RESOURCE_GROUP -o tsv --query "properties.configuration.ingress.fqdn")
VERIFICATION_CODE=$(az containerapp show -n $UI -g $RESOURCE_GROUP -o tsv --query "properties.customDomainVerificationId")

# Add a CNAME record with the URL and a TXT record with the verification code to domain registrar
# (Do this manually)

# Add custom domain name to UI App
az containerapp hostname add --hostname chat.jorisbaan.nl -g $RESOURCE_GROUP -n $UI
# Configure managed certificate for HTTPS
az containerapp hostname bind --hostname chat.jorisbaan.nl -g $RESOURCE_GROUP -n $UI --environment $ACA_ENVIRONMENT --validation-method CNAME
```

只要 CNAME 记录直接指向容器的域名，Container Apps 就会管理我的子域的免费 TLS 证书。

每当我拆解和重新部署一个环境时，UI 的公共 URL 都会改变。我们可以使用更高级的服务，如[Azure Front Door](https://learn.microsoft.com/en-us/azure/frontdoor/front-door-overview)或[Application Gateway](https://learn.microsoft.com/en-us/azure/application-gateway/overview)，以获得稳定的 URL，并作为具有额外安全、全球可用性和边缘缓存的反向代理。

## 4. 资源与成本概述

现在应用程序已部署，让我们看看它使用的所有 Azure 资源的概述。我们创建了其中大部分，但 Azure 还自动创建了一个负载均衡器、公共 IP、私有 DNS 区域、网络监视器和日志分析工作区。

![Azure Portal 中所有资源的截图](img/c5fc9c6bf6e0de5fb9e162e23f460037.png)

Azure Portal 中所有资源的截图。

一些资源是免费的，其他资源在一段时间或计算预算内是免费的，这也是我选择它们的原因之一。以下资源产生的成本最高：

+   **负载均衡器（标准层）**：免费 1 个月，然后每月$18。

+   **容器注册库（标准层）**：免费 12 个月，然后每月$19。

+   **PostgreSQL 灵活服务器（可突发 B1MS 计算层）**：免费 12 个月，然后至少每月$13。

+   **容器应用程序**：每月 50 个 CPU 小时或 200 万次请求免费，然后对于具有单个副本、0.5 个 vCPU 和 1GB 内存的应用程序，每月$10。具有 2 个 vCPU、4GB 内存的 LM API 对于单个副本每月大约$50。

你可以看到，这个小型（但可扩展）应用程序的成本可能会迅速累积到每月数百美元，即使没有 GPU 服务器来运行更强的语言模型！这就是为什么当你阅读这篇文章时，应用程序可能不会上线的原因。

很明显，Azure Container Apps 的成本比我最初想象的要高：它需要一个标准层负载均衡器来实现自动外部入口、HTTPS 和自动扩展。我们可以通过禁用外部入口并部署更便宜的替代方案来解决这个问题——例如，具有自定义反向代理的虚拟机或基本层负载均衡器。尽管如此，标准层 Kubernetes 集群的月成本至少为$150，因此 ACA 在小规模上可能更便宜。

## 5. 路线图

现在，在我们结束之前，让我们看看许多改进部署的方向中的几个。

**持续集成与持续部署**。我会设置一个 CI/CD 流水线，该流水线运行单元和集成测试，并在代码更改时重新部署应用程序。这可能是由新的 git 提交或合并的 pull 请求触发的。这也会使查看服务是否正确部署变得更容易。我还会设置监控和警报，以便快速了解问题（例如，崩溃的 Container App 实例）。

**降低延迟：语言模型服务器**。我会使用类似[Locust](https://locust.io/)或[Azure 负载测试](https://azure.microsoft.com/en-us/products/load-testing)的工具对整个应用程序进行负载测试——模拟真实世界的用户流量。即使没有进行负载测试，我们也明显存在一个瓶颈：LM 服务器。尽管它很小且进行了量化，但在没有并发的情况下，它仍然需要相当长的时间来处理长答案。对于更多用户来说，运行一个具有批处理机制的 GPU 推理服务器会更快速、更高效，该机制在队列中收集多个生成请求——可能使用[Kafka](https://kafka.apache.org/)——并在块上运行批处理推理。

随着用户数量的增加，我们可能需要几个基于 GPU 的 LM 服务器，它们从同一个队列中消费。对于 GPU 基础设施，我会考虑 Azure 虚拟机或者更高级的 Azure 机器学习。

llama.cpp 推理引擎适用于基于 CPU 的单用户推理。当迁移到 GPU 服务器时，我会考虑更适合批处理推理的推理引擎，如[vLLM](https://github.com/vllm-project/vllm)或[Huggingface TGI](https://huggingface.co/docs/text-generation-inference/en/index)。显然，为了提高响应质量，还需要一个更好（更大）的模型——这取决于具体的使用场景。

## 6. 最后的想法

我希望这个项目能让人一瞥一个由 AI 驱动的生产级 Web 应用可能的样子。我试图在现实工程和尽量简化之间取得平衡，以保持其简单、便宜、易于理解，并限制我的时间和计算预算。遗憾的是，我无法长时间保持应用程序运行，因为它很快就会每月花费数百美元。如果有人能帮助提供 Azure 信用额度以保持应用程序运行，请告诉我！

一些关于使用托管服务的总结：尽管 Azure 容器应用抽象了一些 Kubernetes 的复杂性，但了解底层 Kubernetes 概念仍然非常有用。自动创建的不可见基础设施，如公共 IP、负载均衡器和入口控制器，增加了未预见的成本，并使得理解情况变得困难。此外，与 Kubernetes 相比，ACA 文档有限。然而，如果你知道自己在做什么，你可以非常快速地设置一些东西。
