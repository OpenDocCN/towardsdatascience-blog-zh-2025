### OpenStreetMap – 简要介绍

OpenStreetMap 是一个旨在创建一个开放和免费世界地图的国际项目，请参阅[`www.openstreetmap.org`](https://www.openstreetmap.org)。它可以在开放数据库许可（ODbL）下访问，并且可以使用名为 Overpass API 的只读 API（cf. [`overpass-api.de)`](http://overpass-api.de))获取数据。在这篇文章中，我们使用一个名为*overpy*的 Python 库来查询 Overpass API 以从 OpenStreetMap 接收数据，有关详细信息请参阅[`pypi.org/project/overpy/`](https://pypi.org/project/overpy/)。

在我们开始之前，让我们了解一下 OpenStreetMap 的**数据模型**是如何工作的。基本上，其数据模型由*节点*、*线路*和 _ 关系 _ 组成。例如，*节点*确实具有由纬度和经度确定的特定位置，如一棵树、一张长椅或一栋建筑的角落。可以添加标签和注释，以及这些节点是更复杂空间结构（如线路、建筑或其他区域）的构建块。*线路*是一系列节点的有序列表，如果第一个节点是最后一个节点，则结果为多边形或线条。然后它可能代表多边形中的建筑或公园，在后一种情况下可能代表河流或公路。

此外，*关系*描述了诸如由多个道路组成的车路线等情景。在这篇文章中，我们专注于节点和道路。根据这个基本数据模型，任何对象可能携带的信息比其地理位置更多。有诸如“道路”或“便利设施”之类的标签可用，或者商店的营业时间。有了这些分类在手，人们可以提取适合大量空间用例的数据。

关于**数据质量**的说明：一个庞大的开放社区正在收集数据。这个社区在任何时间任何地点都无法存在。因此，在使用这些数据时要谨慎，并批判性地思考数据。一般来说，基础设施的变化缓慢，但随时间和空间的变化而变化。社会和商业事实也在变化，跟上这种步伐相当困难。在处理 OpenStreetMap 数据时，请记住这一点。

### 图论——简介

在其最简单的数学描述中，图是一个由**节点** *V* 和**边** *E* 组成的元组，即 *(V,E)*。说实话，这听起来非常抽象。让我们给它注入生命力。有许多例子可以说明节点，也称为涡流，可以代表什么。例如，它可以是地球上由纬度和经度描述的某个点。或者，它简单地可以代表社交网络中的人。

另一方面，边是两个节点之间的连接。关于上面的两个上下文，边可以是连接地球上两个地点的道路，或者两个人之间“认识”的关系。这实际上是一个现实生活关系的模型，远远超过了常规的表格关系。此外，这种基本的**图结构**可以通过许多不同的方式来丰富。

例如，人们可能会想到非对称的边，即我们有一去不复返的街道或人们知道某人，但另一个人却不知道这个人。这些图被称为**有向图**。此外，人们还可以给每条边添加**权重**，结果是一个 *(V, E, w)* 的元组。例如，添加两个地点之间道路的长度有助于找出不同地点之间的距离。在社交网络中，人们可以引入一个关于“你有多了解一个人？”的关系，其值介于 0 和 1 之间。

到目前为止，我们仅用这个理论描述了两种基本情况——导航和社交网络——还没有解决问题，也没有定义任何问题陈述。人们可能会想到以下问题：

+   我如何从地点 A 找到去地点 B 的路？

+   我的社交网络中最有影响力的人是谁？

+   我该如何在城市中心找到一条往返路线，参观 5 个不同的景点？

+   小世界实验：社交网络中的平均路径长度是多少？

每一篇都可能成为一篇新的媒介文章，但它们并不属于这一篇。让我们专注于给定的 OpenStreetMap 数据以及两个位置之间的最短路径。我们从著名的**Dijkstra 算法**开始，这是所有常见导航系统构建最短路径的基础。基本思想如下：跟随到节点的已知最小距离的边。在每一步中，节点的每个邻域的距离都在更新。如果有更小的距离，则更新，如果没有，则保持实际距离。我们将使用这个算法来找到从 A 地到 B 地通过柏林的最短路径。

关于图算法的评论：这个特定的问题陈述有两个重要的方面。

1.  许多这些算法的运行时间都很长。也就是说，运行时间复杂度可能仅通过添加更多数据点就急剧增加，因为有如此多的新可能性。

1.  许多时候没有唯一的解决方案，因为可能存在两条或多条路径，例如，最小化某些距离或目标函数。

1.  许多图问题属于 NP 类，即称为 NP 难。我们将在下一段中简要讨论这意味着什么。有时没有算法可以找到最佳解决方案。因此，算法被用来启发式地找到解决方案。

### 旅行商问题 – 往返柏林

通常，**旅行商问题（TSP）**描述了确定多个地点访问顺序的挑战，这样你只访问每个地点一次，除了你的起点，并且路线是最短的。除了柏林的往返旅行外，这个任务还出现在许多其他不同的领域。

不深入细节，这是一个 NP 难问题，即假设类**P**不等于**NP**，不存在在多项式最坏情况运行时间内找到最短行程的算法。
