## 为什么需求预测不像看起来那么简单

你刚刚在一家零售公司找到了你的第一份数据科学家的工作。你的新老板听说每个大公司都使用人工智能来提高其库存管理效率，所以你的第一个项目就是部署一个需求预测模型。你对时间序列预测了如指掌，但当你开始处理交易数据库时，你感到不确定，不知道从何开始。

在商业中，用例比简单的预测或预测机器学习任务具有更多层次和复杂性。这种差异是我们大多数人获得专业经验后才会学到的东西。让我们分解一下本指南将涵盖的不同步骤：

1.  你检查过错误值了吗？你是预测销售还是需求？

1.  你需要预测所有产品吗？你能提取更多相关信息吗？

1.  你是否定义了一个基线预测集？你的模型是局部还是全局的？你选择了哪些库和模型？

1.  你是如何评估你模型的表现的？你的模型预测会随着时间的推移而恶化吗？

1.  你将如何提供预测？你会在公司的云服务中部署模型吗？

## 数据集

### 你已经收集了所有相关的数据吗？

假设您的公司已经向您提供了一个包含不同产品和不同销售地点销售的交易数据库。这些数据被称为面板数据，这意味着您将同时处理许多时间序列。

交易数据库可能具有以下格式：销售日期、销售发生的地点标识符、产品标识符、数量，以及可能的价格。根据这些数据的收集方式，它将以不同的方式汇总，按时间（每日、每周、每月）和按组（按客户或按地点和产品）。

但这真的是您进行需求预测所需的所有数据吗？是的，也不是。当然，您可以处理这些数据并做出一些预测，如果序列之间的关系不复杂，一个简单的模型可能就足够了。但如果你正在阅读这个教程，你很可能对在数据不那么简单的情况下预测需求感兴趣。在这种情况下，如果您能访问以下附加信息，它可能会成为游戏规则的改变者：

+   **历史股票数据**：了解缺货发生的时间至关重要，因为即使销售数据没有反映出来，需求可能仍然很高。

+   **促销数据**：折扣和促销也可能影响需求，因为它们会影响顾客的购物行为。

+   **事件数据**：如后文所述，可以从日期索引中提取时间特征。然而，假日数据或特殊日期也可能影响消费。

+   **其他领域数据**：任何可能影响您正在处理的产品需求的其他数据都可能与此任务相关。

### 让我们开始编码！

对于这个教程，我们将使用按产品和销售地点汇总的月度销售数据。这个示例数据集来自[Stallion Kaggle 竞赛](https://www.kaggle.com/datasets/utathya/future-volume-prediction)，记录了通过批发商（代理商）分发给零售商的啤酒产品（SKU）。第一步是格式化数据集并选择我们想要用于训练模型的列。正如代码片段所示，为了简化，我们将所有事件列合并为一个名为“特殊日”的列。如前所述，这个数据集缺少库存数据，因此如果发生缺货，我们可能会误解实际需求。

```py
# Load data with pandas
sales_data = pd.read_csv(f'{local_path}/price_sales_promotion.csv')
volume_data = pd.read_csv(f'{local_path}/historical_volume.csv')
events_data = pd.read_csv(f'{local_path}/event_calendar.csv')

# Merge all data
dataset = pd.merge(volume_data, sales_data, on=['Agency','SKU','YearMonth'], how='left')
dataset = pd.merge(dataset, events_data, on='YearMonth', how='left')

# Datetime
dataset.rename(columns={'YearMonth': 'Date', 'SKU': 'Product'}, inplace=True)
dataset['Date'] = pd.to_datetime(dataset['Date'], format='%Y%m')

# Format discounts
dataset['Discount'] = dataset['Promotions']/dataset['Price']
dataset = dataset.drop(columns=['Promotions','Sales'])

# Format events
special_days_columns = ['Easter Day','Good Friday','New Year','Christmas','Labor Day','Independence Day','Revolution Day Memorial','Regional Games ','FIFA U-17 World Cup','Football Gold Cup','Beer Capital','Music Fest']
dataset['Special_days'] = dataset[special_days_columns].max(axis=1)
dataset = dataset.drop(columns=special_days_columns)
```

## 预处理

### 你检查过错误值了吗？

虽然这部分内容更为明显，但仍有必要提及，因为它可以避免将错误数据输入到我们的模型中。在交易数据中，寻找零价交易、销售量超过剩余库存、停售产品的交易以及类似情况。

### 你是在预测销售还是需求？

这是我们预测需求时应该做出的一个关键区分，因为目标是预测产品的需求以优化补货。如果我们只观察销售而不观察库存值，当出现缺货时，我们可能会低估需求，从而在我们的模型中引入偏差。在这种情况下，我们可以忽略缺货后的交易，或者尝试用正确的方式填充这些值，例如，使用需求的移动平均数。

### 让我们开始编码！

在本教程所选数据集的情况下，预处理相当简单，因为我们没有库存数据。我们需要通过填充正确的值来纠正零价交易，并为折扣列填充缺失值。

```py
# Fill prices
dataset.Price = np.where(dataset.Price==0, np.nan, dataset.Price)
dataset.Price = dataset.groupby(['Agency', 'Product'])['Price'].ffill()
dataset.Price = dataset.groupby(['Agency', 'Product'])['Price'].bfill()

# Fill discounts
dataset.Discount = dataset.Discount.fillna(0)

# Sort
dataset = dataset.sort_values(by=['Agency','Product','Date']).reset_index(drop=True)
```

## 特征提取

### 你需要预测所有产品吗？

根据预算、成本节约以及你使用的模型等一些条件，你可能不想预测整个产品目录。比如说，经过实验后，你决定使用神经网络。这些通常训练成本较高，需要更多的时间和大量资源。如果你选择训练和预测完整的产品集，你的解决方案成本将会增加，甚至可能使公司投资变得不值得。在这种情况下，一个很好的替代方案是根据特定标准对产品进行细分，例如使用你的模型来预测仅产生最高收入的产品。剩余产品的需求可以使用更简单、更便宜的模式进行预测。

### 你能提取更多相关信息吗？

特征提取可以应用于任何时间序列任务，因为你可以从日期索引中提取一些有趣的变量。特别是在需求预测任务中，这些特征很有趣，因为一些消费者习惯可能是季节性的。提取星期几、月份的星期或年份的月份可能有助于你的模型识别这些模式。正确编码这些特征是关键，我建议你阅读有关周期性编码的内容，因为它在某些情况下可能更适合时间特征。

### 让我们开始编码！

在这个教程中，我们首先要做的是对产品进行分段，只保留那些高周转的产品。在进行特征提取之前完成这一步骤可以帮助你在有很多低周转序列且不打算使用时减少性能成本。对于计算周转，我们只将使用训练数据。为此，我们事先定义了数据分割。请注意，我们有 2 个日期用于验证集，VAL_DATE_IN 表示那些也属于训练集但可以用作验证集输入的日期，而 VAL_DATE_OUT 表示从哪个点开始将使用时间步长来评估模型的输出。在这种情况下，我们将销售量占全年 75%的所有序列标记为高周转，但你可以在源代码中实现的函数周围进行一些实验。之后，我们进行第二次分段，以确保我们有足够的历史数据来训练模型。

```py
# Split dates 
TEST_DATE = pd.Timestamp('2017-07-01')
VAL_DATE_OUT = pd.Timestamp('2017-01-01')
VAL_DATE_IN = pd.Timestamp('2016-01-01')
MIN_TRAIN_DATE = pd.Timestamp('2015-06-01')

# Rotation 
rotation_values = rotation_tags(dataset[dataset.Date<VAL_DATE_OUT], interval_length_list=[365], threshold_list=[0.75])
dataset = dataset.merge(rotation_values, on=['Agency','Product'], how='left')
dataset = dataset[dataset.Rotation=='high'].reset_index(drop=True)
dataset = dataset.drop(columns=['Rotation'])

# History
first_transactions = dataset[dataset.Volume!=0].groupby(['Agency','Product'], as_index=False).agg(
    First_transaction = ('Date', 'min'),
)
dataset = dataset.merge(first_transactions, on=['Agency','Product'], how='left')
dataset = dataset[dataset.Date>=dataset.First_transaction]
dataset = dataset[MIN_TRAIN_DATE>=dataset.First_transaction].reset_index(drop=True)
dataset = dataset.drop(columns=['First_transaction'])
```

由于我们正在处理月度汇总数据，可提取的时间特征并不多。在这种情况下，我们包括位置，它只是序列顺序的数值索引。可以通过指定给 Darts 的编码器来在训练时间计算时间特征。此外，我们还计算了前四个月的移动平均和指数移动平均。

```py
dataset['EMA_4'] = dataset.groupby(['Agency','Product'], group_keys=False).apply(lambda group: group.Volume.ewm(span=4, adjust=False).mean())
dataset['MA_4'] = dataset.groupby(['Agency','Product'], group_keys=False).apply(lambda group: group.Volume.rolling(window=4, min_periods=1).mean())

# Darts' encoders
encoders = {
    "position": {"past": ["relative"], "future": ["relative"]},
    "transformer": Scaler(),
}
```

## 训练模型

### 你是否定义了一组基线预测？

就像在其他用例中一样，在训练任何花哨的模型之前，你需要建立一个想要超越的基线。通常，在选择基线模型时，你应该追求简单到几乎没有成本的东西。在这个领域的一个常见做法是使用时间窗口内需求的移动平均作为基线。这个基线可以在不要求任何模型的情况下计算，但为了代码的简洁性，在这个教程中，我们将使用 Darts 的基线模型，*NaiveMovingAverage*。

### 你的模型是局部还是全局？

你正在处理多个时间序列。现在，你可以选择为每个时间序列训练一个局部模型，或者只为所有序列训练一个全局模型。没有“正确”的答案，两者都取决于你的数据。如果你有数据，你认为当按店铺、产品类型或其他分类特征分组时具有相似的行为，你可能从全局模型中受益。此外，如果你有非常高的序列量，并且想要使用训练后存储成本更高的模型，你也可能更喜欢全局模型。然而，如果你在分析数据后认为序列之间没有共同的模式，你的序列量可管理，或者你没有使用复杂模型，选择局部模型可能更好。

### 你选择了哪些库和模型？

处理时间序列有很多选项。在这个教程中，我建议使用 Darts。假设你正在使用 Python，这个预测库非常容易使用。它提供了管理时间序列数据、分割数据、管理分组时间序列以及执行不同分析的工具。它提供了广泛的全球和局部模型，因此你可以运行实验而无需切换库。可用的选项示例包括基线模型、统计模型如 ARIMA 或 Prophet、基于 Scikit-learn 的模型、基于 Pytorch 的模型以及集成模型。有趣的选择包括像时间融合转换器（TFT）或时间序列深度编码器（TiDE）这样的模型，这些模型可以学习分组序列之间的模式，并支持分类协变量。

### 让我们开始编码！

开始使用不同的 Darts 模型的第一步是将 Pandas DataFrame 转换为时间序列 Darts 对象，并正确分割它们。为此，我实现了两个不同的函数，这些函数使用 Darts 的功能来执行这些操作。价格、折扣和事件的特征将在预测发生时已知，而对于计算特征，我们只知道过去值。

```py
# Darts format
series_raw, series, past_cov, future_cov = to_darts_time_series_group(
    dataset=dataset,
    target='Volume',
    time_col='Date',
    group_cols=['Agency','Product'],
    past_cols=['EMA_4','MA_4'],
    future_cols=['Price','Discount','Special_days'],
    freq='MS', # first day of each month
    encode_static_cov=True, # so that the models can use the categorical variables (Agency &amp; Product)
)

# Split
train_val, test = split_grouped_darts_time_series(
    series=series,
    split_date=TEST_DATE
)

train, _ = split_grouped_darts_time_series(
    series=train_val,
    split_date=VAL_DATE_OUT
)

_, val = split_grouped_darts_time_series(
    series=train_val,
    split_date=VAL_DATE_IN
)
```

我们将要使用的第一个模型是朴素移动平均（NaiveMovingAverage）基线模型，我们将将其与其他模型进行比较。这个模型非常快，因为它不学习任何模式，只是根据输入和输出维度执行移动平均预测。

```py
maes_baseline, time_baseline, preds_baseline = eval_local_model(train_val, test, NaiveMovingAverage, mae, prediction_horizon=6, input_chunk_length=12)
```

通常，在深入研究深度学习之前，你会尝试使用更简单且成本更低的模型，但在这个教程中，我想专注于两个对我效果很好的特殊深度学习模型。我使用这两个模型通过使用每日汇总销售数据和不同的静态和连续协变量以及库存数据，来预测多个商店数百种产品的需求。重要的是要注意，这些模型在长期预测方面比其他模型表现更好。

第一个模型是时间融合转换器（Temporal Fusion Transformer）。这个模型允许你同时处理大量时间序列（即，它是一个全局模型），并且在处理协变量时非常灵活。它可以处理静态、过去（值仅在过去已知）和未来（值在过去和未来都已知）的协变量。它能够学习复杂的模式，并支持概率预测。唯一的缺点是，尽管它优化得很好，但调整和训练可能会很昂贵。根据我的经验，它可以给出非常好的结果，但如果资源有限，调整超参数的过程会花费太多时间。在这个教程中，我们将使用大多数默认参数来训练 TFT，以及与基线模型相同的输入和输出窗口。

```py

    "patience": 50,
    "min_delta": 1e-3,
    "mode": "min",

    "force_reset": True,
    "batch_size": 128,
    "random_state": 42,
}

# TFT params
best_hp = {
 'optimizer_kwargs': {'lr':0.0001},
 'loss_fn': MAELoss(),
 'use_reversible_instance_norm': True,
 'add_encoders':encoders,
 }

## COMMENT TO LOAD PRE-TRAINED MODEL

    model_cls = TFTModel,
    common_model_args = common_model_args,
    specific_model_args = best_hp,
    model_name = 'TFT_model',
    past_cov = past_cov,
    future_cov = future_cov,
    train_series = train,
    val_series = val,
)

# Predict

                    series            = train_val,
                    past_covariates   = past_cov,
                    future_covariates = future_cov,
                    n                 = 6
                )
```

第二个模型是时序深度编码器。这个模型比 TFT 稍微新一些，它使用的是密集层而不是 LSTM 层，这使得模型的训练时间大大减少。Darts 的实现也支持所有类型的协变量和概率预测，以及多个时间序列。关于这个模型的论文显示，它在预测基准测试中可以匹配或超越基于 transformer 的模型。在我的情况下，由于调整成本较低，我能够在相同的时间或更短的时间内使用 TiDE 比使用 TFT 模型获得更好的结果。对于这次教程，我们只是进行了一次主要使用默认参数的首次运行。请注意，对于 TiDE，所需的 epoch 数量通常小于 TFT。

## 评估模型

### 你是如何评估你模型的性能的？

虽然典型的时序指标对于评估你的模型在预测方面的表现很有用，但建议更进一步。首先，在测试集上进行评估时，你应该丢弃所有出现缺货的序列，因为你不会将你的预测与真实数据进行比较。其次，将领域知识或关键绩效指标（KPI）纳入评估也很有趣。一个关键指标可能是，使用你的模型你能赚多少钱，避免缺货。另一个关键指标可能是，通过避免过剩库存的短保质期产品，你能节省多少钱。根据你价格的不稳定性，你甚至可以使用自定义损失函数来训练你的模型，例如价格加权的平均绝对误差（MAE）损失。

### 你的模型预测会随着时间的推移而恶化吗？

将数据分为训练集、验证集和测试集对于评估可能投入生产的模型的性能是不够的。仅仅通过测试集评估一个短期的时间窗口，你的模型选择会受到你的模型在非常具体的预测窗口中表现如何的偏见。Darts 提供了一个易于使用的回测实现，允许你通过预测移动时间窗口来模拟你的模型随时间的变化表现。通过回测，你还可以模拟每 N 步重新训练模型。

### 让我们开始编码！

如果我们从所有序列的 MAE 的角度来看我们模型的成果，我们可以看到明显的赢家是 TiDE，因为它设法最大限度地减少了基线误差，同时保持了较低的时间成本。然而，假设我们啤酒公司的最佳利益是同样减少缺货和过剩库存的货币成本。在这种情况下，我们可以使用价格加权的 MAE 来评估预测。

在计算了所有序列的价格加权 MAE 之后，TiDE 模型仍然是最好的模型，尽管结果可能有所不同。如果我们计算使用 TiDE 相对于基线模型在 MAE 方面的改进，改进率为 6.11%，但在货币成本方面，改进略有增加。相反，当仅考虑销售量而不是将价格纳入计算时，使用 TFT 的改进更大。

对于这个数据集，由于数据是按月汇总的，数据量有限，所以我们没有使用回测来比较预测。然而，如果可能的话，我鼓励你使用你的项目进行回测。在源代码中，我包括了此函数，以便轻松使用 Darts 进行回测：

```py
def backtesting(model, series, past_cov, future_cov, start_date, horizon, stride):
  historical_backtest = model.historical_forecasts(
    series, past_cov, future_cov,
    start=start_date,
    forecast_horizon=horizon,
    stride=stride,  # Predict every N months
    retrain=False,  # Keep the model fixed (no retraining)
    overlap_end=False,  
    last_points_only=False  
  )
  maes = model.backtest(series, historical_forecasts=historical_backtest, metric=mae)

  return np.mean(maes)
```

## 部署模型

### 你将如何提供预测？

在这个教程中，我们假设你已经在使用一个预定义的预测范围和频率。如果没有提供，这也是一个独立的使用案例，其中应考虑交付或供应商的提前期。了解你的模型预测需要多频繁是很重要的，因为这可能需要不同级别的自动化。如果你的公司每两个月需要一次预测，那么可能没有必要在自动化这项任务上投入时间、金钱和资源。然而，如果你的公司每周需要两次预测，而你的模型生成这些预测需要更长的时间，那么自动化流程可以节省未来的努力。

### 你会在公司的云服务中部署模型吗？

遵循之前的建议，如果你和你的公司决定部署模型并将其投入生产，遵循 MLOps 原则是个好主意。这将允许任何人将来轻松地进行更改，而不会破坏整个系统。此外，一旦模型投入生产，监控模型的表现也很重要，因为可能会发生概念漂移或数据漂移。如今，许多云服务提供管理机器学习模型开发、部署和监控的工具。这些服务的例子包括 Azure Machine Learning 和 Amazon Web Services。
